# クラウドプラクティショナー試験対策ドキュメント 執筆フロープロンプト

---

## このファイルの使い方

各ステップのプロンプトを、順番に Claude Code へ貼り付けて実行してください。
ユーザーレビューが必要なステップでは、Claude が確認を求めて停止します。

---

## ステップ1: 全体構成の作成とレビュー

```
aws-cp-planner エージェントを使って、以下の手順で全体構成 draft.md を作成してください。

### 作業内容

1. `.spec/plan.md` を読み込み、既存の章構成（第0章〜第16章・付録A〜C）を維持しながら、
   各章に以下の情報を拡充した `.spec/draft.md` を作成する。
   - 対応試験ドメイン
   - セクション構成（目的・主な内容・学習のポイント・図解の推奨・推定読了時間）
   - 章末確認問題の方針
   - 関連章へのリンク方針

   章の追加・削除・順序変更は行わないこと。対応サービスの追加は可だが、削除・変更は
   コメントとして別途提案すること。

2. draft.md が保存できたら、cloud-practitioner-doc-reviewer エージェントを
   「ベテランエンジニア」ロール限定で起動し、draft.md の全体構成をレビューさせる。
   - レビュー結果は `.spec/review-results/review-draft.md` に保存する
   - 観点: 技術的正確性・試験範囲の網羅性・章間の論理的つながり・不足トピック

3. レビュー結果（`.spec/review-results/review-draft.md`）を読み込み、
   指摘事項（重要度「高」「中」）を draft.md に反映して上書き保存する。

4. 完了後、私にレビューを求めてください。提示内容:
   - draft.md のパス
   - レビューで指摘された主な変更点のサマリー（箇条書き3〜5点）
```

## ステップ2: 執筆タスクの分割

```
（ステップ1のユーザーレビューが完了してから実行）

aws-cp-planner エージェントを使って、並列執筆用のタスク分割を行ってください。

### 作業内容

`.spec/draft.md` を読み込み、以下の方針で `.spec/task.md` を作成する。

分割方針:
- 1タスク = writer エージェント 1インスタンスが担当する単位
- 本編（第0〜16章）は章単位でタスクを切る
- 付録A（A-1〜A-7）はサブセクション単位でまとめて1タスクとする
- 付録B・付録Cはそれぞれ1タスクとする
- 各タスクには以下を含める:
  - output_path: docs/chapters/[XX-filename].md
  - scope: 担当章・付録の範囲
  - related_chapters: 参照すべき関連章のタスクID
  - special_notes: 他章と重複するサービスの扱い、draw.io ファイルの命名規則など

draw.io ファイルの命名規則: `[chapter-id]-[description].drawio.png`
（例: `ch01-web-architecture.drawio.png`）

完了後、私にレビューを求めてください。提示内容:
- task.md のパス
- タスク数と並列実行グループの構成（依存関係があるタスクがあれば明示）
```

## ステップ3: 並列執筆

```
（ステップ2のユーザーレビューが完了してから実行）

`.spec/task.md` に記載された全タスクを、cloud-practitioner-writer エージェントを
使って並列で執筆してください。

### 各 writer エージェントへの共通指示

- `.spec/draft.md` の該当章のセクション構成・学習のポイントに従って執筆する
- `.spec/plan.md` の「課題」と「対応サービス」を必ず反映する
- related_chapters に記載された章の output_path を参照し、サービスの重複説明を避ける
- 図解が推奨されているセクションでは diagram-creator エージェントを呼び出す
  - draw.io ファイル名は `special_notes` の命名規則に従う
  - 挿入先ドキュメントの配置場所を確認してから相対パスを伝える
- 執筆スタイル: 敬体（です・ます調）、IT初心者向け、一文は短く、例え話を多用

### 実行方法

- Agent ツールを呼び出す際は必ず `mode: "bypassPermissions"` を設定する
  （ファイル書き込み権限のブロックを防ぐため）
- Agent ツールは必ず**フォアグラウンド**で起動する（`run_in_background: true` は使わない）
  （バックグラウンドでは bypassPermissions が効かずファイル書き込みがブロックされるため）
- related_chapters に依存関係がないタスクは同時並列で起動する
- 依存関係があるタスク（他章を参照する必要があるもの）は、参照先の執筆完了後に起動する

全タスクの執筆完了後、完了したファイル一覧を私に報告してください。
```

## ステップ4: レビューと修正

```
（ステップ3が完了してから実行）

以下の手順で、全執筆ファイルのレビューと修正を行ってください。

### 4-1. 並列レビュー

cloud-practitioner-doc-reviewer エージェントを使い、
`docs/chapters/` 配下の全ファイルを対象に、以下の2ロールでレビューする。

- ロール: IT初心者 + ベテランエンジニア（デフォルトの全ロールレビューで可）
- 各章のレビュー結果を `.spec/review-results/review-[chapter-id].md` に保存する
- 章ごとにエージェントを並列起動して効率化する
- Agent ツール呼び出し時は必ず `mode: "bypassPermissions"` を設定する

### 4-2. 修正

レビュー結果（重要度「高」「中」の指摘）を cloud-practitioner-writer エージェントで反映する。

- 各章のレビューファイルを読み込み、該当する原稿ファイルを上書き修正する
- 重要度「低」の指摘は修正せず、`.spec/review-results/deferred-issues.md` に集約する
- 章ごとにエージェントを並列起動して効率化する
- Agent ツール呼び出し時は必ず `mode: "bypassPermissions"` を設定する

全修正完了後、私に報告してください。提示内容:
- 修正が入ったファイル数と主な修正内容のサマリー（ロール別）
- 対応を保留した「低」指摘の件数
```

3. draft.md との差分で主要な変更があった点（あれば）

私が確認後、次のアクション（追加修正 or 完了）を指示します。

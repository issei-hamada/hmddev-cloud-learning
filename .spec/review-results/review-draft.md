# draft.md レビュー結果（ベテランエンジニア視点）

## 総評

全体として、CLF-C02の4ドメインを意識した構成で、主要サービスを網羅的にカバーしている点は評価できる。章の順序も「基礎概念 → コンピューティング → サーバーレス → ストレージ → DB → ネットワーク → ...」と技術的な積み上げが妥当である。

一方で、ベテランエンジニアの視点からは以下の懸念がある。

1. **コメントで示唆されているが本文に反映されていない重要サービスが複数ある**（Bedrock、CDK、Pricing Calculator）。これらはCLF-C02で実際に出題されるため、コメントのまま放置すべきではない。
2. **第2分野（セキュリティとコンプライアンス）が30%と最大配分であるにもかかわらず、セキュリティ関連の章が第9章の1章のみに集中している**。他の章でのセキュリティ観点の横断的な補強が弱い。
3. **一部のサービス説明に技術的な不正確さ・不足がある**（EBSのマルチアタッチ、Auroraのスループット表現など）。
4. **試験頻出のいくつかのトピックが欠落している**（後述の不足トピック一覧を参照）。

スコア: 3.5 / 5

## 指摘事項

### 重要度「高」

#### H-001: Amazon Bedrockの欠落
- **該当箇所**: 第13章（AI/ML サービス）
- **問題点**: HTMLコメントで「Amazon Bedrock（基盤モデルのAPIアクセス）の追加を検討」と記載されているが、セクション構成・対応サービスに含まれていない。Bedrockは2024年以降のCLF-C02で確実に出題される重要サービスであり、生成AIの文脈でAmazon Qと並ぶ主要サービスである。
- **推奨対応**: セクション「13-7. 生成AIサービス」を追加し、Amazon Bedrock（基盤モデルへのAPIアクセス、複数のFMプロバイダーを選択可能）を正式に記載する。Amazon Qとの関係（QはBedrockの上に構築されている）も補足すべき。

#### H-002: AWS Pricing Calculatorの欠落
- **該当箇所**: 第15章（コスト管理と最適化）
- **問題点**: HTMLコメントで「AWS Pricing Calculator（見積もりツール）の追加を検討」と記載されているが、本文に含まれていない。「事前にコストを見積もる」サービスとして試験で直接問われるサービスであり、Cost ExplorerやBudgetsとの使い分けが頻出。
- **推奨対応**: セクション15-2またはセクション15-5付近にPricing Calculatorを正式に追加する。「Cost Explorer = 過去の実績分析」「Pricing Calculator = 導入前の見積もり」「Budgets = 予算管理・アラート」の3つの使い分けを明示する。

#### H-003: AWS CDKの欠落
- **該当箇所**: 第11章（CI/CD とデベロッパーツール）
- **問題点**: HTMLコメントで「AWS CDK（Cloud Development Kit）をCloudFormationの関連として言及を追加することを検討」と記載されているが、本文に含まれていない。CDKはCLF-C02の出題範囲に含まれており、CloudFormationとの関係で問われることがある。
- **推奨対応**: セクション11-3にCDKの言及を追加する。「CDK = プログラミング言語でインフラを定義し、内部的にCloudFormationテンプレートを生成する」という位置づけで簡潔に説明する。

#### H-004: EBSの「1つのEC2にアタッチ」の記述は不正確
- **該当箇所**: 第3章 セクション3-5「Amazon EBS」
- **問題点**: 「EBSは1つのEC2にアタッチ」と記載されているが、これは不正確。EBS io2ボリュームはマルチアタッチ（最大16インスタンス）をサポートしている。CLF-C02レベルではマルチアタッチの詳細は問われにくいが、「1つのEC2にしかアタッチできない」と断言するのは技術的に誤りであり、ベテランエンジニアが読むと信頼性を損なう。
- **推奨対応**: 「EBSは基本的に1つのEC2にアタッチする（一部ボリュームタイプではマルチアタッチも可能）」と修正する。または注釈として「io2ではマルチアタッチをサポート」と補足する。

#### H-005: ドメイン4（請求、料金、サポート）の配分に対するカバレッジが薄い
- **該当箇所**: 全体構成
- **問題点**: ドメイン4は12%の配分だが、第15章と付録Bの2箇所のみで扱っている。12%は65問中約8問に相当し、合否を分ける可能性がある。特に以下の頻出トピックが不足している:
  - AWS Pricing Calculator（前述）
  - AWSの料金モデルの詳細（リージョンごとの料金差、データ転送料金の詳細）
  - AWS Free Tierの具体例（Lambda 100万リクエスト/月、S3 5GB等）
- **推奨対応**: 第15章の内容を拡充し、上記トピックを追加する。特にFree Tierの具体的なサービス・数量は試験でそのまま問われることがある。

#### H-006: Auroraのスループット表現「RDSの最大5倍」は文脈が不十分
- **該当箇所**: 第4章 セクション4-3「Amazon Aurora」
- **問題点**: 「RDSの最大5倍のスループット」と記載されているが、これはMySQL互換の場合であり、PostgreSQL互換の場合は「最大3倍」と言われている。この区別なく「5倍」と断言するのは不正確。
- **推奨対応**: 「MySQL互換で最大5倍、PostgreSQL互換で最大3倍のスループット」と明記するか、少なくとも「MySQL互換の場合」という限定を付ける。

### 重要度「中」

#### M-001: CodeCommit / CodeDeployの言及がない
- **該当箇所**: 第11章（CI/CD とデベロッパーツール）
- **問題点**: AWS CodePipelineとCodeBuildは記載されているが、CodeCommit（ソースリポジトリ）とCodeDeploy（デプロイの自動化）が記載されていない。CodeCommitは2024年に新規利用が制限されたが、CLF-C02の出題範囲にはまだ含まれている可能性がある。CodeDeployはCI/CDパイプラインの「デプロイ」フェーズを担うサービスとして依然として重要。
- **推奨対応**: セクション11-2にCodeDeployを追加し、CodePipeline（オーケストレーション）→ CodeBuild（ビルド）→ CodeDeploy（デプロイ）の流れを明示する。CodeCommitについては「現在は新規利用不可だが、ソースリポジトリとしてGitHubやBitbucketとの連携が可能」と補足する。

#### M-002: AWS CloudShellの言及がない
- **該当箇所**: 第0章 セクション0-5「AWSの操作方法」
- **問題点**: AWSの操作方法としてコンソール、CLI、SDKの3つが挙げられているが、CloudShell（ブラウザベースのCLI環境）が言及されていない。CloudShellはCLF-C02で出題対象であり、「AWSの操作方法」の文脈で問われることがある。
- **推奨対応**: セクション0-5にCloudShellを追加する。「コンソール上からCLIを直接実行できるサービスで、ローカル環境の設定不要」と簡潔に説明する。

#### M-003: VPCエンドポイント（ゲートウェイ型/インターフェース型）の説明が不足
- **該当箇所**: 第5章 セクション5-6
- **問題点**: PrivateLinkは言及されているが、VPCエンドポイントの概念（ゲートウェイエンドポイント：S3/DynamoDB向け、インターフェースエンドポイント：PrivateLink経由）の分類が明示されていない。VPCエンドポイントはセキュリティとコストの両面で試験に出る。
- **推奨対応**: セクション5-6にVPCエンドポイントの2種類（ゲートウェイ型=S3/DynamoDB、インターフェース型=その他のAWSサービス）を明記する。

#### M-004: Step Functionsの重複
- **該当箇所**: 第2章と第6章
- **問題点**: Step Functionsが第2章（サーバーレス）と第6章（非同期処理とアプリケーション統合）の両方で対応サービスに含まれている。どちらが主たる解説箇所なのかが不明確。ベテランエンジニアとして、教材の構造的整合性の観点で気になる点。
- **推奨対応**: Step Functionsの主たる解説箇所を第6章に統一し、第2章では「ワークフロー管理が必要な場合は第6章のStep Functionsを参照」とする、または明確に棲み分けを記述する。

#### M-005: Lambda@EdgeおよびCloudFront Functionsの言及がない
- **該当箇所**: 第1章 セクション1-7「Amazon CloudFront」
- **問題点**: CloudFrontの説明にLambda@EdgeやCloudFront Functionsの言及がない。CLF-C02では「エッジでのコード実行」の文脈で問われることがあり、Lambda@Edgeは知っておくべきサービス。
- **推奨対応**: セクション1-7に「CloudFront Functions / Lambda@Edge = エッジロケーションでの軽量なコード実行が可能」の1行を追加する。

#### M-006: Amazon Keyspacesの欠落
- **該当箇所**: 第4章 セクション4-5「その他のデータベースサービス」
- **問題点**: Amazon Keyspaces（Cassandra互換のマネージドDB）が言及されていない。CLF-C02のデータベースサービスの選択問題で選択肢に登場する可能性がある。
- **推奨対応**: セクション4-5に「Keyspaces = Apache Cassandra互換のマネージドDB」を1行追加する。同様にAmazon QLDB（量子台帳データベース）やAmazon Timestream（時系列DB）も概要レベルで言及することを推奨する。

#### M-007: Elastic IPの説明がない
- **該当箇所**: 第1章または第5章
- **問題点**: Elastic IP（静的パブリックIPアドレス）の概念が記載されていない。EC2インスタンスの停止・起動でパブリックIPが変わる問題と、Elastic IPで固定する方法は基本的な知識。また、未使用のElastic IPに課金される点はコスト管理の文脈でも出題される。
- **推奨対応**: 第1章のEC2セクションまたは第5章のVPCセクションでElastic IPを簡潔に説明する。

#### M-008: AWS Configのセクション配置の検討
- **該当箇所**: 第10章 セクション10-4
- **問題点**: AWS Configは第10章（監視・運用と管理）に配置されているが、コンプライアンスの文脈ではドメイン2（セキュリティとコンプライアンス）にも深く関連する。第9章（セキュリティ）の関連章に第10章のConfigへの参照がない。
- **推奨対応**: 第9章の関連章に「第10章のAWS Configはコンプライアンス自動チェックの文脈で重要」との参照を追加する。

#### M-009: AWSアカウントのルートユーザーでのみ実行可能な操作の明示
- **該当箇所**: 第9章 セクション9-2
- **問題点**: ルートユーザーについて「日常利用しない」「MFAを有効化する」は記載されているが、ルートユーザーでのみ実行可能な操作（アカウントの閉鎖、サポートプランの変更、IAMユーザーの課金情報アクセス有効化等）が明示されていない。これはCLF-C02で頻出。
- **推奨対応**: ルートユーザーでのみ可能な操作の代表例を3-4個リストアップする。

#### M-010: 章番号の欠番
- **該当箇所**: 全体構成
- **問題点**: 第0章〜第16章の構成で、章番号が0から始まっている。これ自体は問題ないが、実質17章+付録3章の構成で全20章相当となる。推定読了時間の合計は約530分（約9時間）で、教材としてはやや分量が多い。ただし網羅性を重視する方針であれば妥当とも言える。
- **推奨対応**: 章番号体系は現状維持で問題ないが、学習ロードマップ（付録C）で「必須章」「推奨章」「参考章」を分類し、学習者が優先順位をつけやすくすることを推奨する。

- **推奨対応**: 「Dedicated Hosts = 物理サーバー単位の占有（ソケット/コア単位のライセンスに対応）」「Dedicated Instances = インスタンスレベルの分離（物理サーバーの特定は不可）」の違いを簡潔に記載する。

#### L-002: Gateway Load Balancer (GWLB) の言及がない
- **該当箇所**: 第1章 セクション1-4
- **問題点**: ELBの種類としてALB/NLB/CLBが記載されているが、Gateway Load Balancer（GWLB）が言及されていない。CLF-C02での出題頻度は低いが、ELBの種類を網羅する観点で含めるべき。
- **推奨対応**: 「GWLB = サードパーティのネットワークアプライアンス（ファイアウォール等）をスケーラブルに展開するためのロードバランサー」を1行追加する。

#### L-003: S3のアクセス制御（バケットポリシー、ACL）の言及が薄い
- **該当箇所**: 第3章
- **問題点**: S3の機能として耐久性やストレージクラスは詳述されているが、アクセス制御（バケットポリシー、S3 Access Points、ブロックパブリックアクセス）の記載がない。セキュリティの観点で重要。
- **推奨対応**: セクション3-2にS3のアクセス制御の概要を追加するか、第9章のセキュリティで横断的に言及する。

#### L-004: Amazon S3 Transfer Accelerationの言及がない
- **該当箇所**: 第3章
- **問題点**: S3への高速アップロードを実現するTransfer Accelerationが記載されていない。CloudFrontのエッジロケーションを活用する仕組みとして、第1章のCloudFrontとの関連性もある。
- **推奨対応**: セクション3-2に1行追加する。

#### L-005: AWS Well-Architected Toolの配置
- **該当箇所**: 第10章 セクション10-7
- **問題点**: Well-Architected Toolが第10章の「アドバイザリーと最適化ツール」に含まれているが、Well-Architected Framework自体は第0章で解説されている。第0章からの参照がない。
- **推奨対応**: 第0章のWell-Architected Frameworkのセクションに「実際のワークロードの評価にはWell-Architected Tool（第10章）を使用する」と参照を追加する。

## 肯定的評価

1. **章の順序設計が論理的**: 第0章で基礎概念を固め、第1章〜第8章でテクノロジー系サービスを積み上げ、第9章でセキュリティを横断的に整理し、第10章以降で管理・分析・移行・コストへと展開する流れは、ベテランエンジニアの知識体系とも整合しており自然である。

2. **各章に「課題 → 解決」の構造がある**: オンプレミスの課題からAWSサービスによる解決へという流れは、実務経験のあるエンジニアにとって「なぜこのサービスが存在するのか」を再確認するのに有効。

3. **関連章の相互参照が充実**: 各章末に関連章への参照が明記されており、章間の知識の関連性が把握しやすい。クロスリファレンスとして機能している。

4. **試験頻出ポイントの明示**: 学習のポイント列に「頻出」「ひっかけ」等の試験対策情報が含まれており、効率的な学習が可能。

5. **ドメイン対応の明示**: 各章に対応する試験ドメインが明記されており、学習の進捗とドメインカバレッジの対応が取りやすい。

6. **DR戦略の4パターン（第7章）やサーバーレスアーキテクチャパターン（第2章）など、実務的なアーキテクチャパターンが含まれている**: ベテランエンジニアが既存知識と対応づけて理解しやすい。

7. **付録Aの通信基礎は初心者向けだが、付録B（サポートプラン）と付録C（試験対策）はベテランにも有用**: 特にサポートプランの比較は実務でもよく確認する内容である。

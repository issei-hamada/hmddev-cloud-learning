**対象読者**: IT初心者・クラウド未経験者
**コンセプト**: アーキテクチャや要件ごとにAWSサービスを分類し、「なぜ必要か → 何が課題か → AWSでどう解決するか」の流れで理解する
**対応試験**: AWS Certified Cloud Practitioner (CLF-C02)

### 第0章　導入 ― クラウドの基本概念と AWS の全体像

**対応試験ドメイン**: 第1分野: クラウドのコンセプト（24%）、第2分野: セキュリティとコンプライアンス（30%）

#### 章の概要

**目的**: クラウドコンピューティングとは何かを理解し、AWSの全体像（グローバルインフラ、責任共有モデル、Well-Architected Framework）を把握する。以降の章すべての土台となる概念を習得する。

**課題**: クラウド以前はサーバーを自前で購入・設置・管理する必要があり、初期投資が高額で柔軟性に欠けていた。「クラウドに移行するとなぜ嬉しいのか」を理解するための出発点を提供する。

#### セクション構成

| セクション | 主な内容 | 学習のポイント |
|---|---|---|
| 0-1. クラウドとは何か | IaaS / PaaS / SaaS の定義と違い。日常の例え（電気・水道のように必要なぶんだけ使う）。オンプレミスとの比較。 | IaaS / PaaS / SaaS の区別は頻出。「どのサービスがどのモデルに該当するか」を問う問題が出る。 |
| 0-2. AWS グローバルインフラストラクチャ | リージョン、アベイラビリティゾーン（AZ）、エッジロケーションの概念と関係。リージョン選択の判断基準（レイテンシー、法規制、コスト、サービス提供状況）。 | リージョンとAZの関係は最頻出。「AZは物理的に離れたデータセンター群」「1リージョン=2つ以上のAZ」を正確に押さえる。エッジロケーションとリージョンの違いも問われる。 |
| 0-3. Well-Architected Framework | 6つの柱（運用上の優秀性、セキュリティ、信頼性、パフォーマンス効率、コスト最適化、持続可能性）の概要。 | 各柱の名前と概要を把握すればよい。深い理解は各章で行う。「持続可能性」は比較的新しい柱であることに注意。 |
| 0-4. 責任共有モデル | AWSの責任範囲（クラウドのセキュリティ）と顧客の責任範囲（クラウドにおけるセキュリティ）の境界。IaaS / PaaS / SaaS ごとの責任範囲の違い。 | 試験最頻出トピックの1つ。「OSのパッチ適用は誰の責任か」「物理的なデータセンターのセキュリティは誰の責任か」のような具体的な問いが出る。 |
| 0-5. AWSの操作方法 | マネジメントコンソール、AWS CLI、AWS CloudShell（ブラウザベースのCLI環境、ローカル設定不要）、SDKの概要。Infrastructure as Code（IaC）の考え方とCloudFormationの紹介。 | コンソール・CLI・CloudShell・SDKの4つの操作方法を押さえる。CloudShell = 「コンソール上からCLIを直接実行できるサービスで、ローカル環境の設定不要」。CloudFormationは「テンプレートからインフラを自動構築する」という概要レベルで十分。 |

#### 図解の推奨

- **クラウドサービスモデルの階層図**: IaaS / PaaS / SaaS がそれぞれ何を管理するかを積み木のように図示。オンプレミスと並べて比較。
- **AWSグローバルインフラストラクチャの構成図**: リージョン > AZ > データセンター、およびエッジロケーションの関係を地図ベースで図示。
- **責任共有モデルの境界図**: AWS側と顧客側の責任範囲を上下に分け、IaaS / PaaS / SaaS ごとに境界線がどう変わるかを示す。

#### 推定読了時間

30分

#### 対応サービス

- AWS マネジメントコンソール
- AWS CLI
- CloudFormation（IaC の導入として）

#### 章末確認問題の方針

- 問題数: 8問
- 形式: 択一選択式
- 出題の観点:
  - IaaS / PaaS / SaaS の分類
  - リージョン・AZ・エッジロケーションの違い
  - 責任共有モデルにおける責任範囲の判断
  - Well-Architected Framework の柱の名前と概要

#### 関連章

### 第1章　Web アプリケーションのホスティング

**対応試験ドメイン**: 第3分野: クラウドテクノロジーとサービス（34%）、第1分野: クラウドのコンセプト（24%）

#### 章の概要

**目的**: 最も基本的なWebシステムの構成を題材に、コンピューティングサービスの種類と使い分け、負荷分散・自動スケーリング・CDNの概念を理解する。

**課題**: サーバーの調達・OS管理が大変、アクセス増への対応、グローバル配信

#### セクション構成

| セクション | 主な内容 | 学習のポイント |
|---|---|---|
| 1-1. オンプレミスでWebサーバーを立てるとどうなるか | 物理サーバーの調達（数週間）、OS・ミドルウェアのインストール、ハードウェア障害対応。アクセス急増時に対応できないシナリオ。 | 「なぜクラウドか」を体感するための導入。具体的な数字（調達に2〜3ヶ月、初期費用100万円〜）を用いて説得力を持たせる。 |
| 1-2. Amazon EC2 の基本 | 仮想サーバーの概念、インスタンスタイプの分類（汎用・コンピューティング最適化・メモリ最適化・ストレージ最適化）、AMIの概念。Elastic IP（静的パブリックIPアドレス）の概念と注意点（未使用時に課金）。 | インスタンスファミリーの特徴を大まかに押さえる。「t系=汎用・バースト」「c系=コンピューティング最適化」程度の理解で十分。Elastic IP = 「インスタンスを停止しても変わらない固定IPアドレス」。未使用のElastic IPは課金対象になる点も試験で問われる。 |
| 1-3. EC2 の購入オプション | オンデマンド、リザーブドインスタンス、Savings Plans、スポットインスタンス、Dedicated Hosts の違いと選定基準。 | 試験頻出。「スポットは中断される可能性がある」「リザーブドは1年or3年のコミット」「Savings Plansは柔軟性が高い」が重要。ひっかけ: Dedicated Hosts と Dedicated Instances の違い。 |
| 1-4. Elastic Load Balancing | ALB / NLB / CLB の概要、ヘルスチェック、負荷分散の仕組み。 | ALB（HTTP/HTTPS、レイヤー7）と NLB（TCP/UDP、レイヤー4）の違いが問われる。CLBは旧世代であることを押さえる。 |
| 1-5. Auto Scaling | 水平スケーリングと垂直スケーリングの違い、Auto Scalingグループの概念、スケーリングポリシー。 | 「スケールアウト/イン」と「スケールアップ/ダウン」の用語を正確に。弾力性（Elasticity）はクラウドの重要な利点。 |
| 1-6. Amazon Lightsail と Elastic Beanstalk | Lightsail: 小規模向け固定料金のVPS。Beanstalk: アプリコードをアップロードするだけでインフラを自動構築。 | Lightsail は「シンプルで安価な仮想サーバー」、Beanstalk は「PaaS的なデプロイ自動化」と覚える。 |
| 1-7. Amazon CloudFront | CDNの概念、エッジロケーションからのコンテンツ配信、オリジンサーバーとの関係。CloudFront Functions / Lambda@Edge（エッジロケーションでの軽量なコード実行）。 | CloudFront = CDN。エッジロケーション = ユーザーに近い場所からコンテンツを配信。S3やEC2をオリジンに設定できる。Lambda@Edge = 「エッジでのコード実行」として試験で問われることがある。 |

#### 図解の推奨

- **オンプレミス vs クラウドのWebサーバー比較図**: 調達から運用までのタイムラインと責任範囲を対比。
- **EC2購入オプションの比較チャート**: コスト・柔軟性・中断リスクの3軸で各オプションを整理。
- **ELB + Auto Scaling + CloudFront の全体アーキテクチャ図**: ユーザーからのリクエストがCloudFront → ELB → EC2（Auto Scalingグループ）と流れる様子を図示。

#### 推定読了時間

40分

#### 対応サービス

- Amazon EC2（インスタンスタイプ、購入オプション: オンデマンド / リザーブド / スポット / Savings Plans）
- Elastic Load Balancing（負荷分散）
- Auto Scaling（伸縮性）
- Amazon Lightsail（小規模向け簡易ホスティング）
- AWS Elastic Beanstalk（アプリデプロイの自動化）
- Amazon CloudFront（CDN / エッジ配信）

#### 章末確認問題の方針

- 問題数: 10問
- 形式: 択一選択式・複数選択式
- 出題の観点:
  - EC2インスタンスタイプの選定（ユースケースに応じた判断）
  - 購入オプションのコスト比較と適切な選択
  - ELBの種類の使い分け
  - Auto Scalingの挙動（スケールアウト/インのトリガー）
  - CloudFrontの役割とエッジロケーションの関係

#### 関連章

- 第0章の「リージョン・AZ」の概念を前提とする
- 第2章（サーバーレス）は「EC2を使わない別のアプローチ」として対比される

### 第2章　サーバーレスアーキテクチャ ― サーバー管理からの解放

**対応試験ドメイン**: 第3分野: クラウドテクノロジーとサービス（34%）、第1分野: クラウドのコンセプト（24%）

#### 章の概要

**目的**: サーバーレスの概念を理解し、Lambda / API Gateway / Fargate / Step Functions の役割と使い分けを学ぶ。EC2との違いを明確にする。

**課題**: サーバー管理の運用負荷、アイドル時間のコスト、スケーリングの複雑さ

#### セクション構成

| セクション | 主な内容 | 学習のポイント |
|---|---|---|
| 2-1. サーバーレスとは何か | 「サーバーがない」のではなく「サーバーを管理しなくてよい」という意味。EC2（自分でサーバー管理）との対比。 | 「サーバーレス = サーバーが存在しない」は誤り。「インフラ管理をAWSに任せる」が正しい理解。 |
| 2-2. AWS Lambda | イベント駆動型の関数実行、実行時間の上限（15分）、課金モデル（リクエスト数+実行時間）、トリガー（S3イベント、API Gateway、CloudWatchイベント等）。 | 実行時間の上限15分は頻出。「使った分だけ課金」がサーバーレスの経済的メリット。長時間処理にはLambdaは不向き。 |
| 2-3. Amazon API Gateway | RESTful API / WebSocket APIの作成・管理、Lambdaとの連携、スロットリング。 | API Gateway + Lambda の組み合わせは典型的なサーバーレスパターンとして押さえる。 |
| 2-4. AWS Fargate | コンテナをサーバーレスで実行する仕組み。ECS/EKSとの関係。EC2起動タイプとの違い。 | Fargate = 「サーバーレスなコンテナ実行環境」。EC2起動タイプとの違い（インスタンス管理の有無）が問われる。 |
| 2-5. AWS Step Functions（概要のみ） | 複数のLambda関数やサービスを組み合わせたワークフローの定義・実行の概念紹介。詳細は第6章で扱う。 | 「複数のサービスをオーケストレーションする」という概念を把握する。詳細なユースケースや使い分けは第6章を参照。 |

#### 図解の推奨

- **EC2 vs Lambda の責任範囲比較図**: OS管理・スケーリング・パッチ適用などの責任が誰にあるかを対比。
- **サーバーレスWebアプリケーションのアーキテクチャ図**: ユーザー → API Gateway → Lambda → DynamoDB の典型パターン。
- **Step Functions ワークフロー図**: 複数のLambda関数がステートマシンで連携する様子のフローチャート。

#### 推定読了時間

30分

#### 対応サービス

- AWS Lambda
- Amazon API Gateway
- AWS Fargate
- AWS Step Functions（ワークフローのオーケストレーション）

#### 章末確認問題の方針

- 問題数: 8問
- 形式: 択一選択式
- 出題の観点:
  - EC2とLambdaの使い分け（実行時間、管理負荷、コスト）
  - Lambdaの制約（実行時間上限、メモリ上限）
  - サーバーレスサービスの識別（どれがサーバーレスでどれがそうでないか）
  - Fargateの位置づけ（コンテナ + サーバーレス）

#### 関連章

- 第1章（EC2）との対比が重要。「EC2でやる場合」と「サーバーレスでやる場合」を比較できるようにする
- 第4章（データベース）のDynamoDBはLambdaとの組み合わせでよく使われる
- 第6章（非同期処理）のSQS・SNS・EventBridgeはLambdaのトリガーとして頻出
- 第8章（コンテナ）でFargateをより詳しく扱う

### 第3章　データの永続化 ― ストレージ戦略

**対応試験ドメイン**: 第3分野: クラウドテクノロジーとサービス（34%）

#### 章の概要

**目的**: オブジェクト・ブロック・ファイルの3種類のストレージの違いを理解し、用途に応じた使い分けを学ぶ。S3のストレージクラスとライフサイクル管理を重点的に押さえる。

**課題**: データの種類によって最適な保存方法が異なる、コストとアクセス頻度のトレードオフ、バックアップとライフサイクル管理

#### セクション構成

| セクション | 主な内容 | 学習のポイント |
|---|---|---|
| 3-1. ストレージの3つの種類 | オブジェクトストレージ・ブロックストレージ・ファイルストレージの違い。日常の例え（オブジェクト=倉庫のダンボール箱、ブロック=本棚の1冊ずつ、ファイル=共有フォルダ）。 | 3種類の違いは頻出。「S3=オブジェクト」「EBS=ブロック」「EFS=ファイル」の対応を確実に。 |
| 3-2. Amazon S3 の基本 | バケットとオブジェクトの概念、キーの構造、無制限のスケーラビリティ、イレブンナイン（99.999999999%）の耐久性。 | S3の耐久性の数値は頻出。「S3はオブジェクト数に上限なし、1オブジェクトは最大5TB」も押さえる。 |
| 3-3. S3 ストレージクラス | Standard、Standard-IA、One Zone-IA、Glacier Instant Retrieval、Glacier Flexible Retrieval、Glacier Deep Archive、Intelligent-Tiering の特徴と選定基準。 | ストレージクラスの使い分けは最頻出。アクセス頻度とコストの関係、取り出し時間の違い（Glacier系）を正確に。ひっかけ: One Zone-IA はAZ障害に弱い。 |
| 3-4. S3 のライフサイクルポリシーとバージョニング | ライフサイクルルールによるストレージクラスの自動移行、バージョニングによる誤削除防止。 | ライフサイクルポリシーで「作成後30日でIA、90日でGlacier」のような自動移行ができることを押さえる。 |
| 3-5. Amazon EBS | EC2にアタッチするブロックストレージ、ボリュームタイプ（gp3, io2等）、スナップショット。 | 「EBSは基本的に1つのEC2にアタッチ（io2ボリュームタイプではマルチアタッチも可能）」「EFSは複数のEC2で共有可能」の違いは頻出。スナップショットはS3に保存される。 |
| 3-6. Amazon EFS と Amazon FSx | EFS: Linux向け共有ファイルシステム。FSx: Windows向け（FSx for Windows File Server）やハイパフォーマンス向け（FSx for Lustre）。 | EFS = Linux / NFS、FSx for Windows = Windows / SMB の対応を押さえる。 |
| 3-7. ハイブリッドストレージとバックアップ | Storage Gateway（オンプレミスからS3へのキャッシュ付きアクセス）、AWS Backup（一元的なバックアップ管理）、Elastic Disaster Recovery。 | Storage Gateway は「オンプレミスとクラウドの橋渡し」。AWS Backup は「複数サービスのバックアップを一元管理」。 |

#### 図解の推奨

- **ストレージ3種類の比較図**: オブジェクト・ブロック・ファイルの特性（アクセス方法、用途、対応サービス）を表形式で整理。
- **S3ストレージクラスのコスト・アクセス頻度マトリクス**: 横軸にアクセス頻度、縦軸にコストを取り、各クラスをプロット。
- **EBS / EFS / S3 の使い分けフローチャート**: ユースケースに応じてどのストレージを選ぶかの判断フロー。

#### 推定読了時間

35分

#### 対応サービス

- Amazon S3（オブジェクトストレージ、ストレージクラス: Standard / IA / Glacier）
- Amazon EBS（ブロックストレージ）
- Amazon EFS（共有ファイルシステム）
- Amazon FSx
- AWS Storage Gateway（ハイブリッド環境のキャッシュ）
- AWS Backup
- AWS Elastic Disaster Recovery

#### 章末確認問題の方針

- 問題数: 10問
- 形式: 択一選択式・複数選択式
- 出題の観点:
  - ストレージ種類の識別と使い分け
  - S3ストレージクラスの選定（シナリオベース）
  - EBS / EFS / S3 の特性比較
  - ライフサイクルポリシーの設定シナリオ
  - Storage Gatewayの役割

#### 関連章

- 第1章（EC2）で起動したインスタンスに接続するストレージとしてEBSが登場

### 第4章　データベース設計 ― RDB から NoSQL まで

**対応試験ドメイン**: 第3分野: クラウドテクノロジーとサービス（34%）

#### 章の概要

**目的**: アプリケーション要件に応じたデータベースの選択基準を理解する。RDS / Aurora / DynamoDB を中心に、マネージドDBサービスの利点を学ぶ。

**課題**: セルフマネージド vs マネージド、スケーラビリティ、可用性、データモデルの選定

#### セクション構成

**対応試験ドメイン**: 全ドメインに横断的に対応

#### 章の概要

**目的**: 学習リソースの紹介、効果的な学習方法、模擬試験の活用法、試験当日の時間配分のコツなど、試験合格に向けた実践的なアドバイスを提供する。

#### セクション構成

| セクション | 主な内容 | 学習のポイント |
|---|---|---|
| C-1. 試験の全体像と戦略 | 試験形式の復習（65問、90分）、採点対象外15問の存在、合格スコア700/1000。時間配分の目安（1問あたり約80秒）。 | 「わからない問題は印をつけて後回し」「消去法を活用」が基本戦略。 |
| C-2. 効果的な学習方法 | ドメイン別の学習優先度（セキュリティ30% → テクノロジー34%の順）、用語の暗記ではなく概念の理解を重視。学習ロードマップ（章ごとに「必須章」「推奨章」「参考章」に分類して優先順位を提示）。 | 本教材の章と試験ドメインの対応関係を整理。弱点分野の特定方法。必須章（第0・1・3・4・5・9・10・15章）→ 推奨章（第2・6・7・8・11・14章・付録A〜B）→ 参考章（第12・13・16章・付録C）の順で学習効率を高める。 |
| C-3. 模擬試験の活用法 | AWS公式模擬試験、サードパーティの模擬試験の紹介。模擬試験の復習方法（不正解の理由を必ず確認）。 | 模擬試験は「受けて終わり」ではなく「復習が本番」。 |
| C-4. よく出るパターンと対策 | 「このシナリオに最適なサービスは?」パターン、「責任共有モデルでは誰の責任か?」パターン、「コスト最適化のために何をすべきか?」パターン。 | 典型的な問題パターンに慣れておくことで解答速度を上げる。 |
| C-5. 試験当日の注意事項 | 持ち物、テストセンター vs オンライン受験の違い、直前の復習ポイント。 | 冷静に臨むための準備リスト。 |

#### 図解の推奨

- **ドメイン別学習優先度チャート**: 各ドメインの配分と対応する本教材の章を一覧化。
- **学習スケジュールのモデル例**: 2週間 / 1ヶ月 / 2ヶ月の学習計画テンプレート。

#### 推定読了時間

15分

#### 章末確認問題の方針

- 問題数: 設けない
- 代わりに総合模擬問題（20問程度）を別セクションとして提供することを推奨

#### 関連章

- 全章の総まとめとしての位置づけ
- 各ドメインと本教材の章の対応表を提供し、弱点分野への誘導を行う

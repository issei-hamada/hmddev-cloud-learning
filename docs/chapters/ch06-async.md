## 6-1. 密結合 vs 疎結合

### このセクションで学ぶこと

システム設計における「密結合」と「疎結合」の違いを理解し、なぜ疎結合が推奨されるのかを把握します。

### 「結合」とは何か

アプリケーションは通常、1つの巨大なプログラムではなく、複数の「コンポーネント（部品）」が協力して動いています。たとえばECサイトなら、「注文を受け付ける部品」「在庫を管理する部品」「メールを送る部品」「決済を処理する部品」などがあります。

「結合」とは、これらの部品同士がどのようにつながっているかを表す言葉です。つながり方の設計によって、システムの強さや柔軟性が大きく変わります。

### 密結合 ― 直接電話でやりとりするスタイル

密結合（みつけつごう）とは、コンポーネント同士が直接通信しあう構造のことです。

日常生活で例えると、「直接電話をかける」スタイルです。注文を受け付ける担当者が、在庫管理の担当者に直接電話をかけて「この商品の在庫ある？」と聞き、返事を待ってから次の処理に進みます。

この方式はシンプルでわかりやすいのですが、大きな問題があります。

まず、相手が応答できないと自分も止まってしまいます。在庫管理の担当者が席を外していたら、注文受付の担当者はずっと電話口で待たされます。つまり、1つのコンポーネントの障害が、他のコンポーネントにも波及してしまうのです。

次に、処理量が増えたときの対応が難しくなります。注文が急増しても、在庫管理の担当者は1人しかいないので、電話がつながらなくなります。

さらに、部品を入れ替えるのも大変です。在庫管理のシステムを新しくしたい場合、直接通信しているすべてのコンポーネントを修正する必要があります。

### 疎結合 ― メールボックスを介するスタイル

疎結合（そけつごう）とは、コンポーネント同士が直接やりとりせず、「仲介役」を挟んで通信する構造のことです。

こちらは「メールボックスにメモを入れる」スタイルです。注文受付の担当者は、在庫管理の担当者に直接電話する代わりに、メールボックスに「この商品の在庫確認をお願いします」というメモを投函します。在庫管理の担当者は、自分のペースでメモを取り出して処理します。

この方式には大きなメリットがあります。

1つ目は障害の局所化です。在庫管理の担当者が席を外していても、メモはメールボックスに溜まるだけです。注文受付の担当者は止まることなく、次の注文を処理できます。在庫管理の担当者が戻ったら、溜まったメモを順に処理すればOKです。

2つ目は独立したスケーリングです。注文が急増したら、注文受付の担当者を増やせばいいし、在庫確認が追いつかなければ在庫管理の担当者を増やせばいい。それぞれ独立して人数を調整できます。

3つ目は部品の交換が容易なことです。在庫管理のシステムを新しくしても、「メールボックスからメモを取り出す」という仕組みさえ同じなら、他の部品を修正する必要がありません。

![密結合と疎結合の比較](../images/ch06-coupling-comparison.drawio.png)

### AWSにおける「仲介役」

AWSでは、この「仲介役」となるサービスがいくつか用意されています。代表的なものが Amazon SQS（メッセージキュー）、Amazon SNS（通知サービス）、Amazon EventBridge（イベントバス）です。これらのサービスについて、次のセクション以降で詳しく学んでいきます。

疎結合は、AWSが推奨するベストプラクティス集「Well-Architected Framework（ウェルアーキテクテッドフレームワーク）」でも重要な設計原則として位置づけられています。試験でも頻繁に問われるテーマです。

---

> **試験のポイント**
> - 「密結合のシステムをどう改善するか」→ 答えは「SQSやSNSを使って疎結合にする」
> - 疎結合のメリット = 障害の局所化、独立したスケーリング、コンポーネントの交換が容易
> - 「Well-Architected Framework」の設計原則として疎結合が推奨されている
> - 密結合の問題 = 1つの障害が全体に波及する、スケーリングが困難

---

## 6-2. Amazon SQS

### このセクションで学ぶこと

メッセージキューサービス「Amazon SQS」の仕組みと、2種類のキュー（Standard / FIFO）の違いを理解します。

### メッセージキューとは

Amazon SQS（Simple Queue Service）は、AWSが提供するフルマネージドなメッセージキューサービスです。「メッセージキュー」とは、その名のとおり「メッセージの待ち行列（キュー）」のことです。

身近な例で説明しましょう。ラーメン屋の食券機を想像してみてください。お客さん（送信者）は食券機でチケット（メッセージ）を購入し、カウンターに置きます。店員（受信者）は自分のペースでチケットを受け取り、順番に調理します。お客さんは食券を買った時点で「注文完了」なので、店員の作業を待つ必要がありません。

SQSも同じ仕組みです。あるサービス（プロデューサー = メッセージの送り手）がメッセージをキューに送信し、別のサービス（コンシューマー = メッセージの受け手）がキューからメッセージを取り出して処理します。送信側と受信側が直接やりとりしないので、まさに前のセクションで学んだ「疎結合」を実現できます。

### SQSの重要な特徴

SQSにはいくつかの重要な特徴があります。

まず、SQSは「ポーリング（polling）型」のサービスです。ポーリングとは英語で「繰り返し問い合わせる」という意味で、つまり「自分から取りに行く」方式のことです。コンシューマーが定期的にキューを確認して、メッセージがあれば取り出します。メッセージが届いたら自動的に通知されるわけではなく、自分からチェックしに行く必要があります。郵便受けを自分で見に行くのと同じイメージです。

次に、「可視性タイムアウト（visibility timeout）」という仕組みがあります。これは「他のコンシューマーから一時的に見えなくなる時間」のことです。コンシューマーがメッセージを取り出すと、そのメッセージは一定時間、他のコンシューマーからは見えなくなります。これは、同じメッセージを複数のコンシューマーが二重に処理してしまうことを防ぐためです。処理が正常に完了すれば、コンシューマーがメッセージを削除します。もし処理に失敗してタイムアウトすると、メッセージは再びキューに戻り、別のコンシューマーが処理できるようになります。

また、SQSのメッセージの最大サイズは256KBで、メッセージの保持期間はデフォルト4日・最大14日です。大きなデータを扱う場合は、データ本体をS3に保存し、SQSにはS3の場所を示す情報だけを入れるパターンが一般的です。

さらに、「デッドレターキュー（DLQ）」という仕組みも重要です。何度処理を試みても失敗するメッセージがあると、キューに戻されては取り出され、また失敗して……と無限ループになってしまいます。デッドレターキューは、一定回数失敗したメッセージを別のキューに移動させる仕組みです。「不良品を検品ラインから取り出して、別の棚に置いておく」ようなものです。あとから開発者がデッドレターキューを確認して、何が問題だったのかを調査できます。

### Standard キュー vs FIFO キュー

SQSには「Standard キュー」と「FIFO キュー」の2種類があります。この違いは試験で非常によく問われるポイントです。

Standard キューは、処理速度を最優先にしたキューです。特徴は2つ。1つは「最低1回配信（at-least-once delivery）」で、同じメッセージが2回届く可能性がごくまれにあります。もう1つは「順序保証なし（best-effort ordering）」で、メッセージが送信した順番どおりに届くとは限りません。その代わり、ほぼ無制限のスループット（処理能力）を持ち、大量のメッセージを高速に処理できます。実際にはAPIコール数のソフトリミットがありますが、Service Quotasから引き上げ申請が可能です。試験ではStandardキューは「ほぼ無制限のスループット」と覚えてください。

FIFO キューは、正確さを最優先にしたキューです。「FIFO」は「First In, First Out（先入れ先出し）」の略で、最初に入れたメッセージが最初に出てくることを保証します。特徴も2つ。1つは「1回のみ配信（exactly-once processing）」で、同じメッセージが重複して届くことがありません。もう1つは「順序保証あり」で、メッセージは送信した順番どおりに届きます。ただし、Standard キューに比べるとスループットに制限があります（1秒あたり300メッセージ、バッチ処理で最大3000メッセージ）。

どちらを選ぶかは、ユースケース次第です。たとえば画像のサムネイル生成のように、多少順番が前後しても、同じ処理が2回走っても大きな問題にならない場合はStandardキューが向いています。一方、銀行の口座振替のように、処理の順序が重要で、二重処理が許されない場合はFIFOキューを選びます。

| 特徴 | Standard キュー | FIFO キュー |
|------|---------------|------------|
| 配信保証 | 最低1回（重複あり得る） | 1回のみ（重複なし） |
| 順序 | 保証なし | 保証あり |
| スループット | ほぼ無制限 | 制限あり |
| 向いている用途 | 大量処理、順序不問 | 順序・正確性が重要な処理 |

---

> **試験のポイント**
> - SQS = ポーリング型（コンシューマーがメッセージを取りに行く）
> - Standard = 「最低1回配信、順序保証なし、高スループット」
> - FIFO = 「1回のみ配信、順序保証あり、スループットに制限」
> - 「順序を保証したい」「重複を防ぎたい」→ FIFO キュー
> - デッドレターキュー = 処理に繰り返し失敗したメッセージの退避先

## 6-3. Amazon SNS

### このセクションで学ぶこと

「Amazon SNS」のPub/Subモデルの仕組みと、SQSとの違いを理解します。また、SNSとSQSを組み合わせた「ファンアウトパターン」を学びます。

### Pub/Sub モデルとは

Amazon SNS（Simple Notification Service）は、AWSが提供するフルマネージドな通知サービスです。SNSは「Pub/Sub（パブサブ）モデル」という仕組みで動きます。

Pub/Sub は「Publish（発行する）」と「Subscribe（購読する）」の略です。新聞の仕組みに似ています。新聞社（パブリッシャー）が記事を発行すると、新聞を購読している全員（サブスクライバー）に同じ内容が届きます。読者が何人いても、新聞社は1回記事を発行するだけで済みます。

SNSでは、「トピック」と呼ばれるチャンネルに対してメッセージを発行（パブリッシュ）します。そのトピックを購読（サブスクライブ）しているすべての宛先に、メッセージが自動的に配信されます。

### SQS との決定的な違い

SQSとSNSはどちらもメッセージを扱うサービスですが、根本的な違いがあります。

SQSは「ポーリング型」です。受信者が自分からキューを確認して、メッセージを取りに行きます。郵便受けを自分で見に行くスタイルです。そして、1つのメッセージは基本的に1つのコンシューマーが処理します。

SNSは「プッシュ型」です。メッセージが発行された瞬間に、すべてのサブスクライバーに自動的に届けられます。宅配便のように、送り主が発送すれば届け先に届くスタイルです。そして、1つのメッセージが複数のサブスクライバーに同時に届きます。

この違いを一言でまとめると、SQSは「1対1の非同期通信」、SNSは「1対多の通知」です。

### SNS のサブスクライバーの種類

SNSのトピックには、さまざまな種類のサブスクライバーを登録できます。SQSキュー、Lambda関数、メールアドレス、HTTPエンドポイント、SMSなどが対象です。

たとえば、アプリケーションのエラーが発生したときに、SNSトピックにメッセージを発行すると、運用チームにはメールで通知が届き、同時にLambda関数が起動して自動復旧処理を行い、さらにSlackに通知を送る……といったことが一度の発行で実現できます。

なお、SNSにはサブスクリプションフィルターポリシーという機能もあり、メッセージの属性に基づいて特定のサブスクライバーにのみ配信することも可能です。ただし、後ほど紹介するEventBridgeほどの柔軟性はありません。

### ファンアウトパターン ― SNS + SQS の組み合わせ

SNSとSQSを組み合わせた「ファンアウトパターン」は、試験でもよく出題される重要な設計パターンです。「ファンアウト」とは「扇状に広がる」という意味で、1つのメッセージを複数の処理に分配するパターンのことです。

ECサイトの注文処理を例に考えてみましょう。新しい注文が入ったら、在庫の更新、決済処理、確認メールの送信という3つの処理を同時に行いたいとします。

この場合、注文サービスはSNSトピックに「注文が入りました」というメッセージを1回発行します。SNSトピックには3つのSQSキューがサブスクライブされていて、それぞれ「在庫管理キュー」「決済処理キュー」「通知キュー」にメッセージのコピーが届きます。各キューに対応するコンシューマー（Lambda関数など）が、それぞれの処理を独立して行います。

![SNS + SQS ファンアウトパターン](../images/ch06-fanout-pattern.drawio.png)

このパターンの利点は3つあります。まず、注文サービスは1回メッセージを発行するだけで、複数の後続処理に伝達できます。次に、各処理が独立しているので、決済処理が遅延してもメール通知には影響しません。さらに、新しい処理を追加したいとき（たとえばポイント付与処理）、注文サービスを修正せずに、新しいSQSキューをSNSトピックにサブスクライブするだけで済みます。

---

> **試験のポイント**
> - SNS = プッシュ型（サブスクライバーに自動配信）、SQS = ポーリング型（受信者が取りに行く）
> - SNS = 1対多の通知、SQS = 1対1の非同期通信
> - ファンアウトパターン = SNS → 複数のSQSキュー → 各コンシューマー
> - 「1つのイベントを複数のサービスに同時通知したい」→ SNS
> - 「メッセージを溜めておいて、後で順に処理したい」→ SQS

## 6-4. Amazon EventBridge

### このセクションで学ぶこと

イベント駆動アーキテクチャの中心となる「Amazon EventBridge」の仕組みと、SNSとの違いを理解します。

### イベント駆動アーキテクチャとは

イベント駆動アーキテクチャとは、「何かが起きたら（イベント）、それに応じて処理が動く」という設計思想です。第2章で学んだLambdaのトリガーの仕組みを思い出してください。「S3にファイルがアップロードされたら、Lambda関数が起動する」というのも、イベント駆動の一例です。

しかし、システムが大きくなると、さまざまな場所でさまざまなイベントが発生します。「ユーザーがサインアップした」「注文が確定した」「サーバーのCPU使用率が90%を超えた」「外部のSaaSサービスでデータが更新された」……。これらのイベントを一元的に管理し、適切な処理に振り分ける仕組みが必要になります。

### EventBridge とは

Amazon EventBridgeは、AWSが提供するサーバーレスのイベントバスサービスです。「イベントバス」とは、さまざまなイベントが集まり、ルールに基づいて適切な宛先に振り分けられる「バスターミナル（バスの発着所）」のようなものです。

EventBridgeの動作は3つのステップで理解できます。

ステップ1: イベントソース（イベントの発生元）がイベントバスにイベントを送ります。イベントソースは、AWSサービス（EC2の状態変化、S3へのアップロードなど）、自社のアプリケーション、または外部のSaaSサービス（Zendesk、Datadog など）です。

ステップ2: イベントルールが、届いたイベントの内容を確認し、条件に合うイベントだけをフィルタリングします。たとえば「EC2インスタンスが停止した」というイベントだけを拾い、「EC2インスタンスが起動した」というイベントは無視する、といったことができます。

ステップ3: ルールに合致したイベントが、指定されたターゲット（Lambda関数、SQSキュー、Step Functionsなど）に送られ、処理が実行されます。

### SNS との違い

SNSもEventBridgeも「イベントを複数の宛先に届ける」点では似ています。しかし、EventBridgeにはSNSにはない特徴があります。

最大の違いはイベントのフィルタリング機能です。SNSにも前述のサブスクリプションフィルターポリシーがありますが、EventBridgeはそれよりもはるかに柔軟です。EventBridgeは、イベントの中身を細かく確認して、「この条件に合うイベントだけをこの宛先に送る」という柔軟なルーティングが可能です。

もう1つの違いはSaaS統合です。EventBridgeは、Zendesk、Datadog、Auth0などの外部SaaSサービスからのイベントを直接受け取ることができます。SNSにはこの機能がありません。

また、EventBridgeはスケジュール実行もサポートしています。「毎日午前9時にこのLambda関数を実行する」といったcron（定期実行）のような使い方もできます。

なお、EventBridgeはもともと「Amazon CloudWatch Events」という名前のサービスでした。機能が拡張されて独立したサービスとなり、現在の名前に変わっています。試験では「CloudWatch Events の後継がEventBridge」という知識が問われることもあります。

### SQS / SNS / EventBridge の使い分け

3つのサービスの使い分けを整理しましょう。

![SQS / SNS / EventBridge の使い分けフロー](../images/ch06-messaging-selection.drawio.png)

基本的な判断基準はこうです。メッセージを1つの宛先に確実に届けて処理したいなら SQS。1つのメッセージを複数の宛先にシンプルに配信したいなら SNS。イベントの内容に応じて柔軟にルーティングしたい、SaaS連携が必要、またはスケジュール実行をしたいなら EventBridge。実際のシステムでは、これらを組み合わせて使うことが一般的です。

なお、オンプレミス（自社のサーバー環境）で ActiveMQ や RabbitMQ といったメッセージブローカーを使っているシステムをAWSに移行する場合は、Amazon MQ というサービスも選択肢になります。Amazon MQ を使うと、既存のアプリケーションコードをほとんど変更せずに移行できます。新しくシステムを構築する場合は、SQS や SNS を使うのがAWSの推奨です。

オンプレミスの技術をご存じの方は、おおまかに次のように対応づけるとイメージしやすいでしょう。SQS は RabbitMQ や ActiveMQ のキューに近い存在、SNS は RabbitMQ の Exchange（fanout）に近い存在、EventBridge は Apache Kafka のトピック＋フィルタに近い存在です（ただし厳密には異なる点もあります）。

---

> **試験のポイント**
> - EventBridge = イベント駆動アーキテクチャの中心的サービス
> - EventBridge はイベントのフィルタリング・ルーティングが得意（SNSよりも柔軟）
> - EventBridge は外部SaaSサービスとの統合が可能
> - EventBridge = CloudWatch Events の後継
> - スケジュール実行（cron的な定期実行）にも使える
> - オンプレミスの ActiveMQ / RabbitMQ からの移行 → Amazon MQ
> - 新規構築 → SQS / SNS が推奨

---

## 6-5. 疎結合アーキテクチャの設計パターン

### このセクションで学ぶこと

これまで学んだサービスを組み合わせた具体的な設計パターンを理解し、AWS Step Functions によるワークフロー管理の詳細を学びます。

### パターン1: SQS + Lambda のイベント処理

最もシンプルな疎結合パターンです。処理のリクエストをSQSキューに入れ、Lambda関数がキューからメッセージを取り出して処理します。

たとえば、ユーザーがアップロードした画像のサムネイルを作成するケースを考えてみましょう。ユーザーがS3に画像をアップロードすると、S3がSQSキューにメッセージを送ります。Lambda関数がキューからメッセージを受け取り、サムネイルを生成してS3に保存します。

このパターンの利点は、画像のアップロードとサムネイル生成が完全に分離されていることです。大量の画像が同時にアップロードされても、メッセージはキューに溜まるだけで、Lambda関数は処理できるペースで順番に処理します。アップロード側がサムネイル生成の完了を待つ必要がないため、ユーザーの操作がブロックされません。

### パターン2: SNS + SQS のファンアウト

前のセクションで学んだファンアウトパターンです。1つのイベントを複数の独立した処理に分配する場合に使います。SNSトピックに複数のSQSキューをサブスクライブさせることで、各処理が独立して並行実行されます。

前述のECサイトの例に加えて、ログ処理にも使えます。アプリケーションのログをSNSトピックに発行すると、1つのSQSキューではリアルタイム監視用のLambdaが起動し、別のSQSキューではログをS3にアーカイブする処理が走り、さらに別のキューでは異常検知の分析処理が行われる、といった構成です。

### パターン3: EventBridge による複数サービスの連携

EventBridgeを中心に据えたイベント駆動パターンです。さまざまなAWSサービスやアプリケーションで発生するイベントを、EventBridgeで一元的に受け取り、ルールに基づいて適切な処理に振り分けます。

たとえば、EC2インスタンスの状態変化イベントをEventBridgeで受け取り、「インスタンスが停止した」場合はLambda関数で自動復旧を試み、同時にSNSトピック経由で運用チームにメール通知する、という構成が考えられます。「インスタンスが起動した」というイベントは無視する、といったフィルタリングもEventBridgeのルールで実現できます。

### パターン4: Step Functions によるオーケストレーション

第2章で概要を学んだAWS Step Functionsについて、ここでもう少し詳しく見ていきましょう。

Step Functionsは、複数のAWSサービスを組み合わせた「ワークフロー（処理の流れ）」を定義・実行・管理するサービスです。第2章で「オーケストラの指揮者」に例えたことを覚えているでしょうか。指揮者が楽譜に従って各演奏者に指示を出すように、Step Functionsはステートマシン（状態遷移図）に従って各サービスに処理を指示します。

### ステートマシンの構成要素

ステートマシンは、ASL（Amazon States Language）というJSONベースの言語で定義します。ビジュアルエディタ（Workflow Studio）も用意されており、コードを書かずにドラッグ&ドロップで設計することもできます。

ステートマシンは、いくつかの「ステート（状態）」を組み合わせて作ります。主なステートの種類を紹介します。

Task ステートは、実際の処理を行うステートです。Lambda関数を呼び出したり、DynamoDBにデータを書き込んだり、SQSにメッセージを送信したりします。ワークフローの中で「仕事をする」部分です。

Choice ステートは、条件に応じて次の処理を分岐させるステートです。「もし在庫があれば決済処理へ、なければ在庫切れ通知へ」というような条件分岐を表現します。信号機でいえば、「歩行者ボタンが押されたら青に変わる」というルールに相当します。

Parallel ステートは、複数の処理を同時に実行するステートです。「在庫確認」と「クレジットカード認証」を同時に行い、両方が完了してから次に進む、といった並行処理を表現します。

Wait ステートは、指定した時間だけ待機するステートです。「決済完了後、5分待ってから確認メールを送る」といった時間差のある処理に使います。

### Standard ワークフロー vs Express ワークフロー

処理の性質は大きく「じっくり確実に進めたいもの」と「大量に素早くさばきたいもの」に分かれます。Step Functionsはこの2つに対応するために、Standard と Express の2つのワークフロータイプを用意しています。

Standard ワークフローは、最大1年間実行できる長時間のワークフロー向けです。実行履歴が詳細に記録され、処理の途中経過を確認できます。料金は「状態遷移の回数」に基づいて課金されます。注文処理や承認フローなど、長時間かかる可能性がある業務プロセスに向いています。

Express ワークフローは、最大5分間の短時間ワークフロー向けです。大量のリクエストを高スループットで処理できます。料金は「実行回数と実行時間」に基づいて課金されます。IoTデータの処理やリアルタイムのストリーミングデータ変換など、大量かつ短時間の処理に向いています。

### いつ Step Functions を選ぶか

「複数のサービスを順番に、あるいは条件分岐を入れながら実行したい」場合にStep Functionsが適しています。

SQSやSNSは「メッセージを受け渡す」サービスですが、Step Functionsは「処理全体の流れをコントロールする」サービスです。たとえば「注文受付 → 在庫確認 → 在庫ありなら決済 → 決済成功なら発送手配 → メール通知」という一連のフローを、エラーハンドリングや条件分岐も含めて一元管理したい場合に力を発揮します。

SQSやSNSが「仲介役（メールボックス）」だとすれば、Step Functionsは「プロジェクトマネージャー」のような存在です。全体の進捗を管理し、次に何をすべきかを指示し、問題が起きたときの対応方針も持っています。

---

> **試験のポイント**
> - SQS + Lambda = 最もシンプルな疎結合の非同期処理パターン
> - SNS + SQS = ファンアウトパターン（1つのイベントを複数の処理に分配）
> - EventBridge = イベントのフィルタリングとルーティングが必要な場合に選択
> - Step Functions = 複数ステップの処理全体を管理する「オーケストレーション」サービス
> - Step Functions の Standard（長時間・詳細履歴）と Express（短時間・高スループット）の違い
> - 「オーケストレーション」「ワークフロー管理」→ Step Functions を連想する

## 章末確認問題

### 問題1

ECサイトの注文処理システムで、注文受付サービスと在庫管理サービスが直接通信しています。在庫管理サービスに障害が発生すると、注文受付サービスも停止してしまいます。この問題を解決するために最も適切なアプローチはどれですか？

- **A.** 在庫管理サービスのEC2インスタンスのサイズを大きくする
- **B.** Amazon SQS キューを間に挟んで、サービス間を疎結合にする
- **C.** 在庫管理サービスを別のリージョンにデプロイする
- **D.** Amazon CloudFront を導入してキャッシュを活用する

<details>
<summary>解答・解説</summary>

**解答: B**

解説: 密結合の問題（1つのサービスの障害が他に波及する）を解決するには、SQSキューなどの仲介役を挟んで疎結合にすることが有効です。注文受付サービスはSQSにメッセージを送信し、在庫管理サービスが復旧したらキューからメッセージを取り出して処理します。EC2のスケールアップ（A）は障害そのものの解決にはなりません。

</details>

### 問題2

Amazon SQS の FIFO キューの特徴として正しいものはどれですか？

- **A.** ほぼ無制限のスループットを持つ
- **B.** メッセージの順序が保証され、1回のみ配信される
- **C.** メッセージは最低1回配信されるが、重複する可能性がある
- **D.** メッセージの保持期間は最大1日である

<details>
<summary>解答・解説</summary>

**解答: B**

解説: FIFO（First In, First Out）キューは、メッセージの順序保証と1回のみの配信（exactly-once processing）が特徴です。Aはstandardキューの特徴、CもStandardキューの特徴です。SQSのメッセージ保持期間はデフォルト4日、最大14日です。

</details>

---

### 問題3

1つの注文イベントを、在庫更新、決済処理、メール通知の3つのサービスに同時に配信したい場合、最も適切なアーキテクチャはどれですか？

- **A.** Amazon SQS キューを3つ作成し、注文サービスから各キューに直接メッセージを送信する
- **B.** Amazon SNS トピックに発行し、3つの SQS キューをサブスクライブする
- **C.** Amazon EventBridge で3つのルールを作成する
- **D.** AWS Step Functions で3つのタスクを順番に実行する

<details>
<summary>解答・解説</summary>

**解答: B**

解説: 1つのイベントを複数のサービスに同時配信する「ファンアウトパターン」には、SNS + SQS の組み合わせが最適です。SNSトピックに1回メッセージを発行するだけで、サブスクライブしている全SQSキューにメッセージが配信されます。Aの方式では注文サービスが各キューの存在を知っている必要があり、密結合になります。CのEventBridgeでもルーティングは技術的に可能ですが、「同一メッセージを複数のキューに確実に配信する」典型的なファンアウトにはSNS+SQSが最もシンプルで、試験ではこちらが正解として求められます。Dは順番に実行するため「同時配信」には適しません。

</details>

### 問題4

Amazon SQS と Amazon SNS の違いについて正しい説明はどれですか？

- **A.** SQS はプッシュ型、SNS はポーリング型である
- **B.** SQS はポーリング型、SNS はプッシュ型である
- **C.** どちらもプッシュ型で、メッセージを自動配信する
- **D.** どちらもポーリング型で、受信者がメッセージを取りに行く

<details>
<summary>解答・解説</summary>

**解答: B**

解説: SQSはポーリング型で、コンシューマーが定期的にキューを確認してメッセージを取り出します。SNSはプッシュ型で、メッセージが発行されると自動的にすべてのサブスクライバーに配信されます。この違いはSQSとSNSの使い分けの基本です。

</details>

### 問題5

Amazon EventBridge の特徴として正しいものはどれですか？（2つ選択）

- **A.** メッセージをキューに溜めて順番に処理する
- **B.** イベントの内容に基づいて柔軟にフィルタリングとルーティングができる
- **C.** 外部SaaSサービスからのイベントを直接受け取れる
- **D.** FIFO順序保証が組み込まれている
- **E.** メッセージの可視性タイムアウト機能がある

<details>
<summary>解答・解説</summary>

**解答: B, C**

解説: EventBridgeの大きな特徴は、イベントの内容に基づく柔軟なフィルタリング・ルーティング（B）と、Zendesk、Datadogなどの外部SaaSサービスとの直接統合（C）です。Aはメッセージキュー（SQS）の説明です。D（FIFO順序保証）とE（可視性タイムアウト）はSQSの機能です。

</details>

### 問題6

処理の順序が重要で、同じメッセージが二重に処理されることを防ぎたい場合、どのサービスを選択すべきですか？

- **A.** Amazon SQS Standard キュー
- **B.** Amazon SQS FIFO キュー
- **C.** Amazon SNS
- **D.** Amazon EventBridge

<details>
<summary>解答・解説</summary>

**解答: B**

解説: SQS FIFOキューは、メッセージの順序保証（First In, First Out）と1回のみの配信（exactly-once processing）を提供します。Standardキューは順序保証がなく、メッセージが重複する可能性があります。SNSやEventBridgeはメッセージの順序保証や重複排除の仕組みを持ちません。

</details>

### 問題7

Amazon EventBridge は、以前は何というサービス名でしたか？

- **A.** Amazon MQ
- **B.** Amazon CloudWatch Events
- **C.** AWS AppSync
- **D.** Amazon Kinesis

<details>
<summary>解答・解説</summary>

### 問題8

複数のAWSサービス（Lambda関数など）を組み合わせたワークフローを定義し、条件分岐やエラーハンドリングを含めて一元管理したい場合、最も適切なサービスはどれですか？

- **A.** Amazon SQS
- **B.** Amazon SNS
- **C.** Amazon EventBridge
- **D.** AWS Step Functions

<details>
<summary>解答・解説</summary>

**解答: D**

解説: AWS Step Functionsは、複数のサービスを組み合わせたワークフロー（処理の流れ）を、ステートマシンとして定義・実行・管理するサービスです。条件分岐（Choiceステート）、並行処理（Parallelステート）、エラーハンドリングなどを含む複雑な処理フローを一元管理できます。SQS・SNS・EventBridgeはメッセージの受け渡しや通知を担うサービスであり、ワークフロー全体の管理には向いていません。

</details>

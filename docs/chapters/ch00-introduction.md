## 0-1. クラウドとは何か

### このセクションで学ぶこと

「クラウド」という言葉の正確な意味と、オンプレミス（自社運用）との違いを理解します。

### オンプレミスの課題

クラウドが登場する前、企業がシステムを作るには次のような手順が必要でした。

1. サーバーやネットワーク機器を購入する（数百万円〜数千万円）
2. データセンターを用意し、機器を設置する（2〜3ヶ月かかることも）
3. OSやミドルウェアをインストールして設定する
4. 故障したら自分たちで修理・交換する

これでは、新しいビジネスを始めようとしても、インフラの準備だけで半年近くかかってしまいます。また、アクセスが増えたときに素早く対応できません。

### クラウドとは

クラウドコンピューティングとは、**インターネット経由でコンピューターリソース（サーバー、ストレージ、ネットワークなど）を必要なときに必要な分だけ利用できる仕組み**です。

電気と同じイメージです。自家発電機を自分で買って管理する必要はありません。電力会社から、使った分だけ料金を払って利用します。

### IaaS / PaaS / SaaS の違い

クラウドサービスは、「どこまでクラウドに任せるか」によって3つのモデルに分類されます。

| モデル | 意味 | 代表例 | 管理の担当 |
|--------|------|--------|-----------|
| **IaaS** | Infrastructure as a Service | Amazon EC2 | インフラ（サーバー・ネットワーク）をAWSが管理。OSより上は自分で管理 |
| **PaaS** | Platform as a Service | AWS Elastic Beanstalk | OSやミドルウェアもAWSが管理。アプリのコードだけ書けばよい |
| **SaaS** | Software as a Service | Gmail、Zoom、Amazon WorkSpaces | アプリごと提供される。ユーザーはただ使うだけ |

覚え方のコツ: 「IaaS < PaaS < SaaS」の順番で、AWSに任せる範囲が広くなります。

> **図解:** ch00-cloud-service-models.drawio.png — IaaS/PaaS/SaaSとオンプレミスの管理範囲の比較（積み木図）
> （draw.io ファイルは `docs/images/ch00-cloud-service-models.drawio.png` に配置予定）

### クラウドの主なメリット

- **初期コストなし**: サーバーを買わずに始められる
- **従量課金**: 使った分だけ支払う（アイドル時間のコスト削減）
- **素早い展開**: 数分でサーバーを用意できる
- **グローバル展開**: 世界中のリージョンにすぐ展開できる
- **スケーラビリティ**: アクセス増減に合わせて自動で規模を調整できる

---

> **試験のポイント**
> - IaaS / PaaS / SaaS の区別は頻出。「どのサービスがどのモデルに該当するか」が問われる
> - EC2は IaaS、Elastic Beanstalk は PaaS に近いサービス
> - クラウドの利点として「資本的支出（CapEx）から運用コスト（OpEx）への移行」というキーワードも重要
> - オンプレミスとの比較で「俊敏性（Agility）」「弾力性（Elasticity）」も覚えておく

## 0-2. AWSグローバルインフラストラクチャ

### このセクションで学ぶこと

AWSが世界中に展開するインフラの構成（リージョン・AZ・エッジロケーション）を理解します。

### リージョン

**リージョン**とは、AWSが世界各地に設けた**地理的なデータセンター群のエリア**です。2024年時点で30以上のリージョンが存在します（東京、大阪、シンガポール、米国東部など）。

リージョンを選ぶ際の判断基準は4つあります。

1. **レイテンシー（応答速度）**: ユーザーに近いリージョンを選ぶと速くなる
2. **法規制・データ主権**: 国によってはデータを国内に置く必要がある
3. **コスト**: リージョンによって料金が異なる
4. **サービス提供状況**: 新しいサービスはまず米国東部から展開されることが多い

### アベイラビリティゾーン（AZ）

**アベイラビリティゾーン（AZ）**とは、1つのリージョン内にある**物理的に分離されたデータセンター群**です。

重要なポイントをまとめます。

- 1つのリージョンには**2つ以上のAZ**がある（多くは3〜6個）
- AZどうしは**物理的に離れた場所**に設置されている（数km〜数十km）
- 高速な専用線で接続されている
- 1つのAZが災害や停電で止まっても、他のAZは稼働し続ける

複数のAZにリソースを配置することで、障害への耐性（高可用性）を高められます。これを**マルチAZ構成**と呼びます。

### エッジロケーション

**エッジロケーション**とは、世界400か所以上に分散して設置された**コンテンツ配信のための拠点**です。

リージョンとは異なり、フルなAWSサービスは動きません。主にAmazon CloudFront（CDN）やRoute 53（DNS）がエッジロケーションを使います。

| | リージョン | AZ | エッジロケーション |
|-|-----------|----|--------------------|
| 役割 | AWSサービスの提供エリア | 物理的に独立したデータセンター群 | コンテンツ配信の中継点 |
| 数 | 30以上（2024年時点） | リージョンあたり2〜6個 | 400か所以上 |
| 主な用途 | EC2、S3など全般 | 高可用性の確保 | CloudFront、Route 53 |

> **図解:** ch00-global-infrastructure.drawio.png — リージョン・AZ・エッジロケーションの関係（地図ベースの構成図）
> （draw.io ファイルは `docs/images/ch00-global-infrastructure.drawio.png` に配置予定）

---

> **試験のポイント**
> - 「1リージョン = 2つ以上のAZ」は暗記必須
> - AZは「物理的に離れたデータセンター群」という表現を押さえる
> - エッジロケーションはリージョンより数が多い（400以上 vs 30以上）
> - リージョンとエッジロケーションを混同しないこと

## 0-3. Well-Architected Framework

### このセクションで学ぶこと

AWSがクラウドアーキテクチャのベストプラクティスとしてまとめた「Well-Architected Framework」の6つの柱を理解します。

### 6つの柱

Well-Architected Framework は、優れたクラウドシステムを設計するための指針を6つの柱に整理しています。

| 柱 | 概要 |
|----|------|
| **1. 運用上の優秀性 (Operational Excellence)** | 運用プロセスを継続的に改善し、ビジネス価値を提供する |
| **2. セキュリティ (Security)** | 情報・システムを保護し、リスクを管理する |
| **3. 信頼性 (Reliability)** | 障害から自動回復し、要求されたワークロードを実行する |
| **4. パフォーマンス効率 (Performance Efficiency)** | リソースを効率的に使い、需要の変化に対応する |
| **5. コスト最適化 (Cost Optimization)** | 最低限のコストでビジネス価値を提供する |
| **6. 持続可能性 (Sustainability)** | 環境への影響を最小化する（2021年に追加された新しい柱） |

各柱の詳細は後続の章で扱います。この段階では**名前と概要を把握する**だけで十分です。

---

> **試験のポイント**
> - 6つの柱の名前をすべて答えられるようにしておく
> - 「持続可能性」は2021年に追加された比較的新しい柱
> - 具体的なサービスや設計原則との対応は各章で学ぶ

## 0-4. 責任共有モデル

### このセクションで学ぶこと

AWSと利用者の「責任の分担」を理解します。これは試験で最も頻繁に問われるトピックの1つです。

### AWSの責任と顧客の責任

クラウドを使う場合、すべての管理をAWSに任せるわけではありません。AWSと顧客の責任範囲が明確に分かれています。

**AWSの責任（クラウド自体のセキュリティ）**

AWSはクラウドを構成する物理的なインフラを責任を持って管理します。

- 物理的なデータセンターのセキュリティ
- ハードウェア（サーバー、ネットワーク機器）の管理
- ハイパーバイザー（仮想化ソフトウェア）の管理
- AWSが提供するマネージドサービス（RDSのOSパッチなど）

**顧客の責任（クラウドにおけるセキュリティ）**

顧客はクラウド上に配置したリソースを責任を持って管理します。

- EC2インスタンスのOSのパッチ適用
- アプリケーションのセキュリティ
- データの暗号化
- IAMユーザーのアクセス管理
- セキュリティグループの設定

### サービスモデル別の責任範囲

サービスモデルによって、責任の境界が変わります。

| | IaaS（例: EC2） | PaaS（例: Elastic Beanstalk） | SaaS |
|-|----------------|-------------------------------|------|
| 物理インフラ | AWS | AWS | AWS |
| ハイパーバイザー | AWS | AWS | AWS |
| OS | **顧客** | AWS | AWS |
| ミドルウェア | **顧客** | AWS | AWS |
| アプリケーション | **顧客** | **顧客** | AWS |
| データ | **顧客** | **顧客** | **顧客** |

> **図解:** ch00-shared-responsibility.drawio.png — 責任共有モデルの境界図（AWSと顧客の責任範囲をIaaS/PaaS/SaaSごとに表示）
> （draw.io ファイルは `docs/images/ch00-shared-responsibility.drawio.png` に配置予定）

### 具体的な例で確認

- **「物理サーバーのハードウェア障害への対応」** → **AWSの責任**
- **「EC2インスタンスのWindowsOSのセキュリティパッチ適用」** → **顧客の責任**
- **「RDS（マネージドDB）のOSパッチ適用」** → **AWSの責任**（マネージドサービスのため）
- **「S3バケットのアクセス権限設定」** → **顧客の責任**

---

> **試験のポイント**
> - 「クラウドのセキュリティ（AWSの責任）」と「クラウドにおけるセキュリティ（顧客の責任）」という表現を押さえる
> - 物理インフラは常にAWSの責任
> - EC2のOSは顧客の責任（マネージドサービスのOSはAWSの責任）
> - データは常に顧客の責任

## 0-5. AWSの操作方法

### このセクションで学ぶこと

AWSリソースを操作するための4つの方法と、Infrastructure as Code（IaC）の概念を理解します。

### 4つの操作方法

AWSリソースは、以下の4つの方法で操作できます。

**1. AWSマネジメントコンソール**

Webブラウザからアクセスするグラフィカルな管理画面です。直感的に操作でき、初心者に最も使いやすい方法です。

**2. AWS CLI（Command Line Interface）**

ターミナル（コマンドプロンプト）からコマンドを入力してAWSを操作する方法です。自動化や繰り返し作業に向いています。

```
# EC2インスタンスの一覧を表示する例
aws ec2 describe-instances
```

**3. AWS CloudShell**

マネジメントコンソールのブラウザ上で直接CLIコマンドを実行できる環境です。**ローカルへのインストールや設定が不要**なのが特徴です。

> **CloudShell = 「ブラウザ上でCLIをすぐに使える環境」**

**4. SDK（Software Development Kit）**

Python、Java、JavaScript などのプログラミング言語からAWSを操作するためのライブラリです。アプリケーションにAWSの機能を組み込む際に使います。

| 操作方法 | 特徴 | 向いている用途 |
|---------|------|---------------|
| マネジメントコンソール | GUIで直感的 | 設定確認、学習 |
| CLI | コマンドで操作 | 自動化、スクリプト |
| CloudShell | ブラウザ上のCLI | ローカル設定なしでCLIを使いたい時 |
| SDK | プログラムから操作 | アプリへの組み込み |

### Infrastructure as Code（IaC）とCloudFormation

**Infrastructure as Code（IaC）**とは、インフラの構成をコード（テキストファイル）で定義する考え方です。

IaCのメリット:
- **再現性**: 同じ環境を何度でも作れる
- **バージョン管理**: コードの変更履歴を管理できる
- **自動化**: コードを実行するだけでインフラが構築される

**AWS CloudFormation**は、AWSが提供するIaCサービスです。JSONやYAMLで書いたテンプレートファイルを使って、AWSリソースをまとめて自動構築できます。

> CloudFormation の詳細は第11章（CI/CDとデベロッパーツール）で扱います。

---

> **試験のポイント**
> - 4つの操作方法（コンソール・CLI・CloudShell・SDK）を覚える
> - CloudShell = 「ブラウザ上からCLIを実行できる、ローカル設定不要」が頻出
> - CloudFormation = 「テンプレートからインフラを自動構築するIaCサービス」

## 章末確認問題

### 問題1

AWSの責任共有モデルにおいて、**AWSが責任を持つ**のはどれですか？

- **A.** Amazon EC2 インスタンスにインストールされているゲストOSのパッチ適用
- **B.** Amazon S3 バケットへのアクセス権限の設定
- **C.** データセンターの物理的なセキュリティ
- **D.** IAMユーザーのパスワードポリシーの設定

<details>
<summary>解答・解説</summary>

**解答: C**

解説: データセンターの物理的なセキュリティはAWSの責任（クラウドのセキュリティ）です。EC2のOSパッチ、S3のアクセス権限、IAMの設定はすべて顧客の責任（クラウドにおけるセキュリティ）です。

</details>

### 問題3

AWSのクラウドサービスモデルの分類として、**Amazon EC2**はどれに該当しますか？

- **B.** PaaS
- **C.** IaaS

<summary>解答・解説</summary>

### 問題6

エッジロケーションとリージョンの違いについて、正しい説明はどれですか？

- **A.** エッジロケーションの数はリージョンよりも少ない
- **B.** エッジロケーションでは EC2 インスタンスを起動できる
- **C.** エッジロケーションは主にコンテンツ配信（CloudFront）に使われる
- **D.** エッジロケーションはリージョンと同じ機能を持つ

<details>
<summary>解答・解説</summary>

**解答: C**

解説: エッジロケーションは主に CloudFront（CDN）や Route 53（DNS）のために使われる、コンテンツ配信の中継拠点です。リージョンより数が多く（400以上）、EC2のようなフルなサービスは動きません。

</details>

### 問題7

Amazon RDS（マネージドデータベース）を利用している場合、**顧客の責任**はどれですか？

- **A.** RDSのデータベースエンジンのOSパッチ適用
- **B.** データベースへのアクセス権限とユーザー管理
- **C.** RDSが動作する物理サーバーのハードウェア管理
- **D.** データセンターの電源設備の管理

<details>
<summary>解答・解説</summary>

**解答: B**

解説: RDS はマネージドサービスのため、OSパッチ・ハードウェア・電源管理はAWSの責任です。データベースへのアクセス権限（誰がDBに接続できるか）の設定は**顧客の責任**です。

</details>

### 問題8

クラウドコンピューティングの利点として、正しいものはどれですか？

- **A.** 初期費用（資本的支出）が大きくかかる
- **B.** 需要に応じてリソースを自動的に増減できる
- **C.** データセンターの物理的な管理が顧客の責任となる
- **D.** 利用料金はすべて一律の月額固定料金である

<details>
<summary>解答・解説</summary>

**解答: B**

解説: クラウドの大きな利点の1つは**弾力性（Elasticity）**——需要に応じてリソースを自動的に増減できることです。初期費用は最小限（従量課金）、物理管理はAWS、料金は使用量に応じた従量課金です。

</details>

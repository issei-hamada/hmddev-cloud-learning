## 9-1. AWS のセキュリティモデル

### このセクションで学ぶこと

責任共有モデルをより深く理解し、多層防御とゼロトラストという2つの重要なセキュリティの考え方を学びます。

### 責任共有モデルの深掘り

第0章で、AWSと顧客の責任範囲が分かれている「責任共有モデル」を学びました。ここではもう少し具体的に、サービスごとの責任分界点を見ていきましょう。

ポイントは、サービスが「マネージド」であるほど、AWSが面倒を見てくれる範囲が広がるということです。

たとえば、EC2（仮想マシン）を使う場合、OSのセキュリティパッチ適用は顧客の責任です。一方、RDS（マネージドデータベース）を使う場合は、OSのパッチ適用はAWSが担当します。同じ「データベース」でも、EC2にMySQLを自分でインストールした場合と、RDSを使った場合では、責任範囲が変わるのです。

もう一つ重要なのは、**データの管理は常に顧客の責任**であるという点です。どのサービスを使っていても、データの暗号化を有効にするかどうか、誰にアクセスを許可するかは、顧客が決めなければなりません。

試験では、「この操作の責任はAWSと顧客のどちらか？」という形で問われます。迷ったときは、「物理的なことはAWS、論理的な設定は顧客」と覚えておくと判断しやすくなります。

### 多層防御の考え方

セキュリティの世界では、「1つの対策だけに頼らない」ことが鉄則です。これを「多層防御」と呼びます。

自宅のセキュリティで考えてみましょう。玄関の鍵だけでは不安ですよね。オートロックのエントランス、防犯カメラ、窓の補助錠、ホームセキュリティの通報システムなど、何重もの対策を組み合わせることで安全性を高めます。仮に1つの対策が破られても、次の対策で食い止められるわけです。

AWSでも同じ考え方が使われます。ネットワーク層ではVPCやセキュリティグループで通信を制御し、アプリケーション層ではWAFでWeb攻撃を防ぎ、データ層ではKMSで暗号化し、監視層ではGuardDutyで不審な動きを検知します。このように複数のレイヤーで防御することで、全体のセキュリティを高めるのです。

![多層防御のレイヤー図](../images/ch09-defense-in-depth.drawio.png)

### ゼロトラストの概念

従来のセキュリティは、「社内ネットワークの中は安全、外は危険」という境界型の考え方でした。城壁の中にいれば安全、という発想です。

しかし、クラウドの時代になると、この考え方では不十分です。社員がリモートワークでカフェから接続したり、外部パートナーがシステムにアクセスしたりと、「内側」と「外側」の境界が曖昧になっています。

そこで登場したのが「ゼロトラスト」という考え方です。これは「誰も信用しない、常に確認する」というアプローチです。社内ネットワークからのアクセスであっても、毎回身元を確認し、必要最小限のアクセスだけを許可します。

AWSでは、IAMによる厳密なアクセス制御や、MFA（多要素認証）の活用、最小権限の原則がゼロトラストの実践にあたります。「信用できる相手かどうか」ではなく「その操作に必要な権限を持っているかどうか」で判断するのです。

具体的な実践例としては、IAMポリシーの条件設定で接続元のIPアドレスを制限したり、VPCエンドポイント（第5章で学んだ仕組み）を使ってインターネットを経由せずにAWSサービスにアクセスしたりする手法があります。

---

> **試験のポイント**
> - 責任共有モデルの具体例を押さえる。「物理セキュリティはAWS」「データの暗号化は顧客」
> - マネージドサービス（RDSなど）とアンマネージドサービス（EC2にDB構築）では責任範囲が異なる
> - 多層防御は「複数の防御レイヤーを組み合わせる」こと
> - ゼロトラスト = 「常に検証し、最小権限で運用する」考え方

## 9-2. AWS IAM の基本

### このセクションで学ぶこと

IAM（Identity and Access Management）の4つの基本要素と、ルートユーザーの正しい管理方法を学びます。IAMは試験で最も頻出のサービスです。

### なぜアクセス管理が必要なのか

会社のオフィスを想像してください。全員が同じ鍵を持っていたらどうなるでしょう。経理担当でないのに金庫を開けられる人がいたり、退職した人がまだ入館できたりしたら大問題です。

ITシステムでも同じです。「誰が」「何に」「どんな操作を」できるかを厳密に管理しなければ、情報漏洩や不正操作のリスクが生まれます。AWSでこのアクセス管理を担うのがIAM（アイアム）です。IAMは追加料金なしで利用できます。

### IAM の4つの基本要素

IAMには、「ユーザー」「グループ」「ロール」「ポリシー」という4つの基本要素があります。これらの関係を会社組織にたとえて理解しましょう。

**IAMユーザー**は、AWSにアクセスする個々の人やアプリケーションに対応するアカウントです。会社でいえば、社員一人ひとりに発行する社員証のようなものです。ユーザー名とパスワード（コンソール利用時）、またはアクセスキーとシークレットキーの組み合わせ（プログラムからの利用時）で認証されます。

**IAMグループ**は、ユーザーをまとめるための仕組みです。会社の「部署」にあたります。たとえば「開発部」グループを作り、開発に必要な権限をグループに付与すれば、そのグループに属する全員が同じ権限を持てます。メンバーが増えても、グループに追加するだけで済むので管理が楽になります。

**IAMロール**は、一時的に権限を付与するための仕組みです。社員証というより「来客用の入館証」に近いイメージです。EC2インスタンスやLambda関数など、AWSサービス同士のアクセスにもロールが使われます。ロールは一時的な認証情報を提供するため、ユーザーのアクセスキーを直接コードに埋め込むよりも安全です。この一時的な認証情報は、裏側でAWS STS（Security Token Service）というサービスが発行しています。STSが発行する認証情報には有効期限があるため、万が一漏洩しても一定時間で無効になります。

**IAMポリシー**は、「何を許可し、何を拒否するか」を定義したルールです。会社でいえば「入館規則」や「情報管理規程」にあたります。ポリシーは、ユーザー、グループ、ロールに「アタッチ（取り付け）」して使います。

![IAM の概念関係図](../images/ch09-iam-relationships.drawio.png)

### IAMポリシーの構造

IAMポリシーはJSON（ジェイソン）と呼ばれるテキスト形式で記述されます。JSONとは、データの構造を人間にも読みやすい形で表現するフォーマットです。試験では細かな文法を書く必要はありませんが、基本的な構造は理解しておきましょう。ポリシーの核となるのは次の3つの要素です。

**Effect**（効果）: 「Allow」（許可する）か「Deny」（拒否する）のどちらかです。

**Action**（操作）: どのような操作を対象にするかを指定します。たとえば「s3:GetObject」はS3からファイルを読み取る操作です。

**Resource**（対象リソース）: どのリソースに対する操作かを指定します。特定のS3バケットだけを対象にする、といった指定ができます。

たとえば、「特定のS3バケットからの読み取りだけを許可する」ポリシーは次のようになります。

```json
{
  "Effect": "Allow",
  "Action": "s3:GetObject",
  "Resource": "arn:aws:s3:::my-bucket/*"
}
```

Effectが「Allow」で許可を意味し、Actionが「s3:GetObject」でS3からの読み取り操作を指定し、Resourceで対象のバケットを限定しています。試験ではこのような細かい記述を書く必要はありませんが、3つの要素の意味が分かっていれば問題を読み解けます。

また、ポリシーには大きく2つの種類があることも知っておきましょう。**アイデンティティベースポリシー**は、IAMユーザー、グループ、ロールにアタッチするポリシーです。「この人（やサービス）は何ができるか」を定義します。一方、**リソースベースポリシー**は、S3バケットなどのリソース側にアタッチするポリシーです。「このリソースに誰がアクセスできるか」を定義します。試験では、この2種類の違いを問われることがあります。

ポリシー評価の仕組みも押さえておきましょう。AWSがアクセスを判断する際のルールは次の通りです。まず、明示的なDeny（拒否）が最も強く、常に優先されます。次に、明示的なAllow（許可）が確認されます。どちらも設定されていなければ、デフォルトでDeny（拒否）になります。つまり、「明示的Deny > 明示的Allow > デフォルトDeny」という優先順位です。

### 最小権限の原則

IAMで最も重要な考え方が「最小権限の原則」です。これは、各ユーザーやサービスに、業務に必要な最小限の権限だけを与えるという原則です。

たとえるなら、ホテルのカードキーです。宿泊客には自分の部屋のカードキーだけが渡されます。全部屋を開けられるマスターキーは渡しません。もし宿泊客が何か問題を起こしても、被害は自分の部屋だけに限定されます。

最初から全権限を与えてしまうと、誤操作で重要なデータを削除してしまったり、アカウントが乗っ取られた場合に被害が拡大したりするリスクがあります。「まずは最小限の権限から始め、必要に応じて追加する」が正しいアプローチです。

### ルートユーザーの管理

AWSアカウントを作成したときに最初にできるアカウントが「ルートユーザー」です。ルートユーザーは、そのアカウント内のすべての操作が可能な「万能の管理者」です。

ルートユーザーの管理には、3つの鉄則があります。

1つ目は、**日常業務には使わない**ことです。ルートユーザーは権限が強すぎるため、日常的に使うと誤操作や乗っ取りのリスクが高まります。日常の作業にはIAMユーザーを作成して使います。

2つ目は、**MFA（多要素認証）を必ず有効にする**ことです。MFAとは、パスワードに加えて、スマートフォンのアプリなどで生成されるワンタイムコードも入力しないとログインできないようにする仕組みです。

3つ目は、ルートユーザーの**アクセスキーを作成しない**ことです。アクセスキーが漏洩すると、プログラムから全権限で操作されてしまう危険があります。

### ルートユーザーでのみ可能な操作

ほとんどの操作はIAMユーザーに権限を付与すれば実行できますが、一部の操作はルートユーザーでしか行えません。試験で問われやすいものを挙げます。

- AWSアカウントの閉鎖
- AWSサポートプランの変更
- IAMユーザーへの請求情報アクセスの有効化
- リザーブドインスタンスマーケットプレイスへの出品登録

これらは「アカウント全体に影響する重要な操作」という共通点があります。試験で「ルートユーザーでのみ可能な操作はどれか」と問われたら、アカウントの存在そのものや請求に関わる操作を探しましょう。

---

> **試験のポイント**
> - IAMユーザーは個人、グループは部署、ロールは一時的な権限付与、ポリシーはルール
> - ロールは「一時的な認証情報」を提供する。EC2やLambdaにはロールを使う（アクセスキーを埋め込まない）
> - ポリシーの評価順序: 明示的Deny > 明示的Allow > デフォルトDeny
> - 最小権限の原則 = 必要最低限の権限だけを付与する
> - ルートユーザーは日常利用しない、MFAを有効にする、アクセスキーを作らない
> - ルートユーザーでのみ可能な操作（アカウント閉鎖、サポートプラン変更など）は頻出

## 9-3. IAM Identity Center と Amazon Cognito

### このセクションで学ぶこと

組織の従業員向けのアクセス管理と、アプリケーションのエンドユーザー向けの認証という、2つの異なる認証ニーズに対応するサービスを理解します。

### IAM Identity Center（旧 AWS SSO）

IAM Identity Centerは、組織の従業員が複数のAWSアカウントや業務アプリケーションに、1回のログインでアクセスできるようにするサービスです。いわゆる「シングルサインオン（SSO）」の仕組みです。

会社の社員証にたとえると分かりやすいでしょう。社員証1枚あれば、ビル全体のどのフロアにも入れますし、食堂でも使えるし、社内システムにもログインできます。IAM Identity Centerはこの社員証の役割を果たし、1つのIDで複数のサービスにアクセスできるようにします。

AWS Organizationsと連携して使うことで、組織内の複数のAWSアカウントへのアクセスを一元管理できます。「社員Aは開発アカウントにアクセスできるが、本番アカウントには読み取り専用」といった制御が一箇所でできます。

### Amazon Cognito

Amazon Cognitoは、Webアプリケーションやモバイルアプリのエンドユーザー（一般のお客さん）の認証を管理するサービスです。

ECサイトやスマートフォンアプリで「Googleアカウントでログイン」「Appleでサインイン」といったボタンを見たことがあると思います。Cognitoを使えば、こうしたソーシャルログインの仕組みを簡単にアプリに組み込めます。もちろん、メールアドレスとパスワードでの独自のユーザー登録・ログイン機能も提供できます。

### 2つのサービスの使い分け

IAM Identity Centerは「社員のAWSアクセス管理」、Cognitoは「アプリユーザーのログイン機能」と覚えてください。

たとえば、ある会社が社内の開発者にAWSコンソールへのアクセスを許可する場合はIAM Identity Centerを使います。一方、その会社が運営するECサイトに一般のお客さんがログインする機能を作る場合はCognitoを使います。

---

> **試験のポイント**
> - IAM Identity Center = 組織の従業員向けSSO（複数AWSアカウントへの一元的なアクセス管理）
> - Amazon Cognito = アプリケーションのエンドユーザー向け認証（ソーシャルログイン対応）
> - 「社員がAWSにアクセス」→ IAM Identity Center、「一般ユーザーがアプリにログイン」→ Cognito

## 9-4. 暗号化と鍵管理

### このセクションで学ぶこと

データを暗号化する意味と方法、そしてAWSが提供する暗号鍵管理サービスを理解します。

### なぜ暗号化が必要なのか

暗号化とは、データを特殊な方法で変換して、鍵を持っている人以外には読めないようにする仕組みです。日記に鍵をかけるのと同じ発想です。万が一、誰かが日記帳を盗んでも、鍵がなければ中身を読めません。

クラウドにデータを預ける以上、暗号化は非常に重要です。たとえデータが不正に取得されても、暗号化されていれば中身を見ることはできません。

### 保存時の暗号化と転送時の暗号化

暗号化には大きく2つの種類があります。

**保存時の暗号化（encryption at rest）**は、S3やEBSなどに保存されているデータを暗号化するものです。金庫の中に入れた書類にさらに鍵をかけるようなイメージです。保存されているデータ自体が暗号化されているので、仮にストレージが盗まれても中身は読めません。

**転送時の暗号化（encryption in transit）**は、データがネットワーク上を移動する際に暗号化するものです。TLS/SSL（Webサイトのhttpsが使っている仕組み）がその代表例です。手紙を郵送する際に封筒に入れるようなもので、配送中に中身を覗かれるのを防ぎます。

試験では「保存時」と「転送時」の暗号化を区別する問題が頻出です。「データの保管中のセキュリティ」なのか「データの通信中のセキュリティ」なのかを正確に判断しましょう。

### AWS KMS（Key Management Service）

KMSは、暗号鍵を安全に作成・管理するためのマネージドサービスです。金庫の鍵を専門の管理会社に預けるようなイメージです。

KMSの特徴は、暗号鍵そのものをAWSの安全な環境で管理してくれることです。利用者は「このデータを暗号化して」「このデータを復号して」と指示するだけで、鍵の管理の細かい部分はKMSが面倒を見てくれます。

S3、EBS、RDSなど多くのAWSサービスがKMSと連携しており、暗号化のオプションを有効にするだけで保存時の暗号化を実現できます。

KMSの暗号鍵には、AWSが管理する鍵（AWS管理キー）と、顧客が作成・管理する鍵（カスタマー管理キー）があります。AWS管理キーはAWSサービスが自動的に作成・管理してくれるので手軽です。カスタマー管理キーでは、鍵の有効化・無効化やローテーション間隔の設定など、顧客がより細かく制御できます。試験では「顧客が鍵のライフサイクルを管理したい場合はカスタマー管理キー」と覚えておきましょう。

KMSでは暗号鍵のローテーション（定期的な鍵の更新）も自動化できます。鍵を定期的に変更することで、万が一古い鍵が漏洩しても、影響を最小限に抑えられます。

### AWS CloudHSM

CloudHSMは、専用のハードウェア（HSM = Hardware Security Module）を使って暗号鍵を管理するサービスです。

KMSとの違いは、鍵を管理するハードウェアが完全に専用であるという点です。KMSはAWSが共有の基盤上で鍵を管理しますが、CloudHSMは顧客専用の物理的なハードウェアを提供します。

金庫のたとえでいえば、KMSは「銀行の貸金庫」で、CloudHSMは「自分専用の金庫室をレンタルする」ようなものです。CloudHSMは、金融業界や政府機関など、規制上より厳格な鍵管理が求められる場合に使われます。

### AWS Secrets Manager

Secrets Managerは、データベースのパスワードやAPIキーなど、アプリケーションが使う「秘密情報」を安全に保管・管理するサービスです。

アプリケーションのコードにパスワードを直接書き込んでしまうのは、非常に危険です。コードが漏洩すれば、パスワードも一緒に漏れてしまいます。Secrets Managerに保管しておけば、アプリケーションは実行時にSecrets Managerから安全にパスワードを取得できます。

パスワードの自動ローテーション（定期的な変更）もサポートしているので、パスワードの管理が楽になります。

### AWS Certificate Manager（ACM）

ACMは、TLS/SSL証明書（Webサイトをhttpsにするために必要な電子証明書）を簡単に作成・管理できるサービスです。ACMで発行した証明書は無料で、AWSのサービス（ELB、CloudFrontなど）で利用できます。証明書の有効期限が近づくと自動的に更新してくれるので、更新忘れによるサイトの障害を防げます。

---

> **試験のポイント**
> - 「保存時の暗号化」と「転送時の暗号化（TLS/SSL）」の区別は頻出
> - KMS = マネージドな暗号鍵管理（ほとんどのAWSサービスと連携可能）
> - CloudHSM = 専用ハードウェアでの鍵管理（厳格な規制要件に対応）
> - Secrets Manager = パスワードやAPIキーの安全な保管（自動ローテーション対応）
> - Certificate Manager = TLS/SSL証明書の無料発行と自動更新

---

## 9-5. ネットワークセキュリティ

### このセクションで学ぶこと

Webアプリケーションへの攻撃を防ぐAWS WAF、DDoS攻撃から守るAWS Shield、そしてファイアウォールルールを一元管理するAWS Firewall Managerを学びます。

### セキュリティグループとNACLの復習

第5章で、VPCのネットワークセキュリティとして「セキュリティグループ」と「ネットワークACL（NACL）」を学びました。ここでセキュリティの観点から簡単に復習しておきましょう。

セキュリティグループは「各部屋のドアの鍵」にあたります。EC2インスタンスごとに設定でき、通信の許可ルールを定義します。新しく作成したカスタムセキュリティグループでは、デフォルトですべてのインバウンド通信を拒否し、明示的に許可したものだけが通過します（VPC作成時に自動生成される「デフォルトセキュリティグループ」とは挙動が異なります）。大事な特徴として、セキュリティグループは「ステートフル」です。つまり、行きの通信を許可すれば、戻りの通信は自動的に許可されます。

NACLは「マンション入口のオートロック」にあたります。サブネット単位で設定し、許可ルールと拒否ルールの両方を定義できます。NACLは「ステートレス」なので、行きの通信と戻りの通信を別々にルール設定する必要があります。

### AWS WAF（Web Application Firewall）

WAFは、Webアプリケーションへの悪意ある攻撃を防ぐサービスです。

セキュリティグループやNACLはIPアドレスやポート番号に基づいて通信を制御しますが、WAFはもっと賢く、通信の「中身」を見て判断します。たとえるなら、セキュリティグループが「入館証を確認するガードマン」だとすれば、WAFは「持ち込み荷物をX線検査する手荷物検査員」です。

WAFが防御できる攻撃の代表例としては、SQLインジェクション（データベースを不正に操作しようとする攻撃）やクロスサイトスクリプティング（XSS、Webページに悪意のあるプログラムコードを埋め込む攻撃）があります。

WAFは、CloudFront（CDN）やALB（ロードバランサー）、API Gatewayの前に配置して使います。

### AWS Shield

AWS Shieldは、DDoS攻撃（大量のアクセスを一斉に送りつけてサービスを停止させる攻撃）からAWSリソースを守るサービスです。

Shieldには2つのプランがあります。

**Shield Standard**は、すべてのAWSアカウントに無料で自動適用されています。特に何もしなくても、基本的なDDoS防御が働いています。これは試験で非常によく問われるポイントです。

**Shield Advanced**は、有料の上位プランで、より高度なDDoS防御と、24時間365日のAWS DDoSレスポンスチーム（DRT）によるサポートが受けられます。DDoS攻撃によるスケーリングで増えたコストの補填も含まれます。

### AWS Firewall Manager

Firewall Managerは、AWS Organizations（複数のAWSアカウントを一元管理する仕組み）と連携して、WAFルールやセキュリティグループのルールを組織全体で一元管理するサービスです。

たとえば、「組織内のすべてのアカウントで、WAFの特定のルールセットを適用する」といった統一的な管理が可能になります。アカウントが多い大規模組織で特に役立ちます。

### AWS Network Firewall

AWS Network Firewallは、VPC内の通信をきめ細かく制御するマネージドファイアウォールサービスです。WAFがWebアプリケーション層（HTTPリクエストの中身）を検査するのに対し、Network FirewallはVPCのネットワーク層の通信を制御します。

たとえるなら、WAFが「お店の入口での手荷物検査」だとすれば、Network Firewallは「ビル全体の出入りを管理する守衛室」のようなものです。不正な通信先へのアクセスをブロックしたり、悪意のあるパケットを検知したりできます。

---

> **試験のポイント**
> - WAF = SQLインジェクション・XSSなどのWebアプリケーション層の攻撃を防御
> - Shield Standard = 全AWSアカウントに無料で自動適用（DDoS防御）。これは超頻出
> - Shield Advanced = 有料。DRTサポート、コスト補填あり
> - Firewall Manager = 組織全体でWAFルール等を一元管理
> - Network Firewall = VPCのネットワーク層の通信を制御するマネージドファイアウォール
> - セキュリティグループ（ステートフル）vs NACL（ステートレス）の違いも再確認

---

## 9-6. 脅威検知と脆弱性管理

### このセクションで学ぶこと

不正な活動を検知し、システムの脆弱性を発見するためのAWSサービスを学びます。

### なぜ検知が必要なのか

どんなに防御を固めても、100%の安全はありません。泥棒が入らないように鍵をかけるだけでなく、防犯カメラを設置して不審な動きを監視することも大切です。AWSでも同じように、防御だけでなく「何か起きたときに素早く気づく」仕組みが欠かせません。

### Amazon GuardDuty

GuardDuty（ガードデューティ）は、AWSアカウント内の不審な活動を機械学習を使って自動検知するサービスです。

たとえるなら「AI搭載の防犯カメラ」です。普段の行動パターンを学習し、いつもと違う動き――たとえば深夜に普段使わないリージョンからのログインや、通常ありえない量のデータ転送――を検知するとアラートを出します。

GuardDutyの大きな特徴は、有効化するだけで動作することです。エージェントのインストールや複雑な設定は不要で、VPCフローログ、CloudTrailログ、DNSログなどを自動的に分析してくれます。

### Amazon Inspector

Inspector（インスペクター）は、EC2インスタンスやコンテナの脆弱性（セキュリティ上の弱点）を自動的にスキャンするサービスです。

建物でいえば「耐震診断」のようなものです。構造に問題がないか、弱いところがないかを専門家がチェックしてくれます。Inspectorは、OSに既知のセキュリティ上の問題がないか、ソフトウェアに更新が必要な脆弱性がないかを継続的にチェックし、修正が必要な項目を優先度付きで報告してくれます。

### Amazon Macie

Macie（メイシー）は、S3バケットに保存されたデータの中から、個人情報（PII）やクレジットカード番号などの機密データを自動的に検出するサービスです。

たとえるなら「書類審査の専門スタッフ」です。大量の書類（S3のデータ）の中から、「これは個人情報が含まれています」「クレジットカード番号が暗号化されずに保存されています」と教えてくれます。意図せず機密データが保存されてしまっている場合に、素早く気づくことができます。

### Amazon Detective

Detective（ディテクティブ）は、セキュリティインシデントが発生した後に、原因を調査・分析するためのサービスです。

GuardDutyが「防犯カメラで不審者を検知する」サービスなら、Detectiveは「検知した後に防犯カメラの映像を巻き戻して、不審者がどこから来てどこに行ったかを追跡する探偵」です。GuardDutyが出したアラートを起点に、関連するログやデータを可視化して、何が起きたのかを把握しやすくします。

---

> **試験のポイント**
> - GuardDuty = 機械学習ベースの脅威検知。有効化するだけで動作する
> - Inspector = EC2やコンテナの脆弱性スキャン
> - Macie = S3の機密データ（個人情報等）の検出
> - Detective = セキュリティインシデントの調査・分析
> - それぞれの「検出対象」の違いを区別する: 不審な活動（GuardDuty）、脆弱性（Inspector）、機密データ（Macie）、インシデントの原因（Detective）

---

## 9-7. セキュリティの一元管理とコンプライアンス

### このセクションで学ぶこと

複数のセキュリティサービスの結果を統合管理する方法と、AWSのコンプライアンス対応に関するサービスを学びます。

### AWS Security Hub

Security Hubは、GuardDuty、Inspector、Macieなど複数のセキュリティサービスの検出結果を一つのダッシュボードにまとめて表示するサービスです。

たとえるなら「警備本部の統合モニター」です。防犯カメラ（GuardDuty）、耐震センサー（Inspector）、書類審査（Macie）の情報がバラバラに報告されると大変ですが、Security Hubがあれば一箇所で全体の状況を把握できます。

Security Hubは単に結果をまとめるだけでなく、AWS自身が推奨するセキュリティのベストプラクティスに照らして、自動的にセキュリティ状態を評価してくれる機能も備えています。

### AWS Organizations の SCP（サービスコントロールポリシー）

セキュリティの一元管理という観点で、AWS OrganizationsのSCP（サービスコントロールポリシー）についても触れておきます。SCPは、組織全体で「この操作は禁止する」というガードレールを設定する仕組みです。

たとえば、「どのアカウントでも特定のリージョンは使用禁止」といったルールをSCPで定めると、たとえ個々のIAMポリシーで許可されていても、その操作は実行できません。SCPとIAMポリシーの両方で許可されていないと操作は通らないのです。Organizations自体の詳細は第10章で扱いますが、セキュリティの文脈ではSCPの存在を覚えておきましょう。

### AWS Artifact

Artifact（アーティファクト）は、AWSのコンプライアンスに関する公式レポートや契約書をダウンロードできるセルフサービスのポータルです。

AWSが取得しているISO認証やSOCレポートなどのコンプライアンス認証レポートを入手したいとき、Artifactにアクセスすれば無料でダウンロードできます。「AWSはこのセキュリティ基準を満たしていますか？」という監査対応で必要になる資料がここにあります。

注意点として、Artifactで入手できるのは「AWS側」のコンプライアンスレポートです。顧客自身のコンプライアンス準拠を証明するものではありません。

### AWS Audit Manager

Audit Manager（オーディットマネージャー）は、AWSリソースの利用状況がコンプライアンス基準に準拠しているかを継続的に評価し、監査に必要な証拠を自動的に収集するサービスです。

監査（いわゆる「内部監査」や「外部監査」）の際に、「このルールに従っていることを証明してください」と求められることがあります。Audit Managerは、その証拠資料の収集を自動化してくれるので、手作業で資料を集める手間を大幅に削減できます。

---

> **試験のポイント**
> - Security Hub = 複数のセキュリティサービスの結果を一元表示するダッシュボード
> - Artifact = AWSのコンプライアンス認証レポートをダウンロードする場所
> - Audit Manager = 監査用の証拠収集を自動化するサービス
> - SCP = 組織全体で操作を制限するガードレール。SCPとIAMの両方で許可されないと操作できない
> - 「AWSのコンプライアンスレポートが欲しい」→ Artifact

---

## 9-8. その他のセキュリティサービス

### このセクションで学ぶこと

試験で概要レベルで問われる可能性のある、その他のセキュリティ関連サービスを把握します。

### AWS Directory Service

Directory Serviceは、既存のMicrosoft Active Directory（AD）とAWSを連携させるためのサービスです。

多くの企業は、社内のユーザー管理にActive Directoryを使っています。Directory Serviceを使えば、社内のADのアカウント情報を使ってAWSのリソースにアクセスできるようになります。新たにAWS用のアカウントを作り直す必要がないので、管理の手間が省けます。

### AWS RAM（Resource Access Manager）

RAMは、AWSリソースを他のAWSアカウントと安全に共有するためのサービスです。

たとえば、同じ組織内の複数のアカウントでVPCのサブネットを共有したい場合に使います。リソースをコピーしたり移動したりする必要がなく、「共有」するだけで済むため、管理が効率的になります。

ここまで多くのセキュリティサービスを学びました。最後に、各サービスがセキュリティの「予防」「検知」「対応」「コンプライアンス」のどの領域をカバーしているかを全体像として確認しましょう。

### 問題2

IAMにおいて、EC2インスタンスがS3バケットにアクセスする必要がある場合、最も推奨されるアプローチはどれですか？

- **A.** ルートユーザーのアクセスキーをEC2インスタンスに設定する
- **B.** IAMユーザーを作成し、アクセスキーをEC2のコードに埋め込む
- **C.** S3バケットへのアクセス権限を持つIAMロールをEC2インスタンスにアタッチする
- **D.** S3バケットのアクセスを全員に公開する

<details>
<summary>解答・解説</summary>

**解答: C**

</details>

### 問題4

次のIAMポリシーの評価について正しいものはどれですか？

あるIAMユーザーに対して、「S3へのフルアクセスを許可するポリシー」と「S3の特定バケットへのアクセスを拒否するポリシー」の両方がアタッチされている場合、そのバケットへのアクセスはどうなりますか？

- **A.** 許可される（Allowが適用される）
- **B.** 拒否される（明示的Denyが優先される）

<summary>解答・解説</summary>

**解答: B**

</details>

### 問題10

<details>
<summary>解答・解説</summary>

**解答: C**

解説: **AWS Artifact**は、AWSが取得しているコンプライアンス認証レポートや契約書をセルフサービスでダウンロードできるポータルです。Aはセキュリティダッシュボード、Bは監査証拠の自動収集、DはAWSリソースの設定変更の記録・評価サービスです。

</details>

### 問題11

ある企業が、厳格な規制要件により暗号鍵を専用のハードウェアで管理する必要があります。最も適切なサービスはどれですか？

- **A.** AWS KMS
- **B.** AWS CloudHSM
- **C.** AWS Secrets Manager
- **D.** AWS Certificate Manager

<details>
<summary>解答・解説</summary>

**解答: B**

解説: **AWS CloudHSM**は、顧客専用のハードウェアセキュリティモジュール（HSM）を提供し、暗号鍵を専用ハードウェアで管理できるサービスです。金融業界や政府機関など、規制上厳格な鍵管理が求められる場合に適しています。Aはマネージド型（共有基盤上）の鍵管理、Cはパスワード等の秘密情報管理、DはTLS/SSL証明書の管理です。

</details>

### 問題12

次のセキュリティサービスと役割の組み合わせとして正しいものはどれですか？（2つ選択）

- **A.** AWS WAF — DDoS攻撃からの保護
- **B.** Amazon Inspector — EC2インスタンスの脆弱性スキャン
- **C.** Amazon Detective — セキュリティインシデントの調査・分析
- **D.** AWS Shield Standard — SQLインジェクション攻撃の防御
- **E.** Amazon Macie — VPCフローログの分析

<details>
<summary>解答・解説</summary>

**解答: B、C**

解説: **Amazon Inspector**はEC2インスタンスやコンテナの脆弱性スキャンを行うサービスで、**Amazon Detective**はセキュリティインシデント発生後の調査・分析を行うサービスです。AのDDoS防御はShieldの役割、DのSQLインジェクション防御はWAFの役割、EのVPCフローログ分析はGuardDutyが行います。サービス名と役割を入れ替えたひっかけ問題に注意しましょう。

</details>

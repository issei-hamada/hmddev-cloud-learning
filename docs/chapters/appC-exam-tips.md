# 付録C　試験対策のポイント ― 合格への最短ルート

**対応試験ドメイン**: 全ドメイン横断
**推定読了時間**: 60分

---

## この章で学ぶこと

ここまでの全章で学んできたAWSの知識を、試験合格という目標に向けて整理します。出題傾向の把握、ドメインごとの学習優先度、よくある「ひっかけ問題」のパターン、そして学習スケジュールの組み立て方まで、試験対策に必要な情報をまとめました。最後に総合模擬問題20問に取り組むことで、本番への自信をつけてください。

---

## C-1. 試験の概要と出題傾向

### このセクションで学ぶこと

AWS認定クラウドプラクティショナー試験（CLF-C02）の基本情報と、出題の特徴を把握します。「敵を知る」ことが合格への第一歩です。

### 試験の基本情報

まずは試験の基本的なスペックを押さえておきましょう。

試験時間は90分です。問題数は全部で65問ありますが、このうち採点対象になるのは50問です。残りの15問はAWSが将来の出題のためにテストしている「採点対象外」の問題で、どの問題が対象外かは受験者にはわかりません。つまり、すべての問題に真剣に取り組む必要があります。

合格スコアは1000点満点中700点です。すべての問題が同じ配点とは限りませんが、ざっくり言えば7割程度の正答率が合格ラインの目安になります。

受験料は100 USDです。日本語での受験が可能で、試験会場（テストセンター）での受験と、自宅からのオンライン受験の両方に対応しています。

### 出題ドメインと配分

試験は4つの分野（ドメイン）から出題されます。それぞれの配分は以下のとおりです。

第1分野「クラウドのコンセプト」は全体の24%を占めます。クラウドの基本的な考え方、メリット、IaaS/PaaS/SaaSの違いなどが問われます。

第2分野「セキュリティとコンプライアンス」は全体の30%です。IAMや責任共有モデルなど、セキュリティに関する出題が最も多いドメインです。

第3分野「クラウドテクノロジーとサービス」は全体の34%で、最大の出題範囲です。EC2やS3をはじめとする主要サービスの特徴や使い分けが問われます。

第4分野「請求、料金、サポート」は全体の12%で、出題数は最も少ないですが、確実に得点したい分野です。

![ドメイン別学習優先度チャート](../images/appC-domain-priorities.drawio.png)

### 問題の特徴

問題形式は「択一選択問題」と「複数選択問題」の2種類です。択一選択問題は4つの選択肢から正解を1つ選ぶ形式。複数選択問題は5つの選択肢から正解を2つ選ぶ形式が中心です。複数選択問題では「2つ選べ」と明示されるので、見落とさないようにしましょう。

試験の大きな特徴は、シナリオベースの問題が多いことです。「ある企業がこういう課題を抱えています。どのサービスを使うべきですか？」という形で出題されます。サービス名を暗記するだけでなく、「どんな場面で使うのか」を理解していることが大切です。この本で学んできた「なぜそのサービスが必要なのか」という視点が、そのまま試験対策になります。

> **試験のポイント**: 採点対象外の15問があるため、「この問題は簡単すぎる」「難しすぎる」と感じても気にせず、すべてに全力で取り組みましょう。また、時間配分は「1問あたり約80秒」が目安です。わからない問題は「フラグ」を立てて後から見直す戦略が有効です。

## C-2. ドメイン別の学習優先度と頻出ポイント

### このセクションで学ぶこと

4つのドメインそれぞれで、特にどこを重点的に学習すべきかを整理します。配点の大きいドメインから優先的に取り組むことで、効率よく合格に近づけます。

### セキュリティとコンプライアンス（30% ― 最重要ドメイン）

配点30%のこのドメインは、合否を分ける最重要分野です。以下の5つのテーマは特に頻出ですので、確実に押さえてください。

1つ目はIAM（Identity and Access Management）です。「誰が」「何に」「どんな操作を」できるかを管理するしくみで、ユーザー、グループ、ロール、ポリシーの違いが問われます。特に「IAMロールはサービスやアプリケーションに一時的な権限を付与するしくみ」という点は重要です。詳しくは[第9章 セキュリティ](ch09-security.md)で学びました。

2つ目は責任共有モデルです。「AWSが責任を持つ範囲」と「お客さまが責任を持つ範囲」を正しく区別できるかが問われます。たとえば、物理的なデータセンターのセキュリティはAWSの責任ですが、IAMユーザーのパスワード管理はお客さまの責任です。

3つ目はAWS Shield Standardです。DDoS攻撃（大量のアクセスでサービスを麻痺させる攻撃）からの保護サービスで、すべてのAWSアカウントに無料で自動適用されます。「追加の設定なしで有効」という点がよく出題されます。

4つ目はAmazon GuardDutyです。AIを使ってAWS環境の脅威を検知するサービスです。CloudTrailのログやVPCフローログなどを自動分析し、不審なアクティビティを検出します。

5つ目はAWS WAF（Web Application Firewall）です。SQLインジェクションやXSS（クロスサイトスクリプティング）といったWebアプリケーションへの攻撃を防御するファイアウォールです。ShieldがDDoS攻撃（大量アクセスによる攻撃）を防ぐのに対し、WAFはWebアプリケーションの脆弱性を突く攻撃に対応します。この2つの使い分けは試験で頻出です。

### クラウドテクノロジーとサービス（34% ― 最多出題ドメイン）

出題数が最も多いドメインです。サービスの種類が多いため、「どの場面でどのサービスを使うか」の使い分けが特に重要です。

EC2の購入オプションはほぼ確実に出題されます。オンデマンド（いつでも使える定価）、リザーブドインスタンス（1年または3年契約、支払い方法により最大72%割引。特定のインスタンスタイプに固定される点に注意。なお、AWSは現在より柔軟なSavings Plansを推奨しています）、スポットインスタンス（最大90%割引だが中断される可能性あり）、Savings Plans（使用量をコミットして割引）の違いを整理しておきましょう。詳しくは[第1章 Webアプリケーションのホスティング](ch01-web-hosting.md)で学びました。

S3のストレージクラスも頻出です。S3 Standard（頻繁にアクセスするデータ）、S3 Standard-IA（たまにアクセスするデータ）、S3 Glacier（アーカイブ用、取り出しに時間がかかる）などの使い分けが問われます。詳しくは[第3章 ストレージ](ch03-storage.md)をご覧ください。

RDS（リレーショナルデータベース）とDynamoDB（NoSQLデータベース）の違いもよく出題されます。RDSはデータ間の関連性が重要なケース（受注管理など）、DynamoDBは大量のデータを高速に読み書きするケース（ゲームのスコア管理など）に向いています。詳しくは[第4章 データベース](ch04-database.md)で学びました。

サーバーレスのコンセプトも重要です。AWS Lambda、Amazon API Gateway、DynamoDB、S3などを組み合わせた「サーバー管理が不要なアーキテクチャ」の概念を理解しておきましょう。詳しくは[第2章 サーバーレス](ch02-serverless.md)をご覧ください。

### クラウドのコンセプト（24%）

クラウドの根本的な考え方を問うドメインです。この分野は「理解」が重要で、暗記だけでは得点しにくい特徴があります。

IaaS（サーバーやネットワークを借りる）、PaaS（アプリの実行環境を借りる）、SaaS（完成したアプリを使う）の違いは必須知識です。AWSのサービスがこの3つのどれに当てはまるかを判断できるようにしておきましょう。

AZ（アベイラビリティーゾーン）とリージョンの関係も頻出です。リージョンは地理的に離れた拠点（東京、大阪など）で、各リージョンは複数のAZ（独立したデータセンター群）で構成されています。「マルチAZ構成にすると可用性が高まる」理由を説明できるようにしておきましょう。詳しくは[第7章 高可用性と耐障害性](ch07-resilience.md)で学びました。

DR（災害復旧）戦略の4パターンも押さえておきましょう。バックアップ＆リストア（最低コスト・復旧に時間がかかる）、パイロットライト（最小構成を維持）、ウォームスタンバイ（縮小版を常時稼働）、マルチサイトアクティブ/アクティブ（最高コスト・ほぼゼロダウンタイム）の4段階です。

### 請求、料金、サポート（12%）

出題数は少ないですが、比較的覚えやすい分野なので、確実に得点源にしたいドメインです。

データ転送の課金ルールは必ず覚えてください。AWSへのデータ転送（インバウンド）は無料、AWSからのデータ転送（アウトバウンド）は有料です。郵便で例えると、「届けてもらうのは無料だけど、送るときは切手代がかかる」というイメージです。

AWS無料利用枠の3つのタイプも押さえましょう。「常に無料」（Lambda の月100万リクエストなど）、「12ヶ月無料」（EC2 t2.microなど、アカウント作成から12ヶ月限定）、「トライアル」（特定サービスの短期無料体験）の3種類があります。

サポートプランの比較も出題されます。ベーシック（無料）、デベロッパー、ビジネス、エンタープライズ On-Ramp、エンタープライズの5種類があり、特に「Trusted Advisorのすべてのチェックが使えるのはビジネスプラン以上」「TAM（テクニカルアカウントマネージャー）が付くのはエンタープライズプランのみ」という点が頻出です。詳しくは[第15章 コスト管理](ch15-cost.md)で学びました。

> **試験のポイント**: 学習の優先順位は「テクノロジー（34%）→ セキュリティ（30%）→ コンセプト（24%）→ 請求（12%）」の順で時間を割り振るのが効率的です。ただし、請求分野は覚えれば確実に得点できるので、最後に短時間で仕上げましょう。

## C-3. よく出るひっかけ問題パターン

### このセクションで学ぶこと

試験では、似たようなサービスや概念を混同させる「ひっかけ問題」がよく出ます。ここでは特に間違えやすいポイントを9つ取り上げます。これらの違いを明確に理解しておくだけで、数問は確実に正答できるはずです。

### パターン1: マルチAZ vs リードレプリカ

この2つはどちらもRDS（データベース）に関連する機能ですが、目的がまったく違います。

マルチAZ配置は「可用性（止まらないこと）」のためのしくみです。メインのデータベースに障害が起きたとき、別のAZにあるスタンバイに自動で切り替わります。普段、スタンバイはユーザーからのアクセスを受け付けません。

一方、リードレプリカは「読み取りパフォーマンスの向上」が目的です。データベースのコピーを作り、読み取り処理を分散させることで、メインのデータベースの負担を軽くします。

試験では「データベースの可用性を高めたい」→ マルチAZ、「読み取りが遅い」→ リードレプリカ、という使い分けが問われます。

### パターン2: Shield Standard vs Shield Advanced

AWS Shieldは2つのプランがあり、特にStandardについての出題が多いです。

Shield Standardは、すべてのAWSアカウントに無料で自動適用されます。追加の設定や契約は不要で、基本的なDDoS攻撃からの保護を提供します。

Shield Advancedは月額3,000 USDからの有料プランで、より高度なDDoS保護、24時間365日のDDoS対応チーム（SRT: Shield Response Team。DDoS攻撃への対応を支援する専門チーム）への連絡、DDoS攻撃によるコスト増加の補填などが含まれます。

ひっかけのポイントは「Shield Standardは申し込みが必要か？」という聞き方です。答えは「不要（自動で有効）」です。

### パターン3: CloudWatch vs CloudTrail vs AWS Config

この3つは監視・管理系サービスですが、「何を見ているか」が異なります。

CloudWatch は「リソースのパフォーマンス監視」です。CPU使用率、メモリ使用量、エラー率などのメトリクスを収集し、アラームを設定できます。心電図モニターのイメージです。

CloudTrail は「操作の記録（監査ログ）」です。「誰が」「いつ」「何をしたか」というAPI呼び出し（プログラムやコンソールからAWSサービスに対して行う操作リクエスト）の履歴を記録します。防犯カメラのイメージです。

AWS Config は「リソースの設定変更の記録と評価」です。「セキュリティグループの設定がルールに違反していないか」といった構成管理を行います。衛生検査官のイメージです。

試験では「不正アクセスの調査」→ CloudTrail、「CPU使用率の監視」→ CloudWatch、「設定がルールに準拠しているか」→ AWS Config、という使い分けが問われます。詳しくは[第10章 モニタリング](ch10-monitoring.md)をご覧ください。

### パターン4: SQS Standard vs SQS FIFO

Amazon SQS（メッセージキュー）には2つのタイプがあります。

Standardキューは「最低1回配信」で「順序保証なし」です。同じメッセージが2回届く可能性があり、送った順番どおりに届くとは限りません。ただし、スループット（処理能力）がほぼ無制限です。

FIFOキューは「正確に1回配信」で「順序保証あり」です。メッセージの重複がなく、送った順番どおりに届きます。ただし、デフォルトでは1秒あたり300メッセージ（バッチ処理で3,000）という上限があります（高スループットモードではさらに高い処理能力を利用可能です）。

「メッセージの順序が重要な処理」→ FIFO、「大量のメッセージを高速処理」→ Standard、と覚えましょう。詳しくは[第6章 非同期処理](ch06-async.md)で学びました。

### パターン5: ルートユーザーでしかできない操作

AWSアカウントを作成したときに使ったメールアドレスとパスワードでログインするのが「ルートユーザー」です。普段の作業にはIAMユーザーを使うべきですが、ルートユーザーでしかできない操作がいくつかあります。

代表的なものは、AWSアカウントの解約、ルートユーザーのメールアドレス変更、IAMユーザーの請求情報へのアクセス許可設定、サポートプランの変更などです。

試験では「この操作にはどのユーザーが必要か？」という形で出題されます。「ルートユーザーでしかできない操作」の代表例を覚えておけば対応できます。

### パターン6: データ転送の課金ルール

データ転送の課金ルールは、直感と少し異なる部分があるので注意が必要です。

基本ルールは「インバウンド（AWSへの転送）は無料、アウトバウンド（AWSからの転送）は有料」です。ただし、同じAZ内のデータ転送は無料、異なるAZ間のデータ転送は有料という点も覚えておきましょう。

また、同じリージョン内のS3からCloudFrontへのデータ転送は無料です。CloudFrontを使うことでデータ転送コストを削減できるケースが出題されることがあります。

### パターン7: Dedicated Hosts vs Dedicated Instances

どちらも「他のAWSアカウントと物理サーバーを共有しない」という点では同じですが、細かい違いがあります。

Dedicated Hosts（専有ホスト）は、物理サーバー単位で借りるしくみです。物理サーバーの配置を制御でき、ソケット数やコア数を確認できます。ソフトウェアのライセンス条件で「物理サーバーの情報が必要」なケースに適しています。

Dedicated Instances（ハードウェア専有インスタンス）は、インスタンス単位で借りるしくみです。物理サーバーの細かい情報は得られませんが、他のアカウントと物理ハードウェアを共有しません。

試験では「既存のソフトウェアライセンスを持ち込みたい（BYOL: Bring Your Own License。すでに自社で購入済みのソフトウェアライセンスをAWS上でも使い続けること。たとえば、会社で買ったWindows Serverのライセンスをそのまま使いたい場合など）」→ Dedicated Hosts、「単に他のアカウントと物理サーバーを共有したくない」→ Dedicated Instances、という使い分けが問われます。

### パターン8: Aurora の MySQL互換 vs PostgreSQL互換のパフォーマンス

Amazon Auroraは、MySQLとPostgreSQLに互換性のあるリレーショナルデータベースですが、パフォーマンスの数値が異なります。

MySQL互換のAuroraは、標準的なMySQLの最大5倍のパフォーマンスを発揮します。PostgreSQL互換のAuroraは、標準的なPostgreSQLの最大3倍のパフォーマンスです。

この「5倍」と「3倍」の数字が入れ替えて出題されることがあるので、正確に覚えておきましょう。

### パターン9: Organizations vs SCP vs IAMポリシー

この3つはいずれも「権限の管理」に関連しますが、管理の範囲と目的が異なります。

AWS Organizationsは、複数のAWSアカウントをまとめて管理するサービスです。一括請求（コンソリデーティッドビリング）により、複数アカウントの利用料金をまとめて支払うこともできます。

SCP（サービスコントロールポリシー）は、Organizations内のアカウントに対して「やってはいけないこと」の上限を設定するしくみです。たとえば「このアカウントでは東京リージョン以外のリソースを作成できない」といった制限をかけられます。SCPはアカウント全体に適用される「天井」のようなもので、個々のIAMユーザーの権限がSCPの範囲を超えることはできません。

IAMポリシーは、個別のユーザーやロールに対して「何ができるか」を定義するしくみです。

試験では「複数アカウントの一元管理」→ Organizations、「アカウント単位の最大権限の制限」→ SCP、「個別ユーザーへの権限付与」→ IAMポリシー、という使い分けが問われます。

> **試験のポイント**: ひっかけ問題に対する最良の対策は、「似ているサービスをセットで覚える」ことです。「AとBの違いは何か？」と自問自答しながら学習すると、出題者のワナにはまりにくくなります。

## C-4. 学習スケジュールのモデル例

### このセクションで学ぶこと

試験日までの学習計画を立てましょう。「いつまでに何を終わらせるか」が明確になると、不安が減り、着実に前進できます。ここでは3つのモデルプランを紹介します。ご自身のスケジュールに合わせて選んでください。

![学習スケジュールのモデル例](../images/appC-study-schedule.drawio.png)

### 2週間集中コース（1日2〜3時間）

短期集中で合格を目指すプランです。IT業界での経験がある方や、すでにAWSを少し使ったことがある方に向いています。

1〜3日目は、第0章（クラウドの基礎）と第1〜4章（コンピューティング、サーバーレス、ストレージ、データベース）を一気に読みます。テクノロジー分野の基礎をここで固めます。

4〜5日目は、第5〜8章（ネットワーク、非同期処理、可用性、コンテナ）を学びます。テクノロジー分野の残りを押さえます。

6〜7日目は、第9章（セキュリティ）を重点的に学びます。配点30%の最重要分野なので、丸2日かける価値があります。

8〜9日目は、第10〜11章（モニタリング、CI/CD）と第12〜13章（データ分析、AI/ML）を学びます。

10〜11日目は、第14〜16章（移行、コスト管理、その他のサービス）を学びます。

12〜13日目は、この付録Cのひっかけ問題パターンを復習し、模擬問題に取り組みます。弱点が見つかったら該当する章に戻って復習します。

14日目は、最終確認です。各章の「試験のポイント」だけを流し読みし、不安な箇所を集中的に見直します。

### 1ヶ月標準コース（1日1〜1.5時間）

最もバランスの良いプランです。IT初心者の方でも無理なく進められます。

第1週（1〜7日目）は、第0〜4章を読みます。1日1章ペースで、クラウドの基礎とコアサービスを学びます。各章の確認問題にも取り組みましょう。

第2週（8〜14日目）は、第5〜9章を読みます。ネットワーク、可用性、セキュリティといった重要分野をじっくり学びます。特に第9章には2日かけてください。

第3週（15〜21日目）は、第10〜16章を読みます。モニタリング、CI/CD、データ分析、AI/ML、移行、コスト管理をカバーします。

第4週（22〜30日目）は、復習と模擬問題の期間です。付録Cの内容を確認し、模擬問題に取り組みます。間違えた分野は該当する章に戻って復習します。最後の2〜3日は「試験のポイント」の総復習に充てましょう。

### 2ヶ月じっくりコース（1日30分〜1時間）

IT未経験の方や、じっくり確実に理解しながら進めたい方向けのプランです。

第1〜2週は、第0〜4章を読みます。1日30分〜1時間のペースで、2〜3日に1章のペースです。わからない部分があっても焦らず、確認問題で理解度をチェックしながら進めましょう。

第3〜4週は、第5〜9章を読みます。ネットワークの章は用語が多いので、理解しにくければ飛ばして後から戻っても大丈夫です。セキュリティの章は特に丁寧に読みましょう。

第5〜6週は、第10〜16章を読みます。サービスの数が多くなりますが、「このサービスは何のために使うのか」を一言で説明できるレベルを目指しましょう。

第7週は、付録Cのひっかけ問題パターンを学び、模擬問題に取り組みます。

第8週は、弱点の補強と最終確認です。模擬問題で間違えた分野を中心に復習し、各章の「試験のポイント」を総ざらいします。

### どのコースでも共通のアドバイス

どのコースを選んでも、以下の4点を意識すると効果的です。

1つ目は、章末の確認問題は必ずやることです。問題を解くことで、読んだだけでは気づかない理解の抜けが見つかります。

2つ目は、間違えた問題は「なぜ間違えたか」を必ず確認することです。「そもそも知らなかった」のか「似たサービスと混同した」のかで、復習の仕方が変わります。

3つ目は、完璧を目指さないことです。全サービスを細部まで覚える必要はありません。合格ラインは7割です。頻出ポイントを確実に押さえることが、最も効率の良い合格戦略です。

4つ目は、AWS公式の無料学習リソースを活用することです。AWS Skill Builderでは無料のオンラインコースが多数公開されていますし、Exam Prep Official Question Set（公式練習問題集）で本番に近い形式の問題を無料で解くことができます。公式教材は試験の出題傾向を最も正確に反映しているので、ぜひ活用してください。

> **試験のポイント**: 試験勉強は「1回で完璧に覚える」より「繰り返し触れる」ほうが定着します。1章を完璧に読み込むよりも、まず全体を一通り読んでから、重要な章を繰り返し読むほうが効果的です。

### 解答と解説

<details>
<summary>問1の解答</summary>

**正解: C. Rehost**

コードを変更せずにそのままクラウドに移す方法を「Rehost（リホスト）」と呼びます。「リフト＆シフト」とも呼ばれます。Refactorはアプリケーションをクラウドネイティブに書き換える方法、Replatformはアプリケーションの中核は変えずに一部のコンポーネントだけクラウド向けに変更する方法です。詳しくは[第14章 移行](ch14-migration.md)をご覧ください。
</details>

<details>
<summary>問2の解答</summary>

**正解: C. IAMユーザーのアクセス管理**

責任共有モデルでは、データセンターの物理セキュリティ、ハイパーバイザーの管理、ハードウェアの廃棄はAWSの責任です。お客さまはOS以上のレイヤーの管理が責任範囲で、IAMユーザーのアクセス管理はお客さまの責任に含まれます。詳しくは[第9章 セキュリティ](ch09-security.md)をご覧ください。
</details>

<details>
<summary>問3の解答</summary>

**正解: A. 初期投資を抑えて従量課金で利用できる、C. 需要に応じてリソースを素早く増減できる**

Bは誤りです。クラウドが必ずオンプレミスより安くなるとは限りません。使い方によってはクラウドのほうが高くなることもあります。Dも誤りで、責任共有モデルにより、セキュリティの一部はお客さまの責任です。Eはオンプレミスの特徴で、クラウドのメリットではありません。
</details>

<details>
<summary>問4の解答</summary>

**正解: C. S3 Glacier Deep Archive**

長期アーカイブに最もコスト効率が良いのはS3 Glacier Deep Archiveです。保存コストが最も安い代わりに、データの取り出しに最大12時間かかります。S3 Standard-IAやS3 One Zone-IAはアクセス頻度が低いデータ向けですが、アーカイブ目的ならGlacier Deep Archiveが最適です。詳しくは[第3章 ストレージ](ch03-storage.md)をご覧ください。
</details>

<details>
<summary>問5の解答</summary>

**正解: C. IAMユーザーに最小権限の原則を適用する**

最小権限の原則（必要最小限の権限だけを付与する）はIAMの最も重要なベストプラクティスです。ルートユーザーの日常使用や全ユーザーへの管理者権限付与はセキュリティ上のリスクが高く、推奨されません。MFAはすべてのユーザー、特にルートユーザーに対して常に有効化すべきです。
</details>

<details>
<summary>問6の解答</summary>

**正解: B. マルチAZ配置**

マルチAZ配置は、プライマリDBに障害が発生した場合にスタンバイDBに自動的に切り替わるしくみで、可用性を高めるために使います。リードレプリカは読み取りパフォーマンスの向上が目的です。ElastiCacheやDynamoDB Acceleratorはキャッシュサービスで、RDSの可用性向上の機能ではありません。
</details>

<details>
<summary>問7の解答</summary>

**正解: C. AWS Shield Standard**

AWS Shield Standardは、すべてのAWSアカウントに追加費用なしで自動適用されるDDoS保護サービスです。WAFはWebアプリケーションファイアウォールで別途費用がかかります。GuardDutyは脅威検知サービスです。Shield Advancedは月額3,000 USD以上の有料プランです。
</details>

<details>
<summary>問8の解答</summary>

**正解: B. AWS CloudTrail**

CloudTrailは、AWSアカウント内のAPI呼び出しを記録するサービスです。「誰が」「いつ」「何をしたか」を確認するのに使います。CloudWatchはリソースのパフォーマンス監視、AWS Configはリソースの設定変更の追跡と評価、Inspectorは脆弱性スキャンのサービスです。詳しくは[第10章 モニタリング](ch10-monitoring.md)をご覧ください。
</details>

<details>
<summary>問9の解答</summary>

**正解: C. スポットインスタンス**

中断されても再開できるバッチ処理には、スポットインスタンスが最もコスト効率が良い選択肢です（最大90%割引）。スポットインスタンスはAWSの余剰キャパシティを利用するため安価ですが、AWSの需要に応じて中断される可能性があります。中断耐性のあるワークロードに最適です。詳しくは[第1章 Webアプリケーションのホスティング](ch01-web-hosting.md)をご覧ください。
</details>

<details>
<summary>問10の解答</summary>

**正解: C. Amazon RDSはPaaSに分類される**

RDSはデータベースの実行環境を提供するPaaS（Platform as a Service）です。OSの管理やパッチ適用をAWSが行います。EC2はIaaS（Infrastructure as a Service）、LambdaはサーバーレスサービスでありPaaSに近い位置づけです。S3はストレージのマネージドサービスで、試験ではインフラサービスの一つとして扱われることが多いです。
</details>

<details>
<summary>問11の解答</summary>

**正解: B. NATゲートウェイ**

NATゲートウェイは、プライベートサブネットからインターネットへの送信接続を可能にしつつ、インターネットからの受信接続を拒否するサービスです。インターネットゲートウェイはパブリックサブネット向けで、双方向の通信を許可します。VPCエンドポイントはAWSサービスへのプライベート接続に使います。詳しくは[第5章 ネットワーク](ch05-networking.md)をご覧ください。
</details>

<details>
<summary>問12の解答</summary>

**正解: C. 機械学習を使ってAWS環境の脅威を検知するサービスである**

GuardDutyは、CloudTrailログ、VPCフローログ、DNSログなどを機械学習で分析し、不審なアクティビティを検知するサービスです。AはWAF、BはInspectorの説明です。詳しくは[第9章 セキュリティ](ch09-security.md)をご覧ください。
</details>

<details>
<summary>問13の解答</summary>

**正解: B. 無料**

AWSへのデータ転送（インバウンド）は、原則として無料です。これはAWSの基本的な料金ルールの一つで、よく出題されます。有料になるのはAWSからのデータ転送（アウトバウンド）です。詳しくは[第15章 コスト管理](ch15-cost.md)をご覧ください。
</details>

<details>
<summary>問14の解答</summary>

**正解: B. SQS FIFO**

SQS FIFOキューは「First In, First Out（先入れ先出し）」の名前のとおり、メッセージの順序を保証し、重複を排除する機能を持っています。SQS Standardは順序を保証せず、重複が発生する可能性があります。SNSはPub/Sub型の通知サービス、EventBridgeはイベントバスサービスで、いずれもキューとは異なります。詳しくは[第6章 非同期処理](ch06-async.md)をご覧ください。
</details>

<details>
<summary>問15の解答</summary>

**正解: B. AWS Cost Explorer**

Cost Explorerは過去の使用量とコストを可視化し、トレンドを分析して将来のコストを予測できるサービスです。Budgetsは予算の設定とアラート通知、Pricing Calculatorは新規利用時のコスト見積もり、Trusted Advisorはベストプラクティスに基づく最適化推奨を行うサービスです。詳しくは[第15章 コスト管理](ch15-cost.md)をご覧ください。
</details>

<details>
<summary>問16の解答</summary>

**正解: B. Amazon ECS + AWS Fargate**

Fargateはサーバーレスのコンテナ実行環境で、EC2インスタンスの管理が不要です。ECSはコンテナのオーケストレーションサービスで、Fargateと組み合わせることで「サーバー管理なし」でコンテナを実行できます。EC2起動タイプではEC2インスタンスの管理が必要になります。詳しくは[第8章 コンテナ](ch08-containers.md)をご覧ください。
</details>

<details>
<summary>問17の解答</summary>

**正解: B. 物理インフラストラクチャの管理、D. グローバルネットワークインフラの保護**

物理インフラ（データセンター、サーバー、ネットワーク機器）の管理と、グローバルネットワークインフラの保護はAWSの責任範囲です。データの暗号化設定、セキュリティグループの設定、アプリケーションのアクセス制御はお客さまの責任です。
</details>

<details>
<summary>問18の解答</summary>

**正解: B. テンプレートファイルを使ってAWSリソースを自動作成するサービス**

CloudFormationはIaC（Infrastructure as Code）サービスで、JSONまたはYAML形式のテンプレートファイルに記述した構成を自動的に作成・管理します。同じテンプレートから同じ環境を何度でも再現できるのが大きなメリットです。詳しくは[第11章 CI/CD](ch11-cicd.md)をご覧ください。
</details>

<details>
<summary>問19の解答</summary>

**正解: C. ビジネス**

Trusted Advisorの全チェック項目（5カテゴリすべて）を利用するには、ビジネスプラン以上のサポートプランが必要です。ベーシックプランとデベロッパープランでは、一部のコアチェックのみ利用可能です。詳しくは[第15章 コスト管理](ch15-cost.md)をご覧ください。
</details>

<details>
<summary>問20の解答</summary>

**正解: C. 高可用性**

Well-Architectedフレームワークの6つの柱は、運用上の優秀性、セキュリティ、信頼性、パフォーマンス効率、コスト最適化、持続可能性です。「高可用性」は重要な設計目標ですが、独立した柱の名前ではありません。高可用性に関連する内容は「信頼性」の柱に含まれています。紛らわしいので注意しましょう。
</details>

**正解: A. サーバーの購入費用が固定費から変動費に変わる**

クラウドの大きな経済的メリットの一つは、サーバーなどの設備投資（固定費）が不要になり、使った分だけ支払う従量課金（変動費）に切り替わることです。Bは誤りで、使い方によってはクラウドのほうが高くなるケースもあります。Cも誤りで、ソフトウェアライセンスはクラウドでも必要な場合があります。Dも誤りで、アウトバウンドのデータ転送には料金がかかります。

# 第15章　コスト管理と最適化 ― クラウドの経済性を最大化する

**対応試験ドメイン**: 第4分野: 請求、料金、およびサポート（12%）、第1分野: クラウドのコンセプト（24%）
**推定読了時間**: 30分

---

## この章で学ぶこと

クラウドの大きなメリットのひとつは「使った分だけ払う」という料金モデルです。しかし、このメリットを活かすには、コストの仕組みを正しく理解し、適切に管理する必要があります。この章では、AWSの料金の仕組みからコスト管理ツール、最適化の手法までを体系的に学びます。

- AWSの従量課金モデルと無料利用枠の仕組み
- Cost Explorer / Pricing Calculator / Budgets によるコストの可視化・見積もり・予算管理
- Trusted Advisor / Compute Optimizer を活用したコスト最適化
- Organizations の一括請求によるボリュームディスカウント

## 15-1. クラウドの料金モデル

### このセクションで学ぶこと

AWSの「従量課金」の基本原則、料金に影響する3つの要素、そして無料利用枠の種類を理解します。

### 使った分だけ払う ― 従量課金の原則

AWSの料金モデルは、一言でいえば「使った分だけ払う」です。これは、電気代や水道代の仕組みとよく似ています。

オンプレミス（自社でサーバーを購入・運用する方式）では、サーバーを買った時点で数百万円のコストがかかります。しかも、そのサーバーが夜間や休日にほとんど使われていなくても、電気代や冷却費用はかかり続けます。これは、お客さんが1日3人しか来ないお店に24時間店番を置いているようなものです。

クラウドでは、こうした「使っていない分のコスト」がなくなります。サーバーが不要な時間帯は停止すれば課金されませんし、利用量が増えたときだけスケールアップ（性能を上げること）すればよいのです。初期費用が不要で、利用した分だけ後払い。これがクラウドの料金モデルの基本です。

### 料金に影響する3つの要素

AWSの料金を決める要素は、大きく3つに分かれます。「コンピューティング」「ストレージ」「データ転送」です。

コンピューティングは、サーバーの処理能力に対する料金です。EC2インスタンスを動かしている時間や、Lambda関数が実行された回数などが課金対象になります。インスタンスのスペック（CPUの数やメモリの量）が高いほど、1時間あたりの料金も高くなります。

ストレージは、データを保存する容量に対する料金です。S3に保存したファイルのサイズや、EBSボリュームの容量などが課金対象です。保存するデータが多いほど料金が上がります。

データ転送は、AWSとインターネットの間でやりとりするデータ量に対する料金です。ここで非常に重要なルールがあります。AWSへのデータ転送（インバウンド＝外からAWSへの通信）は無料ですが、AWSからのデータ転送（アウトバウンド＝AWSから外への通信）は課金されます。

![AWS料金の3要素](../images/ch15-pricing-factors.drawio.png)

このルールは、郵便の「着払い」と逆のイメージです。手紙を送る（アウトバウンド）ときにだけ切手代がかかり、受け取る（インバウンド）ときは無料です。試験ではこのルールがひっかけ問題としてよく出題されます。「データをAWSにアップロードする際の転送料金」を聞かれたら、答えは「無料」です。

### 無料利用枠 ― 3つのタイプ

AWSには無料利用枠（Free Tier）が用意されており、一定の範囲内であれば料金がかかりません。無料利用枠には3つのタイプがあり、それぞれ仕組みが異なります。

1つ目は「12か月間無料」です。AWSアカウントを作成してから12か月間、対象サービスを一定量まで無料で使えます。代表的な例はEC2のt2.microインスタンス（月750時間まで）や、S3のストレージ（5GBまで）です。学習や検証目的での利用に最適ですが、12か月を過ぎると通常料金がかかるので注意が必要です。

2つ目は「常時無料（Always Free）」です。アカウント作成からの期間に関係なく、ずっと無料で使えます。代表的な例はAWS Lambdaの月100万リクエストと40万GB秒のコンピューティング時間、そしてAmazon DynamoDBの25GBのストレージです。これらは12か月を過ぎた後も無料のままです。

3つ目は「トライアル（短期無料体験）」です。特定のサービスを初めて使うときに、一定期間だけ無料で試せるものです。たとえばAmazon Inspectorの90日間トライアルなどがあります。

試験では「Lambdaの月100万リクエスト無料はどのタイプか？」のように、具体的なサービスと無料利用枠のタイプを結びつける問題が出ることがあります。Lambdaの無料枠は「常時無料」、EC2 t2.microの無料枠は「12か月間無料」と覚えておきましょう。

なお、無料枠を超えると自動的に通常料金が課金されます。「知らないうちに課金されていた」とならないよう、15-3節で紹介するAWS Budgetsで予算アラートを設定しておくことを強くおすすめします。

---

> **試験のポイント**
> - AWSの料金を決める3要素: コンピューティング、ストレージ、データ転送
> - データ転送: インバウンド（AWS への送信）= 無料、アウトバウンド（AWS からの送信）= 有料
> - 無料利用枠の3種類: 12か月間無料（EC2 t2.micro等）、常時無料（Lambda 100万リクエスト/月等）、トライアル
> - 「従量課金 = 使った分だけ」「初期費用なし」「予約不要」がクラウドの料金原則

## 15-2. コスト可視化と事前見積もり

### このセクションで学ぶこと

AWSのコストを「見える化」するためのツール群を理解します。特に、Cost Explorer、Pricing Calculator、Cost and Usage Reports の3つの使い分けが重要です。

### なぜコストの可視化が必要か

従量課金は「使った分だけ払う」という合理的なモデルですが、裏を返せば「使った分を把握していないと、思わぬ高額請求が来る」ということでもあります。

たとえば、開発チームがテスト用に大きなインスタンスを起動したまま忘れていた、不要になったストレージを削除し忘れていた、というのはよくある話です。クラウドのコスト管理では、まず「いま何にいくらかかっているのか」を見える化することが出発点になります。

### AWS Cost Explorer ― 過去を分析し、未来を予測する

AWS Cost Explorer（コストエクスプローラー）は、AWSの利用料金を可視化・分析するためのサービスです。家計簿アプリのようなものだと考えてください。

Cost Explorerでは、過去の利用実績をグラフで確認できます。マネジメントコンソールからCost Explorerを開くと、月ごとの利用料金が棒グラフで表示され、サービス別やリージョン別にフィルタリングできます。「先月と今月でどのサービスのコストが増えたか」「どのリージョンにコストが集中しているか」といった傾向分析が簡単にできます。

さらに、過去の実績データをもとに将来のコストを予測する機能もあります。「このままのペースで使い続けると、来月の請求はいくらになりそうか」を見積もることができます。

Cost Explorerは「過去の分析」と「将来の予測」を行うツールです。これが次に紹介するBudgetsやPricing Calculatorとの使い分けのポイントになります。

### AWS Pricing Calculator ― 導入前のコスト見積もり

AWS Pricing Calculator（プライシングカリキュレーター）は、AWSのサービスを実際に使い始める前にコストを試算できるツールです。

たとえば「EC2のm5.largeインスタンスを2台、東京リージョンで24時間365日動かしたら、月額いくらになるか」を、実際にインスタンスを起動しなくても計算できます。Webブラウザからアクセスし、使いたいサービスと構成を選んでいくだけで、月額の見積もり金額が表示されます。引っ越し前に新居の家賃や光熱費をシミュレーションするようなものです。

Cost Explorerが「過去の実績を分析する家計簿」なら、Pricing Calculatorは「これから始める前の見積書」です。この違いは試験でよく問われます。

### AWS Cost and Usage Reports ― 詳細な利用レポート

AWS Cost and Usage Reports（CUR）は、AWSの利用状況と料金をもっとも詳細に記録したレポートです。Cost Explorerよりもさらに細かい粒度で、個々のリソースごとの利用量と料金を確認できます。

このレポートはS3バケットにCSV形式で出力されるため、Amazon Athena（アテナ）でクエリを実行して分析したり、社内の経理システムと連携したり、独自の分析ツールで処理したりすることができます。企業の経理部門や財務部門が使うことが多いサービスです。

### コスト配分タグ ― プロジェクト別・部門別の管理

大きな組織では、「このコストは開発部門のもの」「このコストはマーケティング部門のプロジェクトA用」というように、コストを部門やプロジェクトごとに仕分けたいことがあります。

そこで使うのが、コスト配分タグ（Cost Allocation Tags）です。AWSリソースに「部門名」や「プロジェクト名」などのタグ（ラベル）を付けておくことで、Cost ExplorerやCost and Usage Reportsでタグごとにコストを集計できます。

たとえば、すべてのEC2インスタンスに `Department: Marketing` や `Project: CampaignA` というタグを付けておけば、「マーケティング部門のコストは今月いくらか」「プロジェクトAのインフラ費用はいくらか」が一目でわかるようになります。

---

> **試験のポイント**
> - Cost Explorer = 「過去のコスト傾向分析」＋「将来のコスト予測」
> - Pricing Calculator = 「導入前の事前見積もり」（実際にリソースを作らずに試算）
> - Cost and Usage Reports = 最も詳細な利用レポート（S3に出力、経理向け）
> - コスト配分タグ = リソースにラベルを付けて、部門別・プロジェクト別にコスト管理
> - 「過去を見たい → Cost Explorer」「これからの見積もり → Pricing Calculator」の使い分けは頻出

---

## 15-3. 予算管理

### このセクションで学ぶこと

AWS Budgets を使った予算設定とアラート通知の仕組みを理解し、Cost Explorer との違いを押さえます。

### AWS Budgets ― 予算を決めて超過を防ぐ

ここまでで、「過去のコストを分析する（Cost Explorer）」「将来のコストを見積もる（Pricing Calculator）」という手段を学びました。しかし、これらはあくまで「見る」ためのツールです。「使いすぎを防ぐ」ためには、別の仕組みが必要です。

AWS Budgets（バジェッツ）は、AWSの利用料金に対して予算（上限の目安）を設定し、その予算に近づいたり超過したりしたときにアラート（通知）を出すサービスです。

携帯電話の料金プランを思い出してください。「今月のデータ通信量が上限の80%に達しました」という通知が来ることがありますよね。AWS Budgetsも同じように、「今月のAWS利用料金が設定した予算の80%に達しました」といった通知をメールやAmazon SNS（Simple Notification Service、AWSの通知サービス）で受け取ることができます。

Budgetsでは、月額のコスト予算だけでなく、特定のサービスの利用量に対する予算も設定できます。たとえば「EC2の利用時間が月1,000時間を超えたら通知」といった設定も可能です。

### Budgets のアクション ― 予算超過時の自動対応

AWS Budgets には、アラートを送るだけでなく、予算超過時に自動でアクションを実行する機能もあります。たとえば「予算の100%に達したら、特定のIAMポリシーを適用して新しいリソースの作成を制限する」といった対応を自動化できます。

ただし注意したいのは、Budgets自体は「AWSの利用を強制的に止める」ものではないということです。アラートが来ても利用は継続されます。自動アクションを設定しない限り、通知を受け取って自分で対処する必要があります。

### Cost Explorer と Budgets の違い

この2つのサービスはよく混同されます。違いを明確にしておきましょう。

Cost Explorerは「分析」のためのツールです。「いままでいくら使ったか」「どのサービスにコストがかかっているか」「来月はいくらになりそうか」を分析・予測します。一方、Budgetsは「管理・通知」のためのツールです。「予算を設定して、超えそうになったら教えてくれる」という役割を持ちます。

たとえるなら、Cost Explorerは「体重計」で、Budgetsは「ダイエットの目標体重と、近づいたら鳴るアラーム」です。体重計で現状を確認し（Cost Explorer）、目標を設定してアラームをかける（Budgets）という関係です。

---

> **試験のポイント**
> - Budgets = 「予算を設定 → 超過時にアラート通知」
> - Cost Explorer = 「過去の分析・将来の予測」（見るツール）、Budgets = 「予算管理・通知」（管理ツール）
> - Budgetsは利用を自動停止するものではない（通知 + オプションの自動アクション）
> - 試験で「コスト超過時にアラートを受けたい」→ 答えは Budgets

---

## 15-4. コスト最適化の手法

### このセクションで学ぶこと

コストを下げるための具体的な手法と、それを支援するAWSのサービス（Compute Optimizer、Trusted Advisor）を理解します。

### コスト最適化の4つのアプローチ

コストの可視化と予算管理ができたら、次は「どうやってコストを下げるか」です。コスト最適化には、大きく4つのアプローチがあります。

![コスト管理ツールの使い分け](../images/ch15-cost-tools.drawio.png)

### 1. 適切なインスタンスサイズの選定

コスト最適化の第一歩は、「大きすぎるインスタンスを使っていないか」を確認することです。

たとえば、実際にはCPU使用率が平均10%しかないのに、16コアの大きなインスタンスを使っているとしたら、4コアのインスタンスに変更するだけでコストが大幅に下がります。これは、一人暮らしなのに4LDKのマンションに住んでいるようなものです。1LDKに引っ越せば家賃はぐっと安くなります。

この「適切なサイズ選び」を支援してくれるのが AWS Compute Optimizer（コンピュートオプティマイザー）です。Compute Optimizerは、EC2インスタンスやEBSボリューム、Lambda関数の過去の利用データを分析し、「このインスタンスはm5.largeからm5.mediumにダウンサイズしても問題ないですよ」といった推奨を提示してくれます。

### 2. 購入オプションの最適化

第1章で学んだEC2の購入オプションを、コスト最適化の観点から整理しましょう。

![購入オプションのコスト比較](../images/ch15-purchase-options.drawio.png)

オンデマンドインスタンスは、定価でいつでも使える基本の選択肢です。短期間の利用や、利用パターンが読めない場合に適しています。

リザーブドインスタンス（RI）は、1年または3年の利用をコミット（約束）することで、オンデマンド料金と比較して大幅な割引（最大72%程度、支払い方法や契約期間により異なる）が得られます。たとえば、オンデマンドで月額1万円のインスタンスが、3年全前払いのリザーブドなら月額約3,000円相当になるイメージです。24時間365日稼働し続けるデータベースサーバーのように、長期間安定して使うワークロードに最適です。

なお、リザーブドインスタンスにはスタンダードRIとコンバーティブルRIの2種類があります。コンバーティブルRIはインスタンスファミリー（サーバーの種類）を途中で変更できますが、その分割引率はスタンダードRIより低くなります。

Savings Plansは、リザーブドインスタンスと同様に1年または3年のコミットですが、「1時間あたりの利用金額」をコミットする点が異なります。インスタンスタイプやリージョンを柔軟に変更できるため、リザーブドインスタンスより自由度が高いのが特徴です。

スポットインスタンスは、AWSの余剰キャパシティを利用するもので、最大90%の割引が得られます。ただし、AWSが容量を必要とすると2分前の通知で中断される可能性があります。バッチ処理やデータ分析など、中断されても問題ないワークロードに適しています。

購入オプションの詳しい仕組みは第1章で解説していますので、そちらもあわせて確認してください。

### 3. 未使用リソースの特定

クラウド環境が大きくなると、「テスト用に作ったけれど使っていないインスタンス」「誰も接続していないロードバランサー」「アタッチされていないEBSボリューム」といった未使用リソースが増えがちです。これらは何もしていないのに料金だけ発生し続けます。

こうした無駄を見つけてくれるのが AWS Trusted Advisor（トラステッドアドバイザー）です。Trusted Advisorは、第10章で学んだように5つのカテゴリ（コスト最適化、パフォーマンス、セキュリティ、耐障害性、サービスクォータ）でベストプラクティスに基づいたチェックを行うサービスです。

コスト最適化のカテゴリでは、「使用率の低いEC2インスタンス」「アイドル状態のRDSインスタンス」「関連付けられていないElastic IPアドレス」などを検出して通知してくれます。AWSの専属コンサルタントが「ここにムダがありますよ」と教えてくれるイメージです。

なお、Trusted Advisorのすべてのチェック項目を利用するには、ビジネスサポートプラン以上の契約が必要です。サポートプランの詳細については付録B（サポートプラン）で解説予定です。

### 4. ストレージクラスの最適化

S3のストレージクラスを使い分けることも、大きなコスト削減につながります。

頻繁にアクセスするデータはS3 Standard、たまにしかアクセスしないデータはS3 Standard-IA（Infrequent Access）、ほぼアクセスしないアーカイブデータはS3 Glacierに保存することで、ストレージコストを大幅に下げられます。ストレージクラスの詳細は第3章で解説しています。

アクセス頻度が予測しにくい場合は、S3 Intelligent-Tieringが便利です。これはAWSが自動的にアクセスパターンを分析し、最もコスト効率のよいストレージクラスにデータを移動してくれる仕組みです。

このほかにも、サーバーレスアーキテクチャへの移行（第2章で紹介したLambdaの活用など）や、CloudFrontによるデータ転送コストの削減（第1章）もコスト最適化の有効な手法です。

---

> **試験のポイント**
> - Compute Optimizer = 「インスタンスサイズの推奨」（過去の利用データを分析して適切なサイズを提案）
> - Trusted Advisor = 「ベストプラクティスに基づく最適化提案」（未使用リソースの検出など5カテゴリ）
> - 購入オプション: オンデマンド（定価）→ リザーブド/Savings Plans（長期コミットで割引）→ スポット（最大90%割引、中断リスクあり）
> - 「安定稼働のワークロードのコストを下げたい」→ リザーブドまたはSavings Plans
> - 「中断されてもよい処理のコストを最小限にしたい」→ スポットインスタンス

---

## 15-5. 組織のコスト管理

### このセクションで学ぶこと

AWS Organizationsの一括請求（Consolidated Billing）によるメリットと、AWS Marketplaceの概要を理解します。

### AWS Organizations の一括請求

企業が大きくなると、部門ごとやプロジェクトごとに複数のAWSアカウントを使い分けるのが一般的です。開発環境、ステージング環境、本番環境をアカウントで分けたり、セキュリティ上の理由で部門ごとにアカウントを分離したりします。

しかし、アカウントが増えると請求管理が煩雑になります。10個のアカウントがあれば10通の請求書が届き、それぞれ個別に支払い処理をしなければなりません。

AWS Organizations（オーガニゼーションズ）の一括請求（Consolidated Billing）を使えば、複数アカウントの利用料金をひとつの請求にまとめることができます。これは、家族全員の携帯電話料金をまとめて支払う「ファミリープラン」のようなものです。

### ボリュームディスカウント

一括請求の最大のメリットは、ボリュームディスカウント（利用量に応じた値引き）を受けられることです。

AWSの多くのサービスには「使えば使うほど単価が下がる」という料金体系があります。たとえば、S3のストレージ料金は、最初の50TBまでは1GBあたり0.025ドルですが、50TBを超えると単価が下がります。

一括請求を使うと、組織内のすべてのアカウントの利用量が合算されます。アカウントAが30TB、アカウントBが25TBを使っていた場合、合計55TBとして計算されるため、ボリュームディスカウントの恩恵を受けやすくなります。個別に請求されていたら、どちらのアカウントも50TB未満なので割引が適用されないところです。

また、あるアカウントで購入したリザーブドインスタンスやSavings Plansの割引を、同じ組織内の他のアカウントと共有することもできます。組織全体でコストを最適化できるのが一括請求の強みです。

### AWS Marketplace

AWS Marketplace（マーケットプレイス）は、サードパーティ（AWS以外の企業）が提供するソフトウェアやサービスを、AWSの請求にまとめて購入・利用できるオンラインストアです。

スマートフォンのアプリストアを想像してください。App StoreやGoogle Playで必要なアプリを見つけて、ワンクリックでインストールするように、AWS Marketplaceではセキュリティソフト、データベース、機械学習ツールなど、さまざまなソフトウェアを見つけてすぐにAWS環境にデプロイ（導入）できます。

Marketplaceのメリットは、ソフトウェアの購入・ライセンス管理が簡単になることと、料金がAWSの請求書にまとめられるため経理処理がシンプルになることです。

なお、セキュリティ関連のサービスにはコストに影響するものもあります。たとえばAWS Shield Advanced（DDoS攻撃からの高度な保護サービス）は有料のオプションです。セキュリティサービスの詳細とそのコスト影響については第9章を参照してください。

また、オンプレミスからAWSへの移行コストを見積もりたい場合は、Migration Evaluator（第14章）が便利です。現在のオンプレミス環境のデータを収集・分析し、AWS移行後のコストを試算してくれる専用ツールです。

---

> **試験のポイント**
> - 一括請求（Consolidated Billing）= 複数アカウントの請求をまとめる → ボリュームディスカウントのメリット
> - 組織内で リザーブドインスタンス / Savings Plans の割引を共有できる
> - AWS Marketplace = サードパーティ製ソフトウェアのオンラインストア（AWS請求にまとめて管理）

### 問題1

AWSの料金モデルにおいて、データ転送の課金ルールとして正しいものはどれですか？

- **A.** インバウンド（AWSへの転送）もアウトバウンド（AWSからの転送）も有料
- **B.** インバウンドは有料、アウトバウンドは無料
- **C.** インバウンドは無料、アウトバウンドは有料
- **D.** インバウンドもアウトバウンドも無料

<details>
<summary>解答・解説</summary>

**解答: C**

解説: AWSでは、データをAWSに送信する（インバウンド）のは無料ですが、AWSからデータを送信する（アウトバウンド）のは課金されます。これは試験の頻出ポイントです。

</details>

### 問題2

新しいシステムをAWSに構築する前に、月額コストがどのくらいになるか見積もりたい場合、最も適したツールはどれですか？

- **A.** AWS Cost Explorer
- **B.** AWS Budgets
- **C.** AWS Pricing Calculator
- **D.** AWS Cost and Usage Reports

<details>
<summary>解答・解説</summary>

**解答: C**

解説: **AWS Pricing Calculator**は、実際にリソースを作成せずに、サービスの構成と料金を事前に見積もることができるツールです。Cost Explorerは過去の利用実績の分析、Budgetsは予算管理、Cost and Usage Reportsは詳細な利用レポートです。

</details>

### 問題3

AWS Lambdaの月100万リクエスト無料は、無料利用枠のどのタイプに該当しますか？

- **A.** 12か月間無料
- **B.** 常時無料（Always Free）
- **C.** トライアル
- **D.** 無料利用枠の対象外

<details>
<summary>解答・解説</summary>

**解答: B**

解説: Lambdaの月100万リクエスト無料は「常時無料（Always Free）」タイプです。アカウント作成から12か月を過ぎても、この無料枠はずっと利用できます。EC2 t2.micro（月750時間）の無料枠は「12か月間無料」タイプです。

</details>

### 問題4

AWS Organizations の一括請求（Consolidated Billing）を利用するメリットとして正しいものはどれですか？（2つ選択してください）

- **A.** 各アカウントのセキュリティポリシーが自動的に統一される
- **B.** 複数アカウントの利用量が合算され、ボリュームディスカウントを受けやすくなる
- **C.** リザーブドインスタンスの割引を組織内の他のアカウントと共有できる
- **D.** すべてのアカウントの料金が自動的に50%割引になる
- **E.** AWS利用料金の支払い方法が自動的にクレジットカードに統一される

<details>
<summary>解答・解説</summary>

**解答: B、C**

解説: 一括請求の主なメリットは、複数アカウントの利用量を合算してボリュームディスカウントを得られることと、リザーブドインスタンスやSavings Plansの割引を組織内で共有できることです。セキュリティポリシーの統一はSCP（サービスコントロールポリシー）の役割であり、一括請求の機能ではありません。

</details>

### 問題5

過去3か月間のAWS利用料金の傾向を分析し、来月のコストを予測したい場合、最も適したサービスはどれですか？

- **A.** AWS Budgets
- **B.** AWS Cost Explorer
- **C.** AWS Pricing Calculator
- **D.** AWS Trusted Advisor

<details>
<summary>解答・解説</summary>

**解答: B**

解説: **AWS Cost Explorer**は、過去のコストデータを分析し、将来のコストを予測する機能を持つサービスです。Budgetsは予算設定と通知、Pricing Calculatorは導入前の見積もり、Trusted Advisorはベストプラクティスに基づくチェックを行うサービスです。

</details>

### 問題6

常時稼働するWebサーバーを3年間運用する予定です。コストを最も抑えられるEC2の購入オプションはどれですか？

- **A.** オンデマンドインスタンス
- **B.** スポットインスタンス
- **C.** リザーブドインスタンス（3年、全前払い）
- **D.** Dedicated Host

<details>
<summary>解答・解説</summary>

**解答: C**

解説: 常時稼働で利用期間が確定しているワークロードには、**リザーブドインスタンス**が最適です。3年契約・全前払いの場合、最大の割引が得られます。スポットインスタンスは中断リスクがあるため常時稼働のWebサーバーには不適切です。Savings Plansも選択肢として有効ですが、この問題では最大割引のリザーブドが正解です。

</details>

### 問題7

EC2インスタンスのCPU使用率が常に低い状態が続いています。適切なインスタンスサイズの推奨を受けるために使用すべきサービスはどれですか？

- **A.** AWS Trusted Advisor
- **B.** AWS Compute Optimizer
- **C.** AWS Cost Explorer
- **D.** Amazon CloudWatch

<details>
<summary>解答・解説</summary>

**解答: B**

解説: **AWS Compute Optimizer**は、EC2インスタンスの過去の利用データ（CPU、メモリなど）を分析し、最適なインスタンスサイズを推奨するサービスです。Trusted Advisorもコスト最適化の提案をしますが、具体的なインスタンスサイズの推奨はCompute Optimizerの役割です。CloudWatchは監視データの収集、Cost Explorerはコストの分析を行うサービスです。

</details>

### 問題8

AWSの利用料金が設定した予算の80%に達したときに通知を受け取りたい場合、使用するサービスはどれですか？

- **A.** AWS Cost Explorer
- **B.** AWS Budgets
- **C.** AWS Pricing Calculator
- **D.** AWS Cost and Usage Reports

<details>
<summary>解答・解説</summary>

**解答: B**

解説: **AWS Budgets**は、予算を設定してしきい値に達したときにアラート通知を送るサービスです。「予算に対するアラート」「コスト超過の通知」といったキーワードが出たら、Budgetsを選びましょう。Cost Explorerは分析ツール、Pricing Calculatorは事前見積もりツール、Cost and Usage Reportsは詳細レポートです。

</details>

## 3-1. ストレージの3つの種類

### このセクションで学ぶこと

クラウドで使われる3種類のストレージの違いと、それぞれに対応するAWSサービスを理解します。

### なぜストレージの種類を知る必要があるのか

皆さんは普段、パソコンのハードディスクに写真や文書を保存していると思います。しかし、クラウドの世界では「何を、どう保存するか」によって、最適なストレージの種類が異なります。適切なストレージを選ぶことで、コストを抑えつつ必要な性能を得ることができます。

### オブジェクトストレージ

オブジェクトストレージは、データを「オブジェクト」という単位でまとめて保存する方式です。

倉庫に積まれたダンボール箱をイメージしてください。それぞれの箱にはラベル（メタデータ）が貼ってあり、中身が何かすぐにわかります。箱の中身を一部だけ取り出すことはできませんが、箱ごと取り出したり、新しい箱を追加したりするのは簡単です。

写真、動画、バックアップファイル、ログデータなど、一度保存したらそのまま丸ごと読み出すような用途に向いています。AWSでは **Amazon S3** がオブジェクトストレージに該当します。

### ブロックストレージ

ブロックストレージは、データを一定サイズの「ブロック」に分割して保存する方式です。

本棚に並べられた本をイメージしてください。必要な本だけを1冊取り出して読み書きできます。OS（オペレーティングシステム）やデータベースのように、データの一部分を頻繁に読み書きする用途に適しています。

パソコンのハードディスクやSSDと同じ感覚で使えるのが特徴です。AWSでは **Amazon EBS**（Elastic Block Store）がブロックストレージに該当します。

### ファイルストレージ

ファイルストレージは、フォルダとファイルの階層構造（ディレクトリ構造）でデータを管理する方式です。

会社の共有フォルダをイメージしてください。「営業部」フォルダの中に「2024年」フォルダがあり、その中に「売上レポート.xlsx」がある——という、普段パソコンで使い慣れたフォルダ構造そのものです。

複数のサーバーから同じファイルにアクセスしたい場合に便利です。AWSでは **Amazon EFS**（Elastic File System）や **Amazon FSx** がファイルストレージに該当します。

### 3種類の比較

| 種類 | イメージ | アクセス方法 | 主な用途 | AWSサービス |
|------|---------|-------------|---------|------------|
| オブジェクト | 倉庫のダンボール箱 | インターネット経由でURLを使ってアクセス | 画像・動画・バックアップ | Amazon S3 |
| ブロック | 本棚の1冊ずつ | サーバーにディスクとして接続 | OS・データベース | Amazon EBS |
| ファイル | 共有フォルダ | ネットワーク経由でフォルダ共有 | 複数サーバーでのファイル共有 | Amazon EFS / FSx |

> **試験のポイント**
> - 「S3 = オブジェクト」「EBS = ブロック」「EFS = ファイル」の対応は頻出
> - オブジェクトストレージはデータの一部だけを書き換えることができない（丸ごと上書き）
> - ブロックストレージはサーバーに「ディスク」としてアタッチして使う
> - ファイルストレージは複数のサーバーから同時にアクセスできる

## 3-2. Amazon S3 の基本

### このセクションで学ぶこと

AWSで最もよく使われるストレージサービスである Amazon S3 の基本的な仕組みを理解します。

### S3とは

Amazon S3（Simple Storage Service）は、AWSが提供するオブジェクトストレージサービスです。インターネット上の「貸し倉庫」のようなもので、どんな種類のデータでも、ほぼ無制限に保存できます。

Webサイトの画像、アプリケーションのログ、データベースのバックアップ、動画ファイルなど、あらゆるデータの保存先として使われています。AWSの中でも最も基本的なサービスの1つです。

### バケットとオブジェクト

S3では、データを管理するための2つの概念を理解する必要があります。

まず「バケット」です。バケットとは、オブジェクトを入れるための「容器」です。ファイルを入れるフォルダのようなものですが、バケット名はAWS全体で一意（世界中で重複しない名前）でなければなりません。たとえば「my-company-images」というバケットを誰かが作ったら、他の誰もその名前のバケットを作れません。

次に「オブジェクト」です。オブジェクトとは、S3に保存されるデータの1単位です。ファイルそのもの（データ本体）と、そのファイルに関する情報（メタデータ）で構成されます。各オブジェクトは「キー」と呼ばれるパス名で識別されます。たとえば `photos/2024/vacation.jpg` というキーは、フォルダ階層のように見えますが、実際にはただの文字列です。S3にはフォルダという概念はなく、キーの「/」がフォルダのように見えているだけです。

### S3の特長

S3には、試験でもよく問われる重要な特長がいくつかあります。

**無制限のスケーラビリティ**: バケットに保存できるオブジェクトの数に上限はありません。データの総量にも上限がありません。ただし、1つのオブジェクトの最大サイズは5TBです。

**イレブンナインの耐久性**: S3は 99.999999999%（9が11個）の耐久性を持つよう設計されています。これは「1000万個のオブジェクトを保存した場合、1万年に1個のオブジェクトが失われる可能性がある」というレベルの堅牢さです。S3は自動的に、1つのリージョン内の3つ以上のAZにデータをコピーして保存しています（S3 One Zone-IA を除く。詳しくはセクション3-3で説明します）。

**強い整合性**: S3では、データを上書きしたり削除したりした直後でも、常に最新の状態が読み取れます。「さっき更新したのに古いデータが表示される」ということが起きない仕組みになっています。

**静的Webサイトのホスティング**: S3のバケットをWebサーバーのように使って、HTMLやCSSなどの静的なWebサイトを公開することもできます。

---

> **試験のポイント**
> - S3の耐久性「99.999999999%（イレブンナイン）」は頻出の数値
> - 「オブジェクト数は無制限」「1オブジェクト最大5TB」を押さえる
> - S3はデータを最低3つのAZに自動複製する（One Zone-IA を除く）
> - バケット名はグローバルに一意でなければならない
> - S3はサーバーレスサービス（サーバーの管理が不要）

---

### S3のセキュリティ

S3のセキュリティは試験でも頻出のテーマです。まず覚えていただきたいのは、S3バケットはデフォルト（初期設定）でプライベートであるということです。つまり、バケットを作った直後は、作成したAWSアカウントの持ち主だけがアクセスできます。外部の人には公開されていません。

誰がバケットやオブジェクトにアクセスできるかを制御するために、「バケットポリシー」という仕組みがあります。バケットポリシーは、JSON（設定を書くための形式）で「このユーザーにはこのバケットの読み取りを許可する」といったルールを定義します。詳しい書き方は第9章（セキュリティ）で扱いますが、ここでは「バケット単位でアクセス権限を設定できる仕組みがある」ということを押さえてください。

もう1つ重要なのが **S3 Block Public Access**（パブリックアクセスブロック）です。これは、バケットやオブジェクトが誤って外部に公開されてしまうのを防ぐための安全装置です。アカウント全体またはバケット単位で有効にでき、デフォルトで有効になっています。「うっかり全世界に公開してしまった」という事故を防ぐための重要な機能です。

暗号化についても触れておきましょう。S3ではデフォルトで **サーバーサイド暗号化（SSE-S3）** が有効になっています。これは、データをS3に保存する際に自動的に暗号化してくれる仕組みです。利用者が特別な設定をしなくても、保存されたデータは暗号化された状態で守られます。より厳密に暗号化の鍵を管理したい場合は、AWS KMS（Key Management Service）を使った **SSE-KMS** も選択できます。暗号化の詳細は第9章で解説します。

---

> **試験のポイント**
> - S3バケットはデフォルトでプライベート（外部には非公開）
> - バケットポリシーでアクセス制御を設定できる
> - S3 Block Public Access はデフォルトで有効。誤った公開を防ぐ安全装置
> - S3はデフォルトでサーバーサイド暗号化（SSE-S3）が有効
> - より厳密な鍵管理が必要な場合は SSE-KMS を使用

## 3-3. S3 ストレージクラス

### このセクションで学ぶこと

S3にはデータのアクセス頻度に応じた複数のストレージクラスがあり、それぞれコストと取り出し速度が異なります。適切なクラスを選ぶことで、大幅なコスト削減が可能です。

### なぜストレージクラスが必要なのか

すべてのデータが同じ頻度でアクセスされるわけではありません。たとえば、先週撮った写真はよく見返しますが、5年前の写真を開く機会はめったにないでしょう。

S3では、頻繁にアクセスするデータは「すぐ取り出せるが保存コストが高いクラス」に、めったにアクセスしないデータは「取り出しに時間がかかるが保存コストが安いクラス」に保存できます。この仕組みにより、無駄なコストを抑えられます。

### 各ストレージクラスの特徴

**S3 Standard（標準）**

最も基本的なストレージクラスです。頻繁にアクセスするデータ向けで、データの取り出しにかかる時間はミリ秒単位です。保存コストはストレージクラスの中で最も高いですが、データの取り出しにかかる料金はかかりません。Webサイトの画像やアプリケーションのデータなど、日常的に読み書きするデータに適しています。

**S3 Standard-IA（低頻度アクセス）**

IAは「Infrequent Access（低頻度アクセス）」の略です。月に1〜2回程度しかアクセスしないデータ向けです。保存コストはStandardより安いですが、データを取り出すたびに「取り出し料金」がかかります。災害復旧用のバックアップや、たまにしか参照しない過去のレポートなどに適しています。Standard と同じく3つ以上のAZにデータが複製されます。

**S3 One Zone-IA（1ゾーン低頻度アクセス）**

Standard-IA と似ていますが、データが1つのAZにしか保存されません。そのため、Standard-IA よりさらに約20%安くなります。ただし、そのAZに障害が発生するとデータが失われる可能性があります。簡単に再作成できるデータ（例: サムネイル画像のコピー）に向いています。

**S3 Intelligent-Tiering（インテリジェントティアリング）**

アクセスパターンが予測できないデータ向けの便利なクラスです。S3が自動的にアクセス頻度を監視し、よく使われるデータは高速なティア（層）に、使われなくなったデータは低コストなティアに自動移動します。少額の月額モニタリング料金がかかりますが、手動でクラスを切り替える手間がなくなります。

**S3 Glacier Instant Retrieval（グレイシャーインスタントリトリーバル）**

「Glacier（グレイシャー）」は「氷河」という意味です。めったにアクセスしないが、必要なときはすぐに取り出したいデータ向けです。取り出し時間はミリ秒単位（Standardと同等の速さ）ですが、保存コストはStandard-IAよりさらに安くなります。四半期に1回程度アクセスするアーカイブデータに適しています。

**S3 Glacier Flexible Retrieval（グレイシャーフレキシブルリトリーバル）**

年に1〜2回しかアクセスしないような長期アーカイブデータ向けです。取り出しには通常3〜5時間かかります（迅速取り出しオプションを使えば数分も可能）。保存コストは非常に安く、法令で数年間の保存が義務付けられているデータなどに適しています。

**S3 Glacier Deep Archive（グレイシャーディープアーカイブ）**

S3の中で最も保存コストが安いクラスです。取り出しには12〜48時間かかります。7〜10年以上の長期保存が必要な、ほぼアクセスしないデータ向けです。規制対応のために保管しているだけのデータに最適です。

### ストレージクラスの選び方の目安

| ストレージクラス | アクセス頻度 | 取り出し時間 | 保存コスト | 取り出し料金 |
|----------------|------------|------------|----------|------------|
| Standard | 頻繁 | ミリ秒 | 高い | なし |
| Standard-IA | 月1〜2回 | ミリ秒 | やや安い | あり |
| One Zone-IA | 月1〜2回 | ミリ秒 | さらに安い | あり |
| Intelligent-Tiering | 不明 | ミリ秒 | 自動最適化 | なし（監視料あり） |
| Glacier Instant Retrieval | 四半期に1回 | ミリ秒 | 安い | あり |
| Glacier Flexible Retrieval | 年1〜2回 | 3〜5時間 | とても安い | あり |
| Glacier Deep Archive | ほぼなし | 12〜48時間 | 最も安い | あり |

> （draw.io ファイルは `docs/images/ch03-s3-storage-classes.drawio.png` に配置予定）

---

> **試験のポイント**
> - ストレージクラスの使い分けは最頻出テーマ。アクセス頻度とコストの関係を正確に
> - Glacier系の取り出し時間の違いを押さえる（Instant=ミリ秒、Flexible=数時間、Deep Archive=半日〜2日）
> - One Zone-IA は1つのAZにしか保存しないため、AZ障害でデータが失われるリスクがある（ひっかけ注意）
> - Intelligent-Tiering はアクセスパターンが読めないときに選ぶ
> - 「長期アーカイブで最もコストが低い」→ Glacier Deep Archive

## 3-4. S3 のライフサイクルポリシーとバージョニング

### このセクションで学ぶこと

S3のデータを自動的にコスト効率の良いストレージクラスへ移行する仕組みと、誤削除からデータを守る仕組みを理解します。

### ライフサイクルポリシー

データは時間が経つにつれて、アクセスされる頻度が下がるのが一般的です。たとえば、作成したばかりのレポートは頻繁に参照しますが、3ヶ月もすればめったに開かなくなります。1年後にはほぼ見ることはないでしょう。

S3のライフサイクルポリシーを使うと、こうしたデータの「老化」に合わせて、ストレージクラスを自動的に移行できます。たとえばECサイトの注文明細データを例に考えてみましょう。注文直後は問い合わせ対応のために頻繁にアクセスしますが、数ヶ月も経てばほとんど見ることはなくなります。しかし法律上、一定期間は保管義務があります。こうしたケースでは、次のようなルールを設定できます。

- 作成から30日後 → Standard-IA に移行（問い合わせが減ってきた頃）
- 作成から90日後 → Glacier Flexible Retrieval に移行（ほぼ参照しなくなった頃）
- 作成から365日後 → Glacier Deep Archive に移行（保管義務のために残しておく段階）
- 作成から2555日（7年）後 → 自動削除（保管義務期間が終了）

このルールを一度設定すれば、あとはS3が自動的にデータを適切なクラスに移動してくれます。手動でデータを移動する手間がなくなり、コストも自動的に最適化されます。

### バージョニング

バージョニングとは、S3バケット内のオブジェクトの変更履歴をすべて保持する機能です。

Wordの「変更履歴」機能を思い浮かべてください。ファイルを上書き保存しても、過去のバージョンがすべて残っています。誤って削除してしまった場合も、以前のバージョンから復元できます。

バージョニングを有効にすると、オブジェクトを上書きしても古いバージョンが残ります。オブジェクトを削除しても、実際にはデータは消えず「削除マーカー」が付くだけです。削除マーカーを取り除けば、データを復元できます。

ただし注意点もあります。すべてのバージョンが保存されるため、ストレージの使用量（つまりコスト）は増えます。バージョニングとライフサイクルポリシーを組み合わせて、古いバージョンを一定期間後に自動削除するのが現実的な運用です。

---

> **試験のポイント**
> - ライフサイクルポリシーで「Standard → IA → Glacier」のような段階的な自動移行が可能
> - バージョニングにより誤削除からデータを保護できる
> - バージョニングとライフサイクルポリシーは組み合わせて使うのがベストプラクティス
> - ライフサイクルポリシーは「コスト最適化」の文脈で出題されることが多い

---

## 3-5. Amazon EBS

### このセクションで学ぶこと

EC2インスタンスに接続するブロックストレージである Amazon EBS の特徴を理解します。

### EBSとは

Amazon EBS（Elastic Block Store）は、EC2インスタンス（AWSの仮想サーバー）に接続して使うブロックストレージサービスです。

パソコンに外付けハードディスクを接続するのと同じイメージです。EC2にEBSボリューム（ディスク）をアタッチ（接続）すると、EC2からは普通のハードディスクのように見えます。OSのインストールやデータベースの保存先として使うのが一般的です。

第1章でEC2を学んだ際にも少し触れましたが、ここではストレージとしてのEBSの特徴をもう少し詳しく見ていきましょう。

### EBSの特徴

**同一AZ内で利用**: EBSボリュームは特定のAZに作成されます。そのAZ内のEC2インスタンスにのみアタッチできます。別のAZのEC2には直接アタッチできません。

**基本的に1対1の関係**: 通常、1つのEBSボリュームは1つのEC2インスタンスにしかアタッチできません（一部の高性能タイプでは例外もありますが、試験では「基本は1対1」と覚えれば十分です）。

**EC2を停止してもデータは残る**: EC2インスタンスを停止・再起動しても、EBSボリュームのデータはそのまま保持されます。ただし、EC2を「終了（Terminate）」した場合は、設定によってはEBSも一緒に削除されることがあります。

### EC2 インスタンスストアとの違い

EBSと混同しやすいものに「インスタンスストア」があります。インスタンスストアは、EC2インスタンスが動いている物理サーバーに直接つながっているストレージです。EBSがネットワーク経由で接続する「外付けハードディスク」だとすれば、インスタンスストアはパソコンに最初から内蔵されている「内蔵SSD」のようなものです。

インスタンスストアはEC2に物理的に接続されているため非常に高速ですが、EC2インスタンスを停止・終了するとデータがすべて消えてしまいます。そのため、一時的なキャッシュや作業用データの保存には向いていますが、大切なデータの保存には使えません。大切なデータはEBSに保存するのが基本です。

### ボリュームタイプ

EBSにはいくつかのボリュームタイプがあり、用途に応じて選びます。試験で押さえるべき主なタイプは以下の通りです。

**gp3 / gp2（汎用SSD）**: 幅広い用途に使える標準的なボリュームタイプです。起動ボリューム（OSが入るディスク）やWebアプリケーションのデータ保存に使います。gp3はgp2の後継で、性能（読み書き速度やスループット）をボリュームの容量とは独立して設定できるため、コストパフォーマンスに優れています。新しくボリュームを作成する場合はgp3が推奨されています。

**io2 / io1（プロビジョンドIOPS SSD）**: 高いI/O性能（読み書き速度）が必要なワークロード向けです。大規模なデータベースなど、低遅延で大量の読み書きが求められる場面に使います。マルチアタッチ機能が使えるのは io2 です。

**st1（スループット最適化HDD）**: 大量のデータを連続して読み書きする用途に向いたHDDタイプです。ログデータの処理やビッグデータ分析などに適しています。起動ボリュームには使えません。

**sc1（コールドHDD）**: アクセス頻度が低い大量のデータを安く保存したい場合に使います。EBSボリュームタイプの中で最もコストが低いですが、性能も低めです。起動ボリュームには使えません。

### EBSスナップショット

EBSスナップショットは、EBSボリュームのバックアップ（ある時点のコピー）を作成する機能です。

重要なポイントは、スナップショットは **Amazon S3** に保存されるということです。S3のイレブンナインの耐久性でバックアップが守られます。また、スナップショットはリージョン内のどのAZでも使えるため、スナップショットから別のAZに新しいEBSボリュームを作成できます。これは、EBSボリューム自体は1つのAZに固定されるという制約を乗り越える方法でもあります。

---

> **試験のポイント**
> - 「EBSは基本的に1つのEC2にアタッチ」（io2のマルチアタッチは例外）
> - EBSは同一AZ内のEC2にしかアタッチできない
> - EBSスナップショットはS3に保存される
> - gp3（汎用SSD）が最もよく使われるボリュームタイプ。gp3はgp2の後継で新規作成時に推奨
> - HDDタイプ（st1, sc1）は起動ボリュームには使えない
> - インスタンスストアは高速だが一時的。EC2の停止・終了でデータが消える

---

## 3-6. Amazon EFS と Amazon FSx

### このセクションで学ぶこと

複数のサーバーからファイルを共有するためのファイルストレージサービスを理解します。EFS と FSx の使い分けは試験でも問われるポイントです。

### Amazon EFS

Amazon EFS（Elastic File System）は、複数のEC2インスタンスから同時にアクセスできる共有ファイルシステムです。

会社のネットワークにある「共有フォルダ」を思い浮かべてください。複数の社員が同じフォルダにアクセスして、ファイルの読み書きを同時にできますよね。EFSはこれと同じことをAWSのクラウド上で実現します。

EFSの大きな特長をまとめます。

EFSは **Linux** 向けのファイルシステムです。通信プロトコルとしてNFS（Network File System）を使います。Windows のEC2からは利用できません。

EFSは複数のAZにまたがってデータを保存するため、高い可用性と耐久性を備えています。同じリージョン内の複数のAZにあるEC2から同時にアクセスできます。

EFSはストレージ容量を自動的に拡張・縮小します。「最初に容量を決めておく」必要がありません。使った分だけ料金がかかる従量課金制です。

### EBSとEFSの違い

EBSとEFSは混同しやすいですが、決定的な違いがあります。

EBSは基本的に1つのEC2にしかアタッチできません。1つのAZ内でのみ使えます。容量はあらかじめ指定します。パソコンの「内蔵ハードディスク」のイメージです。

EFSは複数のEC2から同時にアクセスできます。複数のAZをまたいで使えます。容量は自動で伸縮します。会社の「共有ファイルサーバー」のイメージです。

### Amazon FSx

Amazon FSx は、特定の用途に最適化されたファイルシステムを提供するサービスです。EFSがLinux向けの汎用ファイル共有なのに対し、FSxはより専門的なニーズに応えます。

**FSx for Windows File Server** は、Windows 環境向けの共有ファイルシステムです。SMB（Server Message Block）プロトコルを使い、Windows のEC2インスタンスやオンプレミスの Windows サーバーからアクセスできます。Active Directory との統合もサポートしています。Windows環境でファイル共有が必要な場合はこちらを選びます。

**FSx for Lustre** は、高性能コンピューティング（HPC）や機械学習のような、大量のデータを高速に処理する必要があるワークロード向けです。S3と連携して、S3上のデータを高速にアクセスできるファイルシステムとして使うこともできます。

> **図解:** ch03-storage-selection.drawio.png — EBS / EFS / S3 の使い分けフローチャート
> （draw.io ファイルは `docs/images/ch03-storage-selection.drawio.png` に配置予定）

---

> **試験のポイント**
> - EFS = Linux / NFS、FSx for Windows = Windows / SMB という対応を確実に覚える
> - 「複数のEC2で共有できるストレージは？」→ EFS または FSx（EBSは基本的に不可）
> - EFSは容量の自動伸縮、EBSは事前に容量を指定
> - FSx for Lustre は高性能コンピューティング向け

---

## 3-7. ハイブリッドストレージとバックアップ

### このセクションで学ぶこと

オンプレミス（自社のデータセンター）とAWSクラウドを連携させるストレージサービスと、バックアップの一元管理サービスを理解します。

### AWS Storage Gateway

AWS Storage Gateway は、オンプレミスのサーバーとAWSクラウドのストレージ（主にS3）をつなぐ「橋渡し」のサービスです。

たとえば、長年オンプレミスでファイルサーバーを運用してきた企業が、一度にすべてのデータをクラウドに移行するのは大変です。Storage Gateway を使えば、オンプレミスのサーバーからはこれまで通りファイルにアクセスしつつ、裏側ではデータがS3に自動的に保存される——という環境を作れます。

よく使われるデータはオンプレミス側にキャッシュ（一時保存）されるので、アクセス速度も維持できます。まさに「オンプレミスとクラウドのいいとこ取り」です。

Storage Gateway は第14章（移行戦略）でも再度登場しますが、ここでは「オンプレミスとS3の橋渡しをするサービス」という役割を押さえておけば十分です。

### AWS Backup

AWS Backup は、AWSの複数のサービスのバックアップを一元的に管理できるサービスです。

バックアップの対象となるのは、EBSボリューム、RDSデータベース、DynamoDBテーブル、EFSファイルシステム、S3バケットなど、多岐にわたります。これらのバックアップを個別に設定・管理するのは大変ですが、AWS Backup を使えば「毎日午前2時にバックアップを取り、30日間保持する」といったポリシーを一か所でまとめて設定できます。

### AWS Elastic Disaster Recovery

AWS Elastic Disaster Recovery（DRS）は、オンプレミスや他のクラウドで動いているサーバーを、AWSに素早く復旧できるよう備えるサービスです。

災害や大規模障害が発生した場合に、AWSの環境でシステムを迅速に立ち上げ直すことができます。通常時はAWS側に軽量なコピーだけを維持するため、コストを抑えながら災害対策ができます。

---

> **試験のポイント**
> - Storage Gateway = 「オンプレミスとクラウド（S3）の橋渡し」。キャッシュによる高速アクセスもポイント
> - AWS Backup = 「複数AWSサービスのバックアップを一元管理」
> - Elastic Disaster Recovery = 「オンプレミスのサーバーをAWSに復旧する災害対策サービス」
> - Storage Gateway はハイブリッド構成のキーワードと一緒に出題されることが多い

### 問題1

Amazon S3 のデータ耐久性はどのくらいですか？

- **A.** 99.9%
- **B.** 99.99%
- **C.** 99.999999999%（イレブンナイン）
- **D.** 100%

<details>
<summary>解答・解説</summary>

**解答: C**

解説: Amazon S3 は 99.999999999%（9が11個、通称イレブンナイン）の耐久性を持つよう設計されています。これはS3がデータを自動的に3つ以上のAZに複製して保存しているためです。100%の耐久性を保証するサービスは存在しません。

</details>

### 問題2

年に1回程度しかアクセスしないコンプライアンスデータを、最も低コストで保存したい場合に適切なS3ストレージクラスはどれですか？

- **A.** S3 Standard
- **B.** S3 Standard-IA
- **C.** S3 Glacier Flexible Retrieval
- **D.** S3 Glacier Deep Archive

<details>
<summary>解答・解説</summary>

**解答: D**

解説: S3 Glacier Deep Archive は最も保存コストが安いストレージクラスです。年に1回程度しかアクセスせず、取り出しに12〜48時間かかっても問題ないコンプライアンスデータの保存に最適です。Glacier Flexible Retrieval も安価ですが、Deep Archive のほうがさらに低コストです。

</details>

### 問題3

Amazon EBS について正しい説明はどれですか？（2つ選択）

- **A.** 1つのEBSボリュームを複数のAZにまたがって使用できる
- **B.** EBSスナップショットはAmazon S3に保存される
- **C.** EBSボリュームは同一AZ内のEC2インスタンスにアタッチできる
- **D.** EBSは複数のEC2インスタンスからの同時アクセスに標準対応している
- **E.** EBSはオブジェクトストレージの一種である

<details>
<summary>解答・解説</summary>

**解答: B, C**

解説: EBSスナップショットはS3に保存されます（Bが正解）。EBSボリュームは特定のAZに作成され、同一AZ内のEC2にのみアタッチできます（Cが正解）。AをまたがるAZでの使用はできません。複数EC2への同時アタッチはio2のマルチアタッチ機能で可能ですが「標準対応」ではありません。EBSはブロックストレージです。

</details>

### 問題4

- **C.** Amazon EFS
- **D.** Amazon FSx for Windows File Server

<details>
<summary>解答・解説</summary>

**解答: C**

### 問題5

S3 ストレージクラスのうち、アクセスパターンが予測できないデータに最適なものはどれですか？

- **D.** S3 Glacier Flexible Retrieval

S3のライフサイクルポリシーで実現できることはどれですか？

S3 One Zone-IA について正しい説明はどれですか？

- **A.** データは3つ以上のAZに複製される

Amazon S3 のセキュリティについて正しい説明はどれですか？（2つ選択）

- **C.** S3 Block Public Access を有効にしないとデータを保存できない
- **D.** バケットポリシーを使ってバケットへのアクセス制御を設定できる

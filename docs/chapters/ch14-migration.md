# 第14章　クラウド移行戦略 ― オンプレミスからの移行

**対応試験ドメイン**: 第1分野: クラウドのコンセプト（24%）、第3分野: クラウドテクノロジーとサービス（34%）
**推定読了時間**: 30分

## この章で学ぶこと

「いまある自社システムをどうやってAWSに移行するか」という実務的なテーマを学びます。移行の動機から、具体的な戦略の選び方、AWSが提供する移行支援サービス、大量データの物理転送手段まで、移行プロジェクト全体の流れを理解することがこの章のゴールです。

- クラウド移行の動機と AWS Cloud Adoption Framework（CAF）の6つの観点
- 7つのR ― 移行戦略の選び方
- AWS の移行支援サービス（Migration Hub、Application Discovery Service など）
- Snow Family を使った大容量データの物理転送
- データベース移行の基本（DMS / SCT の復習）

## 14-1. なぜクラウドに移行するのか

### このセクションで学ぶこと

クラウド移行を検討する企業がどんな動機を持っているのか、そしてAWSが提唱する移行計画のフレームワーク「Cloud Adoption Framework（CAF）」の6つの観点を理解します。

### オンプレミスの限界

多くの企業は、もともと自社のサーバールームやデータセンターにサーバーを置いて、システムを動かしてきました。このような自社所有の設備でシステムを運用するやり方を、オンプレミス（略して「オンプレ」）と呼びます。

オンプレミスは「自分たちですべてをコントロールできる」という安心感がありますが、いくつかの限界があります。

まず、コストの問題です。サーバーを購入するのに数百万〜数千万円の初期費用がかかります。さらに、電気代や冷却設備の維持費、保守担当者の人件費が毎月かかり続けます。サーバーは3〜5年で老朽化するので、定期的に買い替えも必要です。

次に、俊敏性の問題です。新しいサービスを始めたくても、サーバーの発注から設置、ソフトウェアのインストールまで数週間〜数ヶ月かかることもあります。ビジネスのスピードについていけません。

さらに、グローバル展開の問題です。海外にサービスを展開したい場合、現地にデータセンターを建てるのは大変なコストと時間がかかります。

こうした課題を解決するために、多くの企業がクラウドへの移行を検討しています。AWSなら、初期費用なしで使った分だけ支払い、数分でサーバーを用意でき、世界中のリージョンをすぐに使えます。

### AWS Cloud Adoption Framework（CAF）

クラウドへの移行は、単に「サーバーをAWSに引っ越す」だけの話ではありません。組織体制、セキュリティ方針、運用方法など、企業全体で考えるべきことがたくさんあります。

AWS Cloud Adoption Framework（CAF、キャフと読みます）は、クラウド移行を成功させるために「どんな観点から準備すべきか」を6つにまとめたガイドラインです。引っ越しにたとえると、「荷物をまとめる」だけでなく「住所変更の届け出」「子どもの転校手続き」「ご近所への挨拶」など、やるべきことを漏れなくリスト化したチェックリストのようなものです。

6つの観点は次のとおりです。

ビジネス観点は、クラウド投資がビジネス成果につながるようにする視点です。「なぜクラウドに移行するのか」「どんなビジネス価値を得たいのか」を明確にします。

ピープル観点は、組織と人材の準備です。クラウドを使いこなすためのスキルアップやトレーニング、組織の変革管理に関わります。社員がクラウドを使えるように教育することも大事なのです。

ガバナンス観点は、リスク管理やコンプライアンス（法令遵守）の視点です。クラウド利用のルール作りや、コストの管理方法を定めます。

プラットフォーム観点は、技術基盤の設計です。ネットワーク構成、アカウント設計、運用の自動化など、AWSの環境をどう構築するかを決めます。

セキュリティ観点は、データ保護やアクセス制御の設計です。第9章で学んだIAMや暗号化の仕組みをクラウド環境にどう適用するかを考えます。

オペレーション観点は、日々の運用・監視の方法です。第10章で学んだCloudWatchなどを使って、クラウド環境をどう安定運用するかを設計します。

また、CAFでは移行を進めるプロセスとして4つのフェーズも定義されています。Envision（ビジョン策定：なぜクラウドに移行するのか、目指す姿を描く）、Align（体制整備：必要なスキルや組織体制を整える）、Launch（パイロット実行：小規模な移行を実施して成功体験を積む）、Scale（全社展開：パイロットの成果をもとに、残りのシステムを順次移行する）の4つです。試験では「CAFのフェーズ」として問われることがあるので、この流れも押さえておきましょう。

試験では「CAFの6つの観点をすべて暗記する」というより、「この課題はどの観点に関係するか」を判断できることが大切です。たとえば「社員のクラウドスキル研修計画」と聞いたら「ピープル観点」、「クラウド利用のコスト管理ルール」と聞いたら「ガバナンス観点」と結びつけられるようにしておきましょう。

---

> **試験のポイント**
> - CAFの6つの観点: ビジネス、ピープル、ガバナンス、プラットフォーム、セキュリティ、オペレーション
> - CAFの4つのフェーズ: Envision（ビジョン策定）→ Align（体制整備）→ Launch（パイロット実行）→ Scale（全社展開）
> - 各観点が「何を扱うか」を1行で説明できるレベルで押さえる
> - 「社員の教育 → ピープル」「コスト管理ルール → ガバナンス」のように、具体的なシナリオと観点を結びつける練習をする

## 14-2. 7つのRの移行戦略

### このセクションで学ぶこと

既存のシステムをクラウドに移行するとき、「どの方法で移行するか」を決めるための7つの戦略（7つのR）を学びます。これは試験で直接問われる頻出テーマです。

### 引っ越しにもいろいろなやり方がある

自宅を引っ越すとき、荷物の扱い方にはいくつかの選択肢がありますよね。今ある家具をそのまま新居に持っていく方法もあれば、古い家具を捨てて新居に合う家具を買い直す方法もあります。場合によっては「この荷物は使わないから捨ててしまおう」ということもあるでしょう。

クラウド移行もまったく同じです。すべてのシステムを同じ方法で移行する必要はなく、システムごとに最適な戦略を選びます。AWSでは、この戦略を7つのRとして整理しています。

![7つのRの比較チャート](../images/ch14-seven-rs.drawio.png)

### Rehost（リホスト）― そのまま引っ越し

Rehostは「リフト&シフト」とも呼ばれます。既存のシステムを、ほぼそのままの構成でAWSに移します。オンプレミスのサーバーで動いていたアプリケーションを、EC2インスタンス（AWSのレンタルサーバー）の上にそっくり載せ替えるイメージです。

引っ越しにたとえるなら、「今の家具をダンボールに詰めて、そのまま新居に運ぶ」方法です。手間は最小限で済み、素早く移行できます。ただし、クラウドならではのメリット（サーバーレスやマネージドサービスの活用）は、この段階では享受しにくいです。

Rehostは「まずはクラウドに移行して、最適化は後から考える」というスピード重視の場面でよく選ばれます。大量のサーバーを短期間で移行したいときに向いています。

### Replatform（リプラットフォーム）― 少しだけ手を加えて引っ越し

Replatformは「リフト・ティンカー&シフト」とも呼ばれます。ティンカー（tinker）は「ちょっと手を加える」という意味です。アプリケーションのコア部分は変えずに、一部だけクラウドに最適化して移行します。

たとえば、自前で管理していたデータベースをAmazon RDS（AWSのマネージドデータベース）に置き換える、といった変更です。アプリケーションのコード自体はほとんど変えませんが、インフラの一部をAWSのマネージドサービスに切り替えることで、運用負担を軽減できます。

引っ越しにたとえるなら、「大きな家具はそのまま持っていくけど、古い照明器具は新居に合うLED照明に買い替える」ようなものです。

### Refactor / Re-architect（リファクタ / リアーキテクト）― 作り直し

Refactorは、アプリケーションの設計そのものを見直し、クラウドネイティブな技術を使って作り直す方法です。たとえば、ひとつの大きなアプリケーション（モノリス）を、複数の小さなサービスに分割するマイクロサービス化や、サーバーレスアーキテクチャへの移行が該当します。

引っ越しにたとえるなら、「古い家を解体して、新居のライフスタイルに合わせた注文住宅を建て直す」ようなものです。時間もコストもかかりますが、クラウドのメリットを最大限に活かせます。スケーラビリティの向上、運用コストの削減、開発スピードの加速など、長期的なリターンが大きい戦略です。

### Repurchase（リパーチェス）― 買い替え

Repurchaseは、既存のシステムを捨てて、別のSaaS（クラウド上のソフトウェアサービス）製品に乗り換える方法です。

たとえば、自社で運用していたメールサーバーをやめて、Google WorkspaceやMicrosoft 365に移行するケースがこれに当たります。自社開発のCRM（顧客管理）システムをSalesforceに置き換える、というのもRepurchaseです。

引っ越しにたとえるなら、「古い家電を処分して、新しい製品を買い直す」ようなものです。

### Retire（リタイア）― 廃止する

Retireは、「もう使わないシステムを廃止する」という選択肢です。移行プロジェクトで既存のシステムを棚卸ししてみると、「実はもう誰も使っていないシステム」「別のシステムと機能が重複しているもの」が見つかることがあります。こうしたシステムは、移行するのではなく、思い切って廃止してしまいます。

引っ越しにたとえるなら、「もう使わないものは引っ越しの前に断捨離する」ことです。移行対象が減ることで、コストと手間が削減できます。

### Retain（リテイン）― 現状維持

Retainは、「今はクラウドに移行せず、オンプレミスに残しておく」という選択肢です。移行のリスクが高いシステム、規制上クラウドに移せないデータ、まだ移行の準備が整っていないシステムなどが対象になります。

引っ越しにたとえるなら、「今すぐには持っていかず、しばらく元の場所に置いておく」ことです。ただし、将来的には移行を再検討することが前提です。

### Relocate（リロケート）― そのまま移動

Relocateは、仮想マシン（VM：1台の物理サーバーの中にソフトウェアで仮想的に作った"仮想のサーバー"のこと）やコンテナ環境をそのまま、VMware（仮想マシンを管理するための代表的なソフトウェア）などの仮想化基盤（仮想マシンを動かすための土台となるシステム）ごとAWSに移す方法です。Rehostと似ていますが、個別のサーバーではなく、仮想化基盤ごとまとめて移動する点が異なります。VMware Cloud on AWSを使うケースがこれに該当します。なお、VMware Cloud on AWSは提供状況が変化しているため、最新の情報はAWS公式ドキュメントで確認することをおすすめします。

引っ越しにたとえるなら、「家具を一つずつ運ぶのではなく、プレハブの部屋ごとクレーンで持ち上げて新しい土地に設置する」ようなイメージです。

### Rehost / Replatform / Refactor の使い分け

試験では特にこの3つの違いが問われます。ポイントを整理しておきましょう。

Rehostは、アプリケーションもインフラもほぼそのまま移行します。移行スピードは最も速く、工数も最小限ですが、クラウド最適化の度合いは低いです。「まず移行を完了させたい」ときに選びます。AWSサービスとしてはEC2とApplication Migration Service（MGN）を組み合わせて実行するのが典型的です。

Replatformは、アプリケーションのコードはほぼ変えず、インフラの一部をマネージドサービスに置き換えます。移行スピードと最適化のバランスが良い選択肢です。「少しの手間で運用負担を減らしたい」ときに選びます。たとえば、自前のデータベースをAmazon RDSに、自前のキャッシュサーバーをAmazon ElastiCacheに置き換えるようなケースです。

Refactorは、アプリケーションの設計から作り直します。工数とコストは最大ですが、クラウドのメリットを最大限に活かせます。「長期的な競争力を高めたい」ときに選びます。たとえば、モノリスなアプリケーションをAWS Lambda + API Gateway + DynamoDBを使ったサーバーレス構成に再設計するようなケースです。

---

> **試験のポイント**
> - 7つのR: Rehost, Replatform, Refactor, Repurchase, Retire, Retain, Relocate
> - Rehost =「そのまま移行」、Replatform =「少し最適化して移行」、Refactor =「作り直し」が最頻出
> - Rehostは「リフト&シフト」、Replatformは「リフト・ティンカー&シフト」という別名も覚えておく
> - シナリオ問題で「どの戦略を選ぶべきか」を判断できるようにする

## 14-3. 移行支援サービス

### このセクションで学ぶこと

AWSが提供する移行支援サービスの役割を理解します。「何を検出し」「どう計画し」「どう実行するか」という移行プロセスの各フェーズに対応するサービスを把握しましょう。

### 移行は「引っ越し業者」に任せる

大規模な引っ越しを個人で全部やるのは大変です。引っ越し業者に頼めば、荷物の梱包から運搬、設置まで手伝ってくれます。AWSにもクラウド移行を支援する専用のサービスが揃っています。

移行の大まかな流れは、「既存環境の検出」→「移行計画の策定」→「移行の実行」→「最適化」という4つのフェーズです。

![移行プロセスのフロー図](../images/ch14-migration-flow.drawio.png)

### AWS Application Discovery Service ― 既存環境の棚卸し

まず移行の第一歩は、「今、何がどこで動いているかを把握する」ことです。意外かもしれませんが、大きな企業ほど「社内にどんなサーバーが何台あって、どんなアプリが動いているのか」を正確に把握できていないことがあります。

AWS Application Discovery Serviceは、オンプレミス環境にあるサーバーやアプリケーションの情報を自動的に収集するサービスです。サーバーのスペック（CPU、メモリ、ディスク容量）、インストールされているソフトウェア、ネットワークの通信パターンなどを調べてくれます。

引っ越しにたとえるなら、「まず家の中の荷物を全部リストアップする」作業です。何を持っていくか決める前に、何があるかを知る必要があります。

### AWS Migration Hub ― 移行の進捗ダッシュボード

AWS Migration Hubは、移行プロジェクト全体の進捗を一元的に管理するサービスです。複数の移行ツールを使っていても、Migration Hubのダッシュボード（管理画面）ですべての移行状況をまとめて確認できます。

引っ越しにたとえるなら、「引っ越し全体のスケジュール表」です。「キッチン用品の梱包は完了」「リビングの家具は搬出待ち」「新居への搬入は明日」といった進捗を、一つの画面で確認できます。

Migration Hubは自分で直接サーバーを移行するわけではなく、他の移行サービス（Application Migration Serviceなど）と連携して、その進捗を表示する「司令塔」の役割です。

### AWS Application Migration Service（MGN）― サーバー移行の自動化

AWS Application Migration Service（旧称 CloudEndure Migration、略称 MGN）は、オンプレミスのサーバーをAWSに移行する作業を自動化するサービスです。移行元のサーバーにエージェント（小さなソフトウェア）をインストールすると、サーバーのデータをAWSに継続的にレプリケーション（複製）してくれます。そして、準備ができたら切り替え（カットオーバー）を実行して移行を完了します。

つまり、MGNの動作は3つのステップで理解できます。(1) 移行元のサーバーに小さなソフト（エージェント）を入れる、(2) データが自動的にAWSにコピーされ続ける（レプリケーション）、(3) 準備ができたら本番をAWS側に切り替える（カットオーバー）、という流れです。

このサービスは特にRehostの戦略を実行するときに使われます。サーバーの中身をそっくりそのままAWSに「リフト&シフト」できるのです。

### Migration Evaluator ― 移行コストの見積もり

「クラウドに移行したら本当にコスト削減になるの？」という疑問に答えてくれるのが、Migration Evaluatorです。現在のオンプレミス環境のコスト（サーバー購入費、電気代、人件費など）を分析し、AWSに移行した場合のコストと比較するレポートを作成してくれます。

移行プロジェクトを始める前の「ビジネスケース作り」に役立つサービスです。ビジネスケースとは、移行の費用対効果をまとめた提案資料のことです。たとえば「年間1,000万円のサーバー維持費が、AWSなら600万円になります」といった比較レポートを作り、上層部に移行の意義を説明するために使います。

### ハイブリッド接続 ― 移行中のオンプレミスとAWSをつなぐ

移行は一夜にして完了するものではありません。移行期間中は、一部のシステムがまだオンプレミスに残り、一部はすでにAWSで動いているという「ハイブリッド（混在）」な状態が続きます。この期間、オンプレミスとAWSの間を安全かつ安定的に接続する手段が必要です。

AWS Direct Connectは、オンプレミスのデータセンターとAWSの間を専用の物理回線でつなぐサービスです。インターネットを経由しないため、通信が安定しており、大量のデータをやり取りする移行期間中に特に有効です。第5章で学んだ内容の復習になりますが、移行の文脈では「移行期間中の安定したデータ転送経路」として覚えておきましょう。

AWS Site-to-Site VPNは、インターネット経由で暗号化されたトンネルを作り、オンプレミスとAWSを安全に接続するサービスです。Direct Connectほどの安定性はありませんが、導入が比較的簡単で、コストも抑えられます。移行の初期段階や、Direct Connectの開通までのつなぎとして使われることがあります。

---

> **試験のポイント**
> - Application Discovery Service = 「既存環境の棚卸し」（何が動いているかを調べる）
> - Migration Hub = 「移行の進捗ダッシュボード」（全体の進捗を一元管理）
> - Application Migration Service（MGN）= 「サーバー移行の自動化」（Rehostの実行ツール）
> - Migration Evaluator = 「移行コストの見積もり」（ビジネスケース作成）
> - 移行中のハイブリッド接続: Direct Connect（専用回線）とSite-to-Site VPN（暗号化トンネル）
> - 各サービスの役割を1行で説明できるレベルに押さえる

## 14-4. 大容量データの物理転送 ― Snow Family

### このセクションで学ぶこと

ネットワーク経由では現実的でないほど大量のデータを、物理的なデバイスを使ってAWSに転送する方法を学びます。Snow Familyは試験で頻出のテーマです。

### なぜ物理転送が必要なのか

ネットワーク回線でクラウドにデータを転送すればいいのでは、と思うかもしれません。確かに、数GBや数十GBのデータならネットワーク転送で問題ありません。しかし、数十TB（テラバイト）、数百TB、さらにはPB（ペタバイト）やEB（エクサバイト）級のデータとなると、話が変わってきます。

たとえば、100TBのデータを一般的な1Gbpsの回線で転送しようとすると、理論上でも約10日かかります。実際にはネットワークの混雑もあるので、さらに長くなります。10PBのデータなら数年かかる計算です。

こうした「ネットワーク転送では現実的に時間がかかりすぎる」問題を解決するのがSnow Familyです。AWSから物理的なデバイスが届き、データをコピーしてAWSに返送するという、まさに「データの引っ越しトラック」です。

![Snow Family のデータ量対応図](../images/ch14-snow-family.drawio.png)

### AWS Snowcone ― 小型・軽量デバイス

Snowconeは、Snow Familyの中で最も小さなデバイスです。重さ約2.1kgと片手で持てるサイズで、8TB（Snowcone）または14TB（Snowcone SSD）のストレージ容量を持ちます。

Snowconeは、オフィスや工場、遠隔地など、ネットワーク環境が限られた場所でのデータ収集に向いています。ドローンで収集したデータや、辺境地のセンサーデータをSnowconeに保存してAWSに送る、といった使い方ができます。

### AWS Snowball Edge ― 数十〜数百TBの転送

Snowball Edgeは、数十TB〜数百TBのデータ転送に使われるスーツケースサイズのデバイスです。ストレージ最適化モデルでは80TBのストレージ容量があり、コンピューティング最適化モデルでは42TBのストレージに加えて高い処理能力を持ちます。

Snowball Edgeはただのストレージデバイスではなく、デバイス上でデータの前処理（加工や分析）を実行する機能も備えています。これを「エッジコンピューティング」と呼びます。たとえば、船上や工場内などネットワーク接続が不安定な環境で、IoTデータの前処理やML（機械学習）の推論を実行するといった使い方もできます。複数台を組み合わせることで、PB級のデータ転送にも対応できます。

### AWS Snowmobile ― エクサバイト級の超大規模転送

Snowmobileは、最大100PB（ペタバイト）のデータを一度に転送できる、文字通りの「トラック」です。全長約14メートルのコンテナ型トラックがデータセンターに乗り付けて、大量のデータを物理的に運びます。

EB（エクサバイト）級のデータ移行が必要な超大規模プロジェクトで使われます。なお、Snowmobileは現在新規の利用受付が制限されています。試験では選択肢として登場することがありますが、出題の中心はSnowball Edgeです。

### データ量に応じた選び方

試験では「どのくらいのデータ量のときにどのデバイスを選ぶか」が問われます。判断基準はシンプルです。

少量のデータ（数GB〜数TB）であれば、ネットワーク転送（インターネットやDirect Connect）で十分です。Snow Familyを使う必要はありません。

中〜大規模のデータ（数十TB〜数百TB）であれば、Snowball Edgeが適しています。最もよく使われるSnowデバイスです。

超大規模のデータ（数十PB〜EB級）であれば、Snowmobileを検討します。

小規模またはネットワークが限られた環境（〜14TB）では、Snowconeが適しています。

第3章で学んだStorage Gatewayがオンプレミスとクラウドの「日常的なデータ連携」に使われるのに対し、Snow Familyは「大量データの一括移行」に使われるという違いも押さえておきましょう。

---

> **試験のポイント**
> - Snow Family: Snowcone（小型、8〜14TB）、Snowball Edge（数十〜数百TB）、Snowmobile（100PB、トラック）
> - 「大量データをクラウドに移行したい」→ Snow Family
> - データ量に応じたデバイス選択を判断できるようにする
> - Snowball Edgeはデバイス上でのコンピューティング（データ処理）も可能
> - Snowmobileはトラックで物理的に運搬するという規模感を覚えておく

## 14-5. データベース移行

### このセクションで学ぶこと

データベースの移行で使われるAWS DMS（Database Migration Service）とAWS SCT（Schema Conversion Tool）を、移行戦略の文脈で復習します。詳細は第4章で学んだ内容を参照してください。

### サーバーだけでなくデータベースも移行が必要

システムをクラウドに移行するとき、アプリケーションが動くサーバーだけでなく、データベースも一緒に移行する必要があります。しかし、データベースの移行にはいくつか厄介な問題があります。「移行中もサービスを止めたくない」「データベースの種類を変えたい（例: Oracle → Amazon Aurora）」「データ量が大きくて時間がかかる」といった課題です。

### AWS DMS ― データベースの中身を移行

AWS Database Migration Service（DMS）は、データベースのデータをオンプレミスからAWSに移行するサービスです。第4章で詳しく学びましたが、移行戦略の文脈で改めて整理しておきましょう。

DMSの大きな特長は、移行中もデータベースを使い続けられる点です。移行元のデータベースが稼働したまま、データをAWSのデータベースにコピーしてくれます。さらに、継続的なレプリケーション（データの同期）にも対応しているため、カットオーバー（本番切り替え）の直前までデータを同期し続けることができます。

### 同種間移行と異種間移行

データベースの移行には、大きく2つのパターンがあります。

同種間移行（ホモジニアス移行）は、同じ種類のデータベース同士の移行です。たとえば「MySQL → Amazon RDS for MySQL」や「PostgreSQL → Amazon Aurora PostgreSQL互換」のような移行です。データベースの構造（スキーマ）が同じなので、DMSだけで移行できます。

異種間移行（ヘテロジニアス移行）は、異なる種類のデータベース間の移行です。たとえば「Oracle → Amazon Aurora」や「SQL Server → Amazon RDS for PostgreSQL」のような移行です。この場合、データベースの構造や方言（SQL文法の違い）を変換する必要があるため、AWS SCT（Schema Conversion Tool）を使って、まずスキーマを変換してから、DMSでデータを移行します。

7つのRとの関係でいうと、同種間のデータベース移行はReplatformに該当することが多いです。一方、異種間移行でデータベースエンジンを変更する場合は、Refactorの要素が含まれます。

DMSとSCTの詳細については、第4章を復習してください。

---

> **試験のポイント**
> - DMS = データベースのデータを移行（移行中もDB稼働可能）
> - SCT = 異種間移行でスキーマ（データベース構造）を変換するツール
> - 同種間移行 → DMSのみ、異種間移行 → SCT + DMS
> - 詳細は第4章を参照

---

## まとめ ― 移行後は最適化へ

この章では、オンプレミスからAWSへの移行戦略と、それを支えるAWSサービスを学びました。

移行はゴールではなく、スタートです。AWSに移行した後は、クラウドの特性を活かしてコストの最適化やアーキテクチャの改善を続けていくことが大切です。移行後のコスト最適化については、第15章で詳しく学びます。

## 章末確認問題

### 問題1

自社で運用しているWebアプリケーションを、コードをほとんど変更せず、できるだけ短期間でAWSに移行したいと考えています。最も適した移行戦略はどれですか？

- **A.** Refactor
- **B.** Replatform
- **C.** Rehost
- **D.** Repurchase

<details>
<summary>解答・解説</summary>

**解答: C**

解説: **Rehost（リフト&シフト）**は、既存のシステムをほぼそのままの構成でAWSに移行する戦略です。コード変更が最小限で、移行スピードが最も速い方法です。Refactorはアプリケーションの設計から作り直す方法、Replatformは一部をマネージドサービスに置き換える方法です。

</details>

### 問題2

AWS Cloud Adoption Framework（CAF）において、クラウド移行に必要な社員のスキルアップやトレーニング計画は、どの観点に該当しますか？

- **A.** ビジネス観点
- **B.** ピープル観点
- **C.** ガバナンス観点
- **D.** オペレーション観点

<details>
<summary>解答・解説</summary>

**解答: B**

解説: **ピープル観点**は、クラウドを使いこなすための人材育成や組織の変革管理を扱います。社員のスキルアップやトレーニング計画はこの観点に該当します。

</details>

### 問題3

数百TBのデータをオンプレミスのデータセンターからAWSに移行する必要があります。ネットワーク経由では数週間かかると見積もられています。最も適切なAWSサービスはどれですか？

- **A.** AWS Snowcone
- **B.** AWS Snowball Edge
- **C.** AWS Direct Connect
- **D.** AWS Snowmobile

<details>
<summary>解答・解説</summary>

**解答: B**

解説: **AWS Snowball Edge**は、数十TB〜数百TBのデータを物理デバイスでAWSに転送するサービスです。Snowconeは最大14TBまでの小型デバイス、Snowmobileは数十PB〜EB級の超大規模データ向けです。Direct Connectは専用回線ですが、数百TBのデータ転送には時間がかかります。

</details>

### 問題4

既存のアプリケーションのコードはほぼ変更せず、自前で管理していたデータベースをAmazon RDSに置き換えてAWSに移行する戦略は、7つのRのどれに該当しますか？

- **A.** Rehost
- **B.** Replatform
- **C.** Refactor
- **D.** Relocate

<details>
<summary>解答・解説</summary>

**解答: B**

解説: **Replatform（リフト・ティンカー&シフト）**は、アプリケーションのコードはほぼ変えず、インフラの一部をAWSのマネージドサービスに置き換える戦略です。データベースを自前管理からRDSに移行するのは、Replatformの典型的な例です。

</details>

### 問題5

AWS Migration Hubの主な役割として最も適切なものはどれですか？

- **A.** オンプレミスのサーバーを自動的にAWSに移行する
- **B.** 移行プロジェクト全体の進捗を一元的に管理・追跡する
- **C.** オンプレミス環境にあるサーバーの情報を自動収集する
- **D.** AWSへの移行コストを見積もるレポートを作成する

<details>
<summary>解答・解説</summary>

**解答: B**

解説: **AWS Migration Hub**は、移行プロジェクト全体の進捗を一元的に管理するダッシュボードの役割を果たすサービスです。Aの自動移行はApplication Migration Service（MGN）、Cのサーバー情報収集はApplication Discovery Service、Dのコスト見積もりはMigration Evaluatorの役割です。

</details>

### 問題6

オンプレミスのOracleデータベースをAmazon Auroraに移行する場合、必要なAWSサービスの組み合わせとして正しいものはどれですか？

- **A.** AWS DMSのみ
- **B.** AWS SCTのみ
- **C.** AWS SCT と AWS DMS
- **D.** AWS Migration Hub と AWS DMS

<details>
<summary>解答・解説</summary>

**解答: C**

解説: OracleからAuroraへの移行は「異種間移行（ヘテロジニアス移行）」です。まず**AWS SCT**でデータベースのスキーマ（構造）を変換し、次に**AWS DMS**でデータを移行します。同種間移行であればDMSのみで済みますが、異なるデータベースエンジン間の移行にはSCTによるスキーマ変換が必要です。

</details>

### 問題7

あるアプリケーションをクラウドネイティブに再設計し、サーバーレスアーキテクチャで作り直す移行戦略は、7つのRのどれに該当しますか？

- **A.** Rehost
- **B.** Replatform
- **C.** Repurchase
- **D.** Refactor

<details>
<summary>解答・解説</summary>

**解答: D**

解説: **Refactor（Re-architect）**は、アプリケーションの設計を見直し、クラウドネイティブな技術（サーバーレス、マイクロサービスなど）を使って作り直す戦略です。工数とコストは最大ですが、クラウドのメリットを最大限に活かせます。

</details>

次のうち、AWS Cloud Adoption Framework（CAF）の6つの観点に含まれるものを2つ選んでください。（2つ選択）

- **B.** ピープル観点

アプリケーションが扱うデータを安全に保管し、素早く取り出すための「データベース」について学びます。AWSが提供するさまざまなデータベースサービスの特徴と使い分けを理解することが、この章のゴールです。

- リレーショナルDB（RDB）とNoSQLの違い
- マネージドデータベースのメリット
- Amazon RDS / Aurora / DynamoDB の特徴と使い分け
- キャッシュ・グラフDB・ドキュメントDBなどの専門データベース
- AWS DMS / SCT によるデータベース移行

## 4-1. データベースの基本概念

### このセクションで学ぶこと

データベースの種類（リレーショナルDBとNoSQL）の違いと、AWSの「マネージドデータベース」がなぜ便利なのかを理解します。

### なぜデータベースが必要なのか

Webアプリケーションやスマートフォンアプリは、さまざまなデータを扱います。ユーザー情報、注文履歴、商品カタログ、チャットのメッセージ……。これらのデータをファイルにそのまま書き込んで管理するのは、検索が遅くなったり、複数人が同時に書き込むと壊れたりと、問題が山積みです。

データベースは、こうしたデータを整理して保管し、必要なときに素早く正確に取り出すための専用ソフトウェアです。「データの図書館」とイメージしてください。本（データ）をきちんと分類して書棚に並べ、検索システムで素早く見つけ出せるようにしたものです。

### リレーショナルデータベース（RDB）

リレーショナルデータベースは、データを「テーブル」と呼ばれる表の形で管理します。Excel のスプレッドシートを思い浮かべてください。行（レコード）と列（カラム）でデータが整理されています。

たとえば「ユーザーテーブル」なら、行ごとに1人のユーザー情報が入り、列には「ID」「名前」「メールアドレス」「登録日」といった項目が並びます。テーブル同士を関連付け（リレーション）できるのが大きな特徴で、「注文テーブル」から「ユーザーテーブル」を参照して「誰がどの商品を買ったか」を効率的に検索できます。

RDBの操作には **SQL（Structured Query Language、読み方：エスキューエル）** という専用の問い合わせ言語を使います。「SELECT * FROM users WHERE age > 30」のように書くと、30歳以上のユーザーを一覧で取得できます。

RDBは、データの整合性（正確さ）が厳しく求められるシステムに向いています。銀行の口座管理や、ECサイトの注文処理がその典型です。

### NoSQL

一方、NoSQL は「Not Only SQL」の略で、RDBとは異なるデータ構造を持つデータベースの総称です。RDBの「テーブル形式」にとらわれず、もっと柔軟にデータを扱えます。

NoSQLにはいくつかの種類があります。

**キーバリュー型** は、「名前（キー）」と「値（バリュー）」のペアでデータを管理します。ロッカーに番号をつけて荷物を入れるようなイメージです。番号さえ分かれば一瞬で中身を取り出せます。シンプルな構造なので、読み書きが非常に高速です。

**ドキュメント型** は、JSONやXMLのような柔軟な形式でデータを保存します。1つのドキュメント（データのかたまり）に、商品名、価格、レビューなど関連する情報をまとめて入れられます。

**グラフ型** は、データ同士のつながり（関係性）を重視したデータベースです。SNSの「友達の友達」を探したり、レコメンデーション（おすすめ商品の提案）をしたりするのに向いています。

### RDB と NoSQL、どちらを選ぶか

選び方の基本的な考え方はこうです。データの構造がきっちり決まっていて整合性が重要なら RDB。データ構造が柔軟に変わる可能性があり、大量のデータを高速に処理したいなら NoSQL。ただし、どちらか一方だけを使うのではなく、用途に応じて組み合わせるのが一般的です。

![データベースの種類とAWSサービスの対応](../images/ch04-db-types.drawio.png)

### マネージドデータベースのメリット

データベースを使う方法には大きく2つあります。1つは EC2 インスタンス（仮想サーバー）に自分で MySQL などのデータベースソフトをインストールして運用する方法。もう1つは Amazon RDS のような **マネージドサービス** を使う方法です。

「マネージド」とは「AWSが管理してくれる」という意味です。自分でデータベースソフトをインストール・運用する場合、OSのアップデート、セキュリティパッチの適用、バックアップの設定、障害時の対応……と、やるべきことがたくさんあります。マネージドサービスなら、こうした運用作業の大部分をAWSが代行してくれます。

EC2に自分でMySQLをインストールする場合と、Amazon RDS を使う場合の違いを整理すると、次のようになります。

| 項目 | EC2 + 自前MySQL | Amazon RDS |
|------|-----------------|------------|
| OSのパッチ適用 | 自分で対応 | AWSが対応 |
| DBエンジンのアップデート | 自分で対応 | AWSが対応（自動化可能） |
| バックアップ | 自分で設定・管理 | 自動バックアップ機能あり |
| 障害時のフェイルオーバー | 自分で構築 | マルチAZで自動化 |
| スケーリング | 自分で構築 | リードレプリカで読み取り分散 |

マネージドサービスを使うことで、開発者はデータベースの運用ではなく、アプリケーション開発そのものに集中できます。これは第0章で学んだ「クラウドのメリット」の実践例です。

---

> **試験のポイント**
> - 「EC2に自分でDBをインストール」vs「マネージドサービス（RDS等）」の違いは頻出
> - マネージドDBのメリット = バックアップ自動化・パッチ自動適用・高可用性構成の簡素化
> - RDB = テーブル形式・SQL・整合性重視、NoSQL = 柔軟な構造・高速・大規模データ向き

## 4-2. Amazon RDS

### このセクションで学ぶこと

AWSの代表的なマネージドリレーショナルデータベースサービスである Amazon RDS の特徴を理解します。特に「マルチAZ」と「リードレプリカ」の違いは試験の頻出ポイントです。

### Amazon RDS とは

Amazon RDS（Relational Database Service）は、AWSが提供するマネージドなリレーショナルデータベースサービスです。わかりやすく言えば「AWSがデータベースの面倒な管理作業を代行してくれるサービス」です。

RDS は以下の6つのデータベースエンジンに対応しています。

- MySQL
- PostgreSQL
- MariaDB
- Oracle Database
- Microsoft SQL Server
- Amazon Aurora（次のセクションで詳しく解説します）

すでに使い慣れたデータベースエンジンをそのまま使えるので、既存のアプリケーションをクラウドに移行する際にも便利です。

### 自動バックアップ

RDS では自動バックアップ機能が標準で備わっています。最大35日間のバックアップを保持でき、特定の時点にデータベースを復元する「ポイントインタイムリカバリ（特定の時点へのデータ復元）」も可能です。「昨日の14時の状態に戻したい」という要望にも対応できるわけです。

### マルチAZデプロイ

マルチAZデプロイは、データベースの**高可用性**を実現するための仕組みです。

「マルチAZ」とは、第0章で学んだアベイラビリティゾーン（AZ）を複数使うという意味です。プライマリ（メイン）のデータベースを1つのAZに配置し、別のAZに**スタンバイ（待機用）のコピー**を自動的に作成します。プライマリとスタンバイの間ではデータが**同期的にレプリケーション（複製）**されるため、常に同じ内容が保持されます。「同期的」とは、プライマリにデータを書き込むと、スタンバイへのコピーが完了するまで書き込み処理が終わらない方式です。少し時間がかかりますが、プライマリとスタンバイのデータが常に一致しているので安心です。

プライマリのデータベースが障害で停止した場合、AWSが自動的にスタンバイに切り替えます。この切り替えを**フェイルオーバー**と呼びます。AWSが自動的にDNSエンドポイント（接続先の住所のようなもの）を切り替えるため、アプリケーション側での接続先変更は不要です。

ここで重要なのは、マルチAZのスタンバイは**読み取り用途では使えない**ということです。あくまでも障害発生時の「保険」として待機しているだけです。

### リードレプリカ

リードレプリカは、データベースの**読み取りパフォーマンス**を向上させるための仕組みです。

Webサイトでは、データの書き込みより読み取りのほうが圧倒的に多いのが一般的です。商品情報を表示する（読み取り）回数は、新しい商品を登録する（書き込み）回数よりずっと多いですよね。リードレプリカは、プライマリDBの読み取り専用コピーを作成し、読み取り処理を分散させます。

プライマリとリードレプリカの間は**非同期レプリケーション**です。これは、プライマリへの書き込みが完了した後、少し遅れてリードレプリカにコピーが反映される方式です。マルチAZの「同期レプリケーション」と比べると高速に書き込めますが、リードレプリカのデータが最新ではない瞬間がわずかに生じます。

![マルチAZとリードレプリカの違い](../images/ch04-rds-multiaz-vs-readreplica.drawio.png)

### マルチAZ vs リードレプリカの違いを整理する

この2つの違いは試験で非常によく問われます。しっかり区別しましょう。

| | マルチAZ | リードレプリカ |
|---|---------|---------------|
| 目的 | **高可用性**（障害対策） | **読み取りパフォーマンス向上** |
| レプリケーション | 同期 | 非同期 |
| 読み取りに使えるか | 使えない（スタンバイ専用） | 使える（読み取り専用） |
| フェイルオーバー | 自動で切り替わる | 手動で昇格が必要 |
| 作成可能な数 | 1つ（スタンバイ） | Aurora以外: 最大5つ、Aurora: 最大15個 |

覚え方のコツとして、マルチAZ は「災害時の非常口」、リードレプリカは「混雑を解消するための窓口の増設」とイメージしてください。

### Amazon RDS Proxy

Amazon RDS Proxy は、データベースへの接続を効率的に管理する**プロキシ（仲介役）サービス**です。

たとえば Lambda 関数のように、短い時間で大量の接続が発生するケースを考えてみましょう。Lambda は同時に何百・何千と起動されることがありますが、データベースは同時に受け付けられる接続数に限りがあります。RDS Proxy を間に挟むことで、接続を使い回し（プーリング）してデータベースへの負荷を軽減できます。

サーバーレスアーキテクチャ（Lambda + RDS の組み合わせなど）では特に効果的なサービスです。

---

> **試験のポイント**
> - マルチAZ = 高可用性（フェイルオーバー）、リードレプリカ = 読み取りパフォーマンス向上。この違いは必ず押さえる
> - マルチAZは同期レプリケーション、リードレプリカは非同期レプリケーション
> - 「読み取りの負荷を軽減したい」→ リードレプリカ、「障害に備えたい」→ マルチAZ
> - RDS Proxy = Lambda等からの大量接続を効率化するプロキシサービス
> - RDS は6つのエンジンに対応（MySQL, PostgreSQL, MariaDB, Oracle, SQL Server, Aurora）

---

## 4-3. Amazon Aurora

### このセクションで学ぶこと

AWSが独自に設計した高性能データベースエンジン「Aurora」の特徴と、Aurora Serverless について理解します。

### Amazon Aurora とは

Amazon Aurora は、AWS が独自に設計・開発したリレーショナルデータベースエンジンです。Amazon RDS のデータベースエンジンの1つとして提供されていますが、他のエンジン（MySQL、PostgreSQLなど）とは根本的に異なるアーキテクチャを持っています。

Aurora の大きな特徴は **MySQL および PostgreSQL との互換性** を保ちながら、はるかに高いパフォーマンスを発揮する点です。具体的には、MySQL互換の場合は標準MySQLの**最大5倍**、PostgreSQL互換の場合は標準PostgreSQLの**最大3倍**のスループット（処理能力）を実現するとされています。

既存のMySQLやPostgreSQLのアプリケーションコードやドライバーをほぼそのまま使えるため、「今まで使っていたMySQLの資産を活かしつつ、性能を大幅に上げたい」というケースに適しています。

### Aurora のストレージ

Aurora のストレージには独自の工夫があります。データは自動的に**3つのAZにまたがって6つのコピー**として保存されます。つまり、1つや2つのAZに障害が起きても、データが失われることはありません。

さらに、ストレージは使用量に応じて**自動的に拡張**されます。最初に大きな容量を確保しておく必要がなく、10GBから始めて最大128TBまで自動で増えていきます。ストレージの容量不足を心配する必要がないのは、運用上とても助かるポイントです。

### Aurora Serverless

Aurora Serverless は、データベースの処理能力を**ワークロード（実際の負荷）に応じて自動的にスケーリング**してくれるオプションです。

たとえば、社内の経費精算システムのように、月末はアクセスが集中するけれど普段はほとんど使われないシステムがあるとします。従来なら、ピーク時に合わせて大きなデータベースを用意しておく必要がありました。Aurora Serverless なら、負荷が高いときは自動で処理能力を増やし、アクセスがないときは一時停止してコストを抑えることができます。

Aurora Serverless は「使った分だけ支払う」モデルなので、アクセスパターンが読めないアプリケーションや、開発・テスト環境にも適しています。

### Aurora Global Database

Aurora Global Database は、複数のAWSリージョンにまたがって Aurora データベースをレプリケーションする機能です。たとえば、東京リージョンをメインにしつつ、米国や欧州のリージョンにも読み取り用のコピーを配置できます。万が一メインのリージョン全体に障害が発生した場合も、別リージョンに切り替えて運用を継続できます。

DynamoDB にも同様の「Global Tables」（次のセクションで紹介）がありますが、Aurora Global Database はリレーショナルデータベースのクロスリージョン対応が必要なケースで使います。

---

> **試験のポイント**
> - Aurora は RDS のエンジンの1つだが、AWS独自設計で高性能
> - MySQL互換で最大5倍、PostgreSQL互換で最大3倍のスループット（倍率の違いに注意）
> - データは3AZ × 6コピーで高い耐障害性
> - Aurora Serverless = ワークロードに応じた自動スケーリング、可変ワークロードに最適
> - Aurora Global Database = 複数リージョンへのレプリケーション、リージョン障害への対策

---

## 4-4. Amazon DynamoDB

### このセクションで学ぶこと

AWSのフルマネージド NoSQL データベースサービスである DynamoDB の特徴と、RDS/Aurora との使い分けを理解します。

### Amazon DynamoDB とは

Amazon DynamoDB は、AWSが提供する**フルマネージドの NoSQL データベースサービス**です。キーバリュー型およびドキュメント型のデータモデルをサポートしています。

DynamoDB の最大の特徴は「サーバーレス」であることです。サーバーレスとは、利用者がサーバーの管理を一切気にしなくてよいという意味で、第2章の Lambda でも登場しました。DynamoDB では、サーバーの台数やスペックを気にする必要がありません。テーブルを作成してデータを入れるだけで使い始められます。容量もアクセス量に応じて自動でスケーリングされます。

レイテンシー（応答時間）は数ミリ秒と非常に高速です。どれだけデータ量が増えても、この応答速度が維持されるのが DynamoDB の強みです。

DynamoDB には2つの**キャパシティモード**（処理能力の設定方式）があります。**オンデマンドモード**は、アクセス量を事前に予測しなくても、実際のリクエスト数に応じて自動的に処理能力が調整されます。アクセスパターンが読めない場合や、トラフィックが急増する可能性がある場合に向いています。**プロビジョンドモード**は、あらかじめ秒間の読み書き回数を指定しておく方式です。アクセスパターンが安定しているワークロードでは、オンデマンドモードよりもコスト効率がよくなります。

### DynamoDB のデータモデル

DynamoDB のテーブルは、RDB のテーブルとは少し異なります。各アイテム（行に相当）は「パーティションキー」と呼ばれるキーで一意に識別されます。ロッカーの番号札のようなものです。

RDB とは異なり、テーブル内のアイテムごとに持っている属性（列に相当）が異なっていても構いません。あるアイテムには「住所」があるけれど、別のアイテムにはない、ということが許されます。この柔軟性が NoSQL の特徴です。

### DAX（DynamoDB Accelerator）

DAX は DynamoDB 専用の**インメモリキャッシュ**です。「インメモリ」とは、データをメモリ（高速な一時記憶装置）上に保持することで、ディスクから読み出すよりもはるかに速くデータを返せる仕組みです。

通常の DynamoDB は数ミリ秒の応答ですが、DAX を使うと**マイクロ秒**（ミリ秒のさらに1000分の1）レベルの応答が可能になります。リアルタイム入札やゲームのランキングなど、極端に速い応答が求められるユースケースに適しています。

### DynamoDB Global Tables

DynamoDB Global Tables は、複数のAWSリージョンにまたがってテーブルを自動でレプリケーションする機能です。たとえば東京リージョンと米国東部リージョンの両方に同じデータを持つことで、世界中のユーザーに低レイテンシーでサービスを提供できます。

### RDS/Aurora と DynamoDB の使い分け

この使い分けは試験で頻繁に問われるポイントです。

| 観点 | RDS / Aurora | DynamoDB |
|------|-------------|----------|
| データモデル | テーブル（行と列） | キーバリュー / ドキュメント |
| クエリ | SQL を使用 | APIベースの操作 |
| スキーマ | 固定（事前に定義が必要） | 柔軟（アイテムごとに異なる属性が可能） |
| スケーリング | 垂直（インスタンスサイズ変更） | 水平（自動スケーリング） |
| 向いている用途 | 複雑なクエリ、トランザクション処理 | 大量データの高速な読み書き |
| 具体例 | ECサイトの注文管理、会計システム | セッション管理、IoTデータ、ゲームのスコア |

試験では「どのようなデータに対してどのサービスを選ぶか」というシナリオ問題が出ます。「複雑な結合（JOIN）が必要」ならRDB系、「単純なキーによる高速な読み書き」ならDynamoDBと覚えておきましょう。

---

> **試験のポイント**
> - DynamoDB = サーバーレス + NoSQL + キーバリュー + 低レイテンシー
> - DAX = DynamoDB専用のインメモリキャッシュ（マイクロ秒レベルの応答）
> - Global Tables = 複数リージョンへの自動レプリケーション
> - キャパシティモード: オンデマンド = 予測不能なトラフィック向き、プロビジョンド = 安定ワークロードでコスト効率がよい
> - RDS/Aurora と DynamoDB の使い分けは頻出。「SQLが必要か」「スキーマが固定か柔軟か」がカギ

---

## 4-5. その他のデータベースサービス

### このセクションで学ぶこと

AWS にはRDS・Aurora・DynamoDB以外にも、特定の用途に特化したデータベースサービスが多数あります。各サービスの「何に使うのか」を1行で言えるレベルを目指しましょう。

### Amazon ElastiCache ― インメモリキャッシュ

ElastiCache は、頻繁にアクセスされるデータをメモリ上にキャッシュ（一時保存）することで、アプリケーションの応答速度を向上させるサービスです。

対応するキャッシュエンジンは **Redis** と **Memcached** の2つです。Redis はデータの永続化（保存）やレプリケーションが可能で高機能です。一方 Memcached はシンプルな分散キャッシュに特化しています。迷ったら Redis を選ぶケースが多いですが、試験ではこの2つの違いを問われることがあります。

身近な例で言えば、図書館で毎回書庫から本を取りに行くのではなく、よく借りられる人気の本をカウンター横の棚に並べておくイメージです。本（データ）を取りに行く時間が劇的に短縮されます。

ElastiCache はデータベースそのものを置き換えるものではありません。RDS や DynamoDB の手前に「キャッシュ層」として配置し、同じデータへの繰り返しアクセスを高速化する目的で使います。

### Amazon DocumentDB ― MongoDB互換

Amazon DocumentDB は **MongoDB互換** のフルマネージドドキュメントデータベースサービスです。すでにMongoDBを使っている環境からの移行先として設計されています。

JSONのような柔軟な形式でデータを保存できるため、コンテンツ管理システムやカタログ管理など、データ構造が変化しやすいアプリケーションに向いています。

### Amazon Neptune ― グラフDB

Amazon Neptune は、データ間の「つながり」を効率的に扱える**グラフデータベース**です。

SNS のフォロー関係（「AさんがBさんをフォローし、BさんがCさんをフォローしている」）や、商品のレコメンデーション（「この商品を買った人は、こちらの商品も買っています」）のように、データ同士の関係性が重要なユースケースに最適です。

試験では「ソーシャルグラフ」「レコメンデーション」「不正検知」というキーワードが出たら Neptune を思い出してください。

### Amazon Keyspaces ― Apache Cassandra互換

Amazon Keyspaces は、**Apache Cassandra互換** のマネージドデータベースサービスです。Cassandraは大量のデータを複数のサーバーに分散して保存するのに優れたオープンソースのNoSQLデータベースです。

すでにCassandraを使っている環境で、サーバーの運用管理をAWSに任せたい場合に選択します。Cassandra互換なので、既存のCassandraアプリケーションコードやツールをそのまま使えます。

### Amazon QLDB ― 台帳データベース

> **注意**: Amazon QLDB は2025年7月31日をもってサービスを終了しています。試験での出題可能性は低下していますが、台帳データベースの概念として理解しておくことは有益です。

Amazon QLDB（Quantum Ledger Database）は、データの変更履歴を**改ざん不可能な形で記録する**台帳データベースでした。なお、名前に「Quantum（量子）」とありますが、量子コンピューティングとは無関係です。

「台帳」とは、取引の記録帳のことです。銀行の取引履歴や、サプライチェーンにおける商品の移動記録など、「誰が・いつ・何を変更したか」を正確に追跡する必要があるユースケースに適していました。

台帳のような改ざん不可能な記録が必要な場合は、現在は **Amazon Managed Blockchain** などの代替サービスを検討することになります。

### Amazon Timestream ― 時系列データベース

Amazon Timestream は、**時系列データ**（タイムスタンプ付きのデータ）に特化したデータベースです。

IoTセンサーが1秒ごとに温度を送信するデータや、アプリケーションのメトリクス（CPU使用率、メモリ使用量など）のように、「時間の経過とともに蓄積されるデータ」を効率的に保存・クエリするために設計されています。

### 各サービスの用途まとめ

| サービス | 用途を一言で |
|---------|-------------|
| ElastiCache | キャッシュ層（Redis / Memcached） |
| DocumentDB | MongoDB互換のドキュメントDB |
| Neptune | ソーシャルグラフ・レコメンデーション |
| Keyspaces | Apache Cassandra互換のマネージドDB |
| QLDB | 改ざん不可能な変更履歴の記録（※サービス終了済み） |
| Timestream | IoTやメトリクスなどの時系列データ |

---

> **試験のポイント**
> - ElastiCache = インメモリキャッシュ（Redis / Memcached）。データベースの前段に置いて応答速度を上げる
> - Neptune = グラフDB。「ソーシャルグラフ」「レコメンデーション」が出たらこれ
> - DocumentDB = MongoDB互換。「MongoDB」がキーワード
> - Keyspaces = Cassandra互換。「Cassandra」がキーワード
> - QLDB = 改ざん不可能な台帳。「変更履歴の追跡」「監査」がキーワード（※2025年7月にサービス終了済み）
> - Timestream = 時系列データ。「IoTセンサー」「メトリクス」がキーワード

## 4-6. データベース移行

### このセクションで学ぶこと

既存のデータベースをAWSに移行するためのサービス「AWS DMS」と「AWS SCT」の役割を理解します。

### なぜデータベース移行が必要なのか

多くの企業は、すでにオンプレミス（自社のデータセンター）やほかのクラウドでデータベースを運用しています。AWSに移行したいと思っても、「移行中にシステムを止めたくない」「データを1件も失いたくない」という要件があります。そこで活躍するのが AWS DMS です。

### AWS DMS（Database Migration Service）

AWS DMS は、**既存のデータベースをAWSに移行するためのサービス**です。最大の特徴は、移行元のデータベースを**稼働させたまま**移行できることです。

引っ越しに例えるなら、「住みながらの引っ越し」を可能にするサービスです。住んでいる家（移行元DB）で生活を続けながら、新しい家（AWS上のDB）に荷物（データ）を運びます。

DMS を使った移行の基本的な流れは次のとおりです。まず移行元データベース（ソース）と移行先データベース（ターゲット）を設定します。次にDMSがデータをコピーします。初回の一括コピーが終わったら、その後の変更分も継続的にレプリケーションしてくれます（これを「継続的データレプリケーション」または「CDC: Change Data Capture」と呼びます）。

DMS は「同種間の移行」（例: オンプレミスのMySQL → RDS MySQL）にも、「異種間の移行」（例: Oracle → PostgreSQL）にも対応しています。

![DMS移行パターンのフロー](../images/ch04-dms-migration.drawio.png)

### AWS SCT（Schema Conversion Tool）

AWS SCT は、**異なるデータベースエンジン間でスキーマ（テーブル構造やストアドプロシージャ（データベースに保存された処理手順のプログラム））を変換するツール**です。

たとえば Oracle から PostgreSQL に移行する場合、テーブルの定義方法や関数の書き方が異なります。SCT は、移行元のスキーマを分析して、移行先のエンジンに合った形に自動変換してくれます。

DMS と SCT の関係を整理しましょう。DMS は「データを運ぶトラック」、SCT は「荷物の梱包を手伝うスタッフ」のような存在です。同じ種類のデータベース間の移行（MySQL → MySQL など）なら、スキーマの変換は不要なので DMS だけで済みます。異なるエンジン間の移行（Oracle → PostgreSQL など）では、まず SCT でスキーマを変換し、次に DMS でデータを移行するという流れになります。

| 移行パターン | 使うサービス | 例 |
|-------------|-------------|-----|
| 同種間（同じエンジン） | DMS のみ | Oracle → RDS Oracle |
| 異種間（異なるエンジン） | SCT + DMS | Oracle → RDS PostgreSQL |

---

> **試験のポイント**
> - DMS = データベースの移行サービス。稼働中のDBを止めずに移行できる
> - SCT = 異なるDBエンジン間のスキーマ変換ツール
> - 同種間 → DMS だけ、異種間 → SCT + DMS の組み合わせ
> - DMS は移行後も継続的なデータレプリケーション（CDC）が可能
> - DMS/SCT の詳細な移行戦略については、第14章（移行と最新テクノロジー）で改めて扱います

## 章末確認問題

### 問題1

自社のオンプレミス環境にあるデータベースの運用負荷を減らすためにAWSへの移行を検討しています。マネージドリレーショナルデータベースサービスとして適切なものはどれですか？

- **A.** Amazon S3
- **B.** Amazon RDS
- **C.** Amazon DynamoDB
- **D.** Amazon ElastiCache

<details>
<summary>解答・解説</summary>

**解答: B**

解説: Amazon RDS はマネージドリレーショナルデータベースサービスです。S3はオブジェクトストレージ、DynamoDB は NoSQL データベース、ElastiCache はインメモリキャッシュであり、いずれもリレーショナルデータベースではありません。

</details>

---

### 問題2

Amazon RDS でデータベースの**高可用性**を実現したい場合、どの機能を使用しますか？

- **A.** リードレプリカ
- **B.** マルチAZデプロイ
- **C.** Aurora Serverless
- **D.** DynamoDB Global Tables

<details>
<summary>解答・解説</summary>

**解答: B**

解説: マルチAZデプロイは、プライマリDBとスタンバイDBを異なるAZに配置し、障害時に自動フェイルオーバーすることで高可用性を実現します。リードレプリカは読み取りパフォーマンスの向上が目的であり、高可用性のための機能ではありません。

</details>

---

### 問題3

Amazon RDS のリードレプリカについて、正しい説明はどれですか？

- **A.** プライマリDBとの間で同期レプリケーションが行われる
- **B.** プライマリDBが停止すると自動的にフェイルオーバーする
- **C.** 読み取り処理をリードレプリカに分散してパフォーマンスを向上させる
- **D.** 書き込み処理もリードレプリカで実行できる

<details>
<summary>解答・解説</summary>

**解答: C**

解説: リードレプリカは読み取り専用のコピーで、読み取り処理を分散してパフォーマンスを向上させます。プライマリとの間は非同期レプリケーションであり、自動フェイルオーバーの仕組みはありません（それはマルチAZの機能です）。書き込みはプライマリDBでのみ行われます。

</details>

### 問題4

Amazon Aurora の特徴として正しいものはどれですか？（2つ選択）

- **A.** Oracle Database と互換性がある
- **B.** MySQL互換で標準MySQLの最大5倍のスループットを実現する
- **C.** PostgreSQL互換で標準PostgreSQLの最大5倍のスループットを実現する
- **D.** AWS独自に設計されたデータベースエンジンである
- **E.** NoSQLデータベースである

<details>
<summary>解答・解説</summary>

**解答: B, D**

解説: Aurora は AWS独自設計のリレーショナルデータベースで、MySQL互換では最大5倍、PostgreSQL互換では最大3倍のスループットを実現します。Oracle互換ではなく、NoSQLでもありません。CはPostgreSQL互換の倍率が「5倍」となっているため誤りです（正しくは最大3倍）。

</details>

### 問題5

数百万人のユーザーが利用するモバイルゲームのバックエンドで、プレイヤーのセッションデータを低レイテンシーで読み書きする必要があります。最も適切なデータベースサービスはどれですか？

- **A.** Amazon RDS
- **B.** Amazon Aurora
- **C.** Amazon DynamoDB
- **D.** Amazon Neptune

<details>
<summary>解答・解説</summary>

**解答: C**

解説: Amazon DynamoDB はサーバーレスの NoSQL データベースで、ミリ秒レベルの低レイテンシーと自動スケーリングを提供します。セッションデータのような単純なキーバリュー型のデータを大量に高速処理するユースケースに最適です。RDS や Aurora は複雑なクエリが必要な場合に適していますが、この要件ではDynamoDBが最適です。

</details>

### 問題6

SNS（ソーシャルネットワーク）サービスで、ユーザー間のフォロー関係を管理し、「友達の友達」をレコメンドする機能を実装したい場合、最適なAWSデータベースサービスはどれですか？

</details>

### 問題7

アプリケーションの応答速度を向上させるため、頻繁にアクセスされるデータをメモリ上にキャッシュしたいと考えています。どのサービスを使うべきですか？

**解答: B**

</details>

### 問題8

オンプレミスの Oracle データベースを Amazon RDS for PostgreSQL に移行する場合、必要なAWSサービスの組み合わせとして正しいものはどれですか？

- **A.** AWS DMS のみ
- **B.** AWS SCT のみ
- **C.** AWS SCT + AWS DMS
- **D.** Amazon S3 + AWS DMS

<details>
<summary>解答・解説</summary>

**解答: C**

解説: Oracle から PostgreSQL への移行は異種間の移行（異なるデータベースエンジン間）です。まず AWS SCT でスキーマを変換し、その後 AWS DMS でデータを移行します。同種間の移行であれば DMS のみで済みますが、異種間では SCT との組み合わせが必要です。

</details>

### 問題9

DynamoDB のレスポンス時間をさらに短縮し、マイクロ秒レベルの応答を実現したい場合、どのサービスを組み合わせますか？

- **A.** Amazon ElastiCache

- **D.** Amazon Aurora

</details>

### 問題10

AWS DMS（Database Migration Service）の特徴として正しいものはどれですか？

- **A.** 移行中は移行元データベースを停止する必要がある
- **B.** 移行元のデータベースを稼働させたまま移行できる
- **C.** 同じデータベースエンジン間の移行のみサポートしている
- **D.** データベースのスキーマ変換機能を内蔵している

<details>
<summary>解答・解説</summary>

**解答: B**

解説: AWS DMS の最大の特徴は、移行元のデータベースを稼働させたまま（ダウンタイムを最小限にして）移行できることです。同種間・異種間の両方の移行に対応しています。スキーマ変換は DMS ではなく AWS SCT（Schema Conversion Tool）の役割です。

</details>

## A-1. ネットワーク通信の基本

### このセクションで学ぶこと

コンピュータ同士がどのようにデータをやりとりするか、ネットワークの基本的な概念を理解します。

### ネットワークとは

**ネットワーク**とは、複数のコンピュータを繋いでデータをやりとりできるようにした仕組みです。

身近な例で考えてみましょう。会社のオフィスでは、プリンターを複数の人が共有して使えます。それはコンピュータとプリンターがネットワークで繋がっているからです。

### LAN と WAN

ネットワークには、規模によって2種類あります。

- **LAN（Local Area Network）**: 限られた範囲（家庭・オフィス・ビル内）のネットワーク
- **WAN（Wide Area Network）**: 広範囲をカバーするネットワーク。インターネット自体も WAN の一種です

### クライアント / サーバーモデル

Webサイトを見るときの仕組みを考えてみましょう。

1. あなたのパソコン（**クライアント**）が「このページを見せて」とリクエストを送る
2. Webサーバー（**サーバー**）がリクエストを受け取り、ページのデータを返す
3. ブラウザがデータを表示する

このように、「サービスを要求する側（クライアント）」と「サービスを提供する側（サーバー）」に分かれた仕組みを**クライアント / サーバーモデル**と言います。

## A-2. インターネットの仕組み

### このセクションで学ぶこと

インターネットがどのように成り立っているかを理解します。

### インターネットとは

インターネットは「**ネットワークのネットワーク**」です。世界中の無数のネットワーク（企業・学校・家庭のLANなど）がお互いに繋がって構成されています。

### ISP（インターネットサービスプロバイダー）

あなたが自宅でインターネットを使えるのは、**ISP（インターネットサービスプロバイダー）**と契約しているからです。NTT、KDDI、SoftBank などが代表的なISPです。

ISPは、バックボーン（インターネットの中核となる高速通信回線網）に接続し、あなたの家庭のネットワークをインターネットに繋ぎます。

### パケット通信

インターネットではデータを**パケット**と呼ばれる小さな塊に分割して送ります。

宅配便で大きな荷物を複数の小箱に分けて送るイメージです。各パケットはバラバラの経路を通って宛先に届き、受け取った側で元のデータに組み立てます。

## A-3. IPアドレスとサブネット

### このセクションで学ぶこと

ネットワーク上の「住所」にあたるIPアドレスの仕組みと、CIDRブロックの読み方を理解します。

### IPアドレスとは

**IPアドレス**とは、ネットワーク上の各コンピュータを識別するための「住所」です。

`192.168.1.10` のように、0〜255の数字4つをドットで区切った形式で表します（**IPv4**）。

世界中でユニークな「**パブリックIPアドレス**」と、社内ネットワーク内だけで使える「**プライベートIPアドレス**」があります。

| 種類 | 用途 | 例 |
|------|------|-----|
| パブリックIP | インターネット上で一意の住所 | 203.0.113.1 |
| プライベートIP | LAN内の住所（外からは見えない） | 192.168.1.10、10.0.0.1 |

プライベートIPアドレスに使われる主な範囲（RFC1918）:
- `10.0.0.0` 〜 `10.255.255.255`
- `172.16.0.0` 〜 `172.31.255.255`
- `192.168.0.0` 〜 `192.168.255.255`

### CIDR（サイダー）ブロック

`10.0.0.0/16` のような表記を**CIDR（Classless Inter-Domain Routing）**と言います。

スラッシュの後の数字（プレフィックス長）は、「アドレスの何ビットが固定か」を示します。

| CIDR | ネットワーク部 | 利用可能なIPアドレス数の目安 |
|------|----------------|---------------------------|
| /8 | 上位8ビット固定 | 約1,600万個 |
| /16 | 上位16ビット固定 | 約65,000個 |
| /24 | 上位24ビット固定 | 約250個 |
| /32 | すべて固定 | 1個（特定のIPアドレス） |

プレフィックス長の数字が**小さいほどアドレス数が多い**（広いネットワーク）、**大きいほど少ない**（狭いネットワーク）と覚えましょう。

- **B.** LAN内だけで使われるアドレスで、外からは直接見えない
- **C.** IPv6で使われるアドレス形式

<details>
<summary>解答・解説</summary>

**解答: B

プライベートIPアドレスはLAN内のみで使われ、インターネットからは直接到達できません。

</details>

**Q2.** `10.0.0.0/24` のCIDRブロックについて、正しい説明はどれですか？
- **A.** 約65,000個のIPアドレスを含む広いネットワーク
- **B.** 約250個のIPアドレスを含む比較的小さなネットワーク

<details>

</details>

---

> Amazon VPCを作成するとき、VPCのIPアドレス範囲を `10.0.0.0/16` のようなCIDRブロックで指定します。また、VPC内のサブネット（部屋の区切り）も `10.0.1.0/24` のようなCIDRブロックで定義します。第5章で詳しく学びます。

## A-4. TCP/IPの基本概念

### このセクションで学ぶこと

インターネット通信の基盤となるTCP/IPプロトコルスタックの概念を理解します。

### プロトコルとは

**プロトコル**とは、通信のルール（取り決め）です。

手紙を書くときの「宛名は右上、差出人は左下」というような慣習と同じです。通信する双方が同じルールに従うことで、正しくデータをやりとりできます。

### TCP と UDP

データを送る際のプロトコルとして、代表的なものが **TCP** と **UDP** です。

**TCP（Transmission Control Protocol）**
- 「確実に届ける」ことを保証する
- 届いたかどうかを確認しながら送る（確認応答あり）
- 速度よりも確実性が重要な場面で使う
- 例: Webブラウジング（HTTP/HTTPS）、メール、ファイル転送

**UDP（User Datagram Protocol）**
- 確認なしに送りつける（確認応答なし）
- 速度重視、多少のデータ欠損は許容される
- 例: 動画ストリーミング、オンラインゲーム、音声通話

### ポート番号

同じコンピュータで複数のサービスを動かすために、**ポート番号**で区別します。

マンションの「部屋番号」のようなものです。建物の住所（IPアドレス）は同じでも、部屋番号（ポート番号）で誰宛かを区別します。

代表的なポート番号:

| ポート番号 | プロトコル・サービス |
|-----------|-------------------|
| 22 | SSH（サーバーへの安全なリモート接続） |
| 80 | HTTP（通常のWebアクセス） |
| 443 | HTTPS（暗号化されたWebアクセス） |
| 3306 | MySQL（データベース） |
| 5432 | PostgreSQL（データベース） |

### TCP/IP 4層モデル

ネットワーク通信は4つの階層（レイヤー）に分けて考えます。

| 層 | 名前 | 役割 | 例 |
|----|------|------|-----|
| 第4層 | アプリケーション層 | アプリの通信規則 | HTTP、HTTPS、DNS |
| 第3層 | トランスポート層 | データの分割・確認 | TCP、UDP |
| 第2層 | インターネット層 | アドレス指定・ルーティング | IP |
| 第1層 | ネットワークインターフェース層 | 物理的な通信 | Ethernet、Wi-Fi |

**補足: OSI参照モデル（7層）との対応**

AWSの試験やドキュメントでは、TCP/IP 4層モデルではなく**OSI参照モデル（7層）**のレイヤー番号が使われることがあります。特に「ALBはレイヤー7」「NLBはレイヤー4」という表現は頻出です。TCP/IP 4層モデルのアプリケーション層はOSI 7層モデルのレイヤー5〜7に対応します。

| OSI 7層モデル | TCP/IP 4層モデル | AWSでの用例 |
|--------------|-----------------|------------|
| レイヤー7（アプリケーション層） | アプリケーション層 | ALB（Application Load Balancer） |
| レイヤー4（トランスポート層） | トランスポート層 | NLB（Network Load Balancer） |

> **図解:** appA-tcp-ip-layers.drawio.png — TCP/IP 4層モデルの各層と役割
> （draw.io ファイルは `docs/images/appA-tcp-ip-layers.drawio.png` に配置予定）

## A-5. DNS

### このセクションで学ぶこと

ドメイン名（example.com）をIPアドレスに変換する仕組み（DNS）を理解します。

### DNSとは

**DNS（Domain Name System）**とは、ドメイン名をIPアドレスに変換する仕組みです。

電話帳と同じです。「田中さん」という名前で調べると電話番号（数字）が分かるように、「google.com」というドメイン名を調べると「142.250.196.14」のようなIPアドレスが分かります。

### 名前解決の流れ

ブラウザに `https://www.example.com` と入力すると、以下の流れで名前解決が行われます。

1. ブラウザが**DNSリゾルバー**（プロバイダーのDNSサーバー）に問い合わせる
2. DNSリゾルバーが**ルートサーバー**に問い合わせる（「.com を管理しているのは誰か？」）
3. **TLDサーバー**（`.com` を管理）に問い合わせる（「example.com を管理しているのは誰か？」）
4. **権威サーバー**（`example.com` の実際のIPを知っているサーバー）から回答を得る
5. IPアドレスをブラウザに返す

> **図解:** appA-dns-resolution.drawio.png — DNS名前解決のフロー（ブラウザ→DNSリゾルバー→ルート→TLD→権威サーバー）
> （draw.io ファイルは `docs/images/appA-dns-resolution.drawio.png` に配置予定）

### 主なDNSレコードの種類

| レコード種別 | 役割 | 例 |
|------------|------|-----|
| **A レコード** | ドメイン名 → IPv4アドレスへのマッピング | example.com → 203.0.113.1 |
| **CNAME レコード** | ドメイン名 → 別のドメイン名へのエイリアス | www.example.com → example.com |
| **MX レコード** | メールサーバーの指定 | — |
| **TXT レコード** | 任意のテキスト情報 | ドメイン所有確認など |

## A-6. HTTP/HTTPS

### このセクションで学ぶこと

WebブラウザとWebサーバーが通信するプロトコル（HTTP/HTTPS）の仕組みを理解します。

### HTTP とは

**HTTP（HyperText Transfer Protocol）**は、WebブラウザとWebサーバーがデータをやりとりするためのプロトコルです。

### HTTPメソッド

クライアントがサーバーに「何をしてほしいか」を指定するための命令をHTTPメソッドと言います。

| メソッド | 意味 | 例 |
|---------|------|-----|
| **GET** | データを取得する | Webページを表示する |
| **POST** | データを送信する | フォームを送信する |
| **PUT** | データを更新する | プロフィールを更新する |
| **DELETE** | データを削除する | 記事を削除する |

### HTTPステータスコード

サーバーがリクエストに対して返す「結果コード」です。

| コード | 意味 | 例 |
|--------|------|-----|
| **2xx** | 成功 | 200 OK（正常応答） |
| **3xx** | リダイレクト | 301 Moved Permanently（恒久的な移転） |
| **4xx** | クライアントエラー | 404 Not Found（ページが見つからない）、403 Forbidden（アクセス禁止） |
| **5xx** | サーバーエラー | 500 Internal Server Error（サーバー内部エラー） |

### HTTPS と TLS/SSL

**HTTPS（HTTP Secure）**は、HTTP通信を**TLS（Transport Layer Security）**で暗号化したものです。

通信内容を暗号化することで、第三者による盗聴や改ざんを防ぎます。URLが `https://` で始まり、ブラウザに鍵マークが表示されていればHTTPS通信です。

TLS は以前 **SSL（Secure Sockets Layer）**と呼ばれていました。現在は TLS が標準ですが、「SSL証明書」という呼び方は今でも一般的に使われています。

---

## A-7. ファイアウォールとネットワークセキュリティの基礎

### このセクションで学ぶこと

ファイアウォールの概念と、インバウンド/アウトバウンドルール、ステートフル/ステートレスの違いを理解します。

### ファイアウォールとは

**ファイアウォール**とは、ネットワークへの通信を監視し、許可された通信だけを通す「門番」です。

会社のビルの入り口に警備員がいて、「通行証を持っている人だけ入れる」ようなイメージです。

### インバウンドとアウトバウンド

通信の方向を指す重要な用語です。

- **インバウンド（Inbound）**: 外からサーバーへの着信通信
- **アウトバウンド（Outbound）**: サーバーから外への発信通信

Webサーバーの例:
- インバウンド: ユーザーのブラウザ → Webサーバー（ポート80/443）→ **許可が必要**
- アウトバウンド: Webサーバー → インターネット（応答の返送など）→ 設定次第

### ステートフルとステートレス

ファイアウォールの動作方式には2種類あります。これは試験で頻出のひっかけポイントです。

**ステートフル（状態を記憶する）**

「インバウンドを許可した通信の応答は、自動的にアウトバウンドも許可する」という動作をします。

例: ポート80へのインバウンドを許可すれば、その応答のアウトバウンドも自動的に許可されます。アウトバウンドルールを別途設定する必要はありません。

**ステートレス（状態を記憶しない）**

各通信を独立して評価します。インバウンドを許可しても、応答のアウトバウンドは別途明示的に許可する必要があります。

| | ステートフル | ステートレス |
|-|-------------|-------------|
| 特徴 | 通信の状態を記憶する | 各通信を独立して評価 |
| 応答通信の扱い | 自動で許可 | 明示的に許可が必要 |
| ルール設定 | 比較的シンプル | 詳細な設定が必要 |

### 理解度チェック

**Q1.** ファイアウォールの役割として正しいものはどれですか？
- **A.** データを暗号化して安全に送信する
- **B.** ネットワーク通信を監視し、許可された通信だけを通す
- **C.** Webサーバーの負荷を分散させる

<details>
<summary>解答・解説</summary>

**解答: B

ファイアウォールは「許可された通信だけを通す門番」の役割をします。

</details>

**Q2.** ステートフルなファイアウォールの説明として正しいものはどれですか？
- **A.** インバウンドを許可しても、応答のアウトバウンドは別途設定が必要
- **B.** インバウンドを許可すると、応答のアウトバウンドも自動的に許可される
- **C.** すべての通信をデフォルトで許可する

<details>
<summary>解答・解説</summary>

**解答: B

ステートフルは「通信の状態を記憶」するため、許可したインバウンドへの応答は自動的に許可されます。

</details>

**Q3.** インバウンドとアウトバウンドの説明として正しいものはどれですか？
- **A.** インバウンドはサーバーからの送信、アウトバウンドはサーバーへの受信
- **B.** インバウンドはサーバーへの着信、アウトバウンドはサーバーからの発信
- **C.** インバウンドとアウトバウンドは同じ意味

<details>
<summary>解答・解説</summary>

**解答: B

インバウンドは「外からサーバーへの着信」、アウトバウンドは「サーバーから外への発信」です。

</details>

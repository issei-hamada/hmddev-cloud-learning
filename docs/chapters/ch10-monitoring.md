## 10-1. なぜ監視が必要か

### このセクションで学ぶこと

監視の目的と、「監視」と「可観測性（Observability）」の違いを理解します。

### 監視がなければ「気づけない」

たとえば、あなたが飲食店のオーナーだとしましょう。お店の売上が急に落ちたとき、原因がわからなければ対処しようがありません。レジの売上記録、来客数、食材の在庫状況などを日々チェックしていれば、「今週は平日の来客が減っている」「食材のロスが増えている」と気づくことができます。

ITシステムも同じです。サーバーのCPU使用率が異常に高い、ディスクの空き容量がなくなりかけている、エラーが急増している――こうした異変に気づけなければ、障害が起きてから慌てることになります。監視とは、システムの状態を継続的にチェックし、異常を早期に検知するための仕組みです。

### オンプレミスでの監視の課題

自社でサーバーを運用するオンプレミス環境では、監視にもさまざまな手間がかかります。まず、ZabbixやNagios、Prometheusなどの監視ツールの中から何を使うかを選定しなければなりません。ツールを動かすための「監視サーバー」自体も用意して運用する必要があります。監視サーバーが壊れたら、誰も異常に気づけなくなるという皮肉な状況も起こりえます。

さらに、サーバーが増えるたびに監視の設定を追加し、ツールのバージョンアップやセキュリティパッチの適用も行わなければなりません。

### クラウドにおける可観測性（Observability）

クラウドの世界では、「監視（Monitoring）」をさらに発展させた「可観測性（Observability＝オブザーバビリティ）」という考え方が重要になっています。

従来の監視は「あらかじめ想定した項目をチェックして、異常があれば通知する」というアプローチです。たとえば「CPU使用率が90%を超えたらアラートを出す」というルールを事前に設定しておきます。これは「何を見るか」をあらかじめ知っている前提で成り立ちます。

一方、可観測性は「システムの内部状態を、外部から出力されるデータだけで理解できるようにする」という考え方です。想定していなかった問題が起きたときにも、メトリクス（数値データ）、ログ（テキストデータ）、トレース（処理の流れの記録）を組み合わせて原因を突き止められる状態を目指します。

トレースとは、ユーザーのリクエストがシステム内のどのサービスをどの順番で通ったかを記録したものです。たとえばWebサイトで注文ボタンを押したとき、注文処理→在庫確認→決済→確認メール送信という一連の流れが記録されます。どこで時間がかかったのか、どこでエラーが起きたのかを追跡できるので、複数のサービスが連携するシステムでは特に重要です。

飲食店の例えでいえば、監視は「毎日の売上だけを見る」こと。可観測性は「売上・来客数・メニュー別注文数・食材の使用量・スタッフのシフト」など多角的なデータを収集しておいて、何か問題が起きたときに「なぜそうなったのか」を事後的に分析できる状態です。

AWSでは、この可観測性を実現するためのサービスが豊富に用意されています。次のセクションから、それぞれのサービスを見ていきましょう。

---

> **試験のポイント**
> - 監視 = 「あらかじめ設定した基準で異常を検知する」
> - 可観測性（Observability）= 「システムの内部状態を外部から理解できる」、より広い概念
> - 可観測性の3つの柱: メトリクス（数値データ）、ログ（テキストデータ）、トレース（処理の流れ）

## 10-2. Amazon CloudWatch

### このセクションで学ぶこと

AWSリソースの監視サービスである CloudWatch の4つの主要機能（メトリクス・ログ・アラーム・ダッシュボード）を理解します。

### CloudWatch とは

Amazon CloudWatch は、AWSリソースやアプリケーションの状態をリアルタイムで監視するためのサービスです。一言でいえば「AWSの健康診断モニター」のような存在です。

病院の入院患者に取り付けられた心電図モニターを想像してください。心拍数や血圧が常に画面に表示され、異常な数値になるとアラームが鳴ります。CloudWatch は、EC2インスタンスやRDSデータベースなどのAWSリソースに対して、まさに同じことをやってくれます。

### メトリクス ― 数値で状態を把握する

メトリクスとは、リソースの状態を表す数値データのことです。EC2であればCPU使用率やネットワーク通信量、RDSであればデータベースの接続数やストレージの空き容量などが、自動的にメトリクスとして収集されます。

AWSの多くのサービスは、特に設定をしなくても標準メトリクスを CloudWatch に送信してくれます。たとえばEC2インスタンスを起動するだけで、CPU使用率やディスクの読み書き量などが自動的に記録されます。なお、EC2の標準メトリクスにはメモリ使用率やディスク使用率は含まれません。これらを監視するには、CloudWatch Agentをインスタンスにインストールしてカスタムメトリクスとして送信する必要があります。

一方、AWSが自動で収集しない独自のデータを監視したい場合は、カスタムメトリクスを使います。たとえば「アプリケーションにログインしているユーザー数」や「注文処理のキューに溜まっている件数」など、ビジネスに固有のデータを自分で CloudWatch に送信して監視できます。送信にはアプリケーションのコードやAWS CLI、CloudWatch Agentなどを使います。

### ログ ― テキストで状況を記録する

CloudWatch Logs は、アプリケーションやAWSサービスが出力するテキスト形式のログを一元的に収集・保管するサービスです。

ログとは、システムが「いつ・何が起きたか」を時系列で記録したテキストデータです。たとえば「2024年3月15日 14:30:22 ユーザーAがログインに失敗しました」といった記録がログです。

サーバーが複数台ある場合、それぞれのサーバーにログが散らばっていると、問題の調査が大変です。CloudWatch Logs に集約しておけば、一つの画面からまとめて検索・分析できます。さらに、CloudWatch Logs Insights を使えば、収集したログに対してクエリ言語（検索用の専用構文）で絞り込みや集計を行い、問題の原因を効率的に調査できます。

### アラーム ― 異常を通知する

CloudWatch アラーム は、メトリクスの値があらかじめ設定したしきい値を超えたときに通知を送る仕組みです。

たとえば「EC2インスタンスのCPU使用率が80%を5分間超え続けたら、担当者にメールを送る」というアラームを設定できます。これにより、サーバーに負荷がかかっていることにいち早く気づけます。

アラームの通知先には Amazon SNS（第6章で学んだ通知サービス）を使います。メール、SMS、チャットツールへの通知など、柔軟に設定できます。さらに、アラームをトリガーにしてAuto Scalingのスケーリングアクションを実行する、といった自動化も可能です。

### ダッシュボード ― ひと目で全体を把握する

CloudWatch ダッシュボード は、複数のメトリクスやアラームの状態を一つの画面にまとめて表示する機能です。テレビのニュースで天気や株価がまとめて表示されるように、システム全体の状態をひと目で確認できます。

### Amazon EventBridge（旧 CloudWatch Events）

Amazon EventBridge は、AWSサービスやアプリケーションで発生した「イベント（出来事）」を検知し、あらかじめ設定したアクションを自動的に実行するサービスです。もともとは CloudWatch Events という名前でしたが、機能が拡張されて EventBridge に改名されました。

たとえば「EC2インスタンスの状態が"停止"に変わったらLambda関数を実行して管理者に通知する」「毎日午前3時にバッチ処理を起動する」といったイベント駆動の自動化が可能です。CloudWatch アラームが「数値のしきい値超え」に反応するのに対し、EventBridge は「何かが起きた」という出来事そのものに反応するのが特徴です。

---

> **試験のポイント**
> - CloudWatch = AWSリソースのパフォーマンス監視（「いまどうなっているか」を見る）
> - メトリクス（数値）、ログ（テキスト）、アラーム（通知）、ダッシュボード（一覧表示）の4機能
> - 標準メトリクスは自動収集、独自のデータにはカスタムメトリクスを使う（EC2のメモリ使用率は標準メトリクスに含まれない）
> - アラームは SNS と連携して通知を送る
> - EventBridge（旧CloudWatch Events）= イベント（出来事）を検知して自動アクションを実行
> - CloudWatch は「リソースの監視」、後述の CloudTrail は「操作の監査」 ― この違いは頻出

---

## 10-3. AWS CloudTrail

### このセクションで学ぶこと

「誰が・いつ・何をしたか」を記録する監査ログサービス CloudTrail の仕組みを理解します。CloudWatch との違いを明確に区別できるようになることが重要です。

### CloudTrail とは

AWS CloudTrail は、AWSアカウント内で行われたすべてのAPI呼び出し（操作）を記録するサービスです。わかりやすくいえば「AWSの防犯カメラ」です。

オフィスビルの防犯カメラを想像してください。誰がいつ入館し、どの部屋に入り、何をしたかが映像として記録されます。CloudTrail も同じように、「誰が、いつ、どのリソースに対して、何の操作をしたか」をすべて記録します。

たとえば、「管理者Aが3月15日14時にEC2インスタンスを停止した」「ユーザーBが3月16日10時にS3バケットを作成した」といった操作がすべてログとして残ります。

### 管理イベントとデータイベント

CloudTrail が記録するイベント（操作の記録）には、大きく分けて2種類あります。

管理イベントは、AWSリソースの作成・削除・設定変更など、リソースの管理に関する操作の記録です。EC2インスタンスの起動・停止、IAMユーザーの作成、セキュリティグループのルール変更などが該当します。管理イベントはデフォルトで記録されます。

データイベントは、リソース上で行われるデータ操作の記録です。S3バケットへのファイルのアップロード・ダウンロード、Lambda関数の実行などが該当します。データイベントは大量に発生するため、デフォルトでは記録されません。必要に応じて有効化します。

### CloudTrail Lake

CloudTrail のイベント履歴（マネジメントコンソールで直接閲覧できるログ）は、過去90日分のみ保持されます。90日以上保存するには、証跡（Trail）を作成してS3バケットに配信するか、CloudTrail Lake を利用します。S3への配信は長期保存の一般的な方法で、保存期間に制限はありません。

CloudTrail Lake は、CloudTrail のログをSQLのようなクエリ言語で検索・分析できるサービスです。最大7年間（2555日）保存でき、複雑な条件での検索も行えます。S3への配信よりも高度な分析が必要な場合に適しています。

### CloudWatch と CloudTrail の違い ― 最重要ポイント

この2つのサービスの違いは、試験でひっかけ問題として頻出します。しっかり区別しましょう。

CloudWatch は「リソースの状態を監視する」サービスです。「サーバーのCPU使用率はいくつか」「エラーログはどれくらい出ているか」など、リソースの健康状態を数値やログで見守ります。

CloudTrail は「操作を記録する」サービスです。「誰がサーバーを停止したか」「誰がファイルを削除したか」など、人やプログラムの行動を記録します。

病院にたとえると、CloudWatch は「心電図モニター（患者の状態を監視する）」、CloudTrail は「防犯カメラ（誰が病室に出入りしたかを記録する）」です。患者の状態が悪化したことは心電図でわかりますが、誰が点滴を変えたかは防犯カメラでわかります。

---

> **試験のポイント**
> - CloudTrail = 「誰が・いつ・何をしたか」の監査ログ（API呼び出しの記録）
> - 管理イベント（リソースの管理操作）はデフォルトで記録される
> - データイベント（S3オブジェクトへのアクセスなど）はデフォルトでは記録されない
> - CloudWatch = リソースの「状態」を監視 / CloudTrail = 「操作」を記録 ― この違いは頻出
> - イベント履歴は90日間のみ。長期保存にはS3への配信（証跡の作成）または CloudTrail Lake を利用

---

## 10-4. AWS Config

### このセクションで学ぶこと

リソースの設定が「正しい状態」になっているかを継続的にチェックする AWS Config の仕組みを理解します。

### AWS Config とは

AWS Config は、AWSリソースの構成（設定内容）の変更を追跡し、あるべき状態と比較して問題がないかをチェックするサービスです。

レストランの衛生検査官をイメージしてください。定期的にキッチンを巡回し、「冷蔵庫の温度は適切か」「手洗い設備は使える状態か」「食材の消費期限は守られているか」をチェックします。基準を満たしていなければ改善勧告を出します。AWS Config も同じように、AWSリソースの設定が「あるべき姿」に合っているかを継続的にチェックしてくれます。

### 構成変更の追跡

AWS Config は、リソースの設定がいつ・どのように変更されたかの履歴を記録します。たとえば、あるセキュリティグループのインバウンドルールが変更されたとき、「変更前はどうだったか」「いつ変更されたか」が時系列で記録されます。

これにより、「昨日までは問題なく動いていたのに今日から動かなくなった。何が変わったのか？」というトラブルシューティングの場面で、原因を素早く特定できます。

### コンプライアンスルール

AWS Config の大きな特徴は、コンプライアンスルール（Config Rules）の仕組みです。「リソースの設定がこの条件を満たしていなければ非準拠とする」というルールを定義できます。

代表的な例をいくつか挙げましょう。「S3バケットがパブリックアクセス（誰でもアクセスできる状態）を許可していないか」「EC2インスタンスに適切なタグが付いているか」「EBSボリュームが暗号化されているか」――こういったルールを設定しておけば、ルールに違反しているリソースが自動的に検出されます。

### 構成ドリフトの検知

「構成ドリフト」とは、あるべき状態からリソースの設定がずれてしまうことです。最初はセキュリティの基準を満たしていたリソースが、誰かの操作で基準を満たさない状態に変わってしまう――これが構成ドリフトです。

AWS Config のコンプライアンスルールは、このドリフトを自動的に検知してくれます。手作業でひとつひとつ確認する必要がないため、大量のリソースを管理する環境でも安心です。

### CloudWatch / CloudTrail / Config の役割の違い

ここまで3つの監視・管理系サービスを学びました。それぞれの役割を整理しておきましょう。

![CloudWatch / CloudTrail / Config の役割比較](../images/ch10-cloudwatch-cloudtrail-config.drawio.png)

CloudWatch は「いま、リソースがどんな状態か」を見ます。CPU使用率やエラー数などの「健康状態」を監視します。CloudTrail は「誰が、何をしたか」を記録します。API呼び出しの履歴という「行動記録」を残します。Config は「リソースの設定が正しいか」をチェックします。あるべき構成と実際の構成を比較する「設定の検査」を行います。

この3つはよく組み合わせて使われます。たとえば、Config がルール違反を検知したら、CloudWatch アラーム経由で通知を送り、CloudTrail で「誰がいつ設定を変えたか」を調べる、という連携です。

---

> **試験のポイント**
> - AWS Config = リソースの「構成（設定）」が正しいかを継続的にチェック
> - Config Rules でコンプライアンスルールを定義し、違反リソースを自動検出
> - 構成ドリフト（あるべき状態からのずれ）の検知に活用
> - CloudWatch（状態の監視） / CloudTrail（操作の記録） / Config（設定の検査）の3つの違いは頻出
> - 例:「S3バケットがパブリックアクセスを許可していないか」→ Config のユースケース

---

## 10-5. AWS Systems Manager

### このセクションで学ぶこと

EC2インスタンスなどのリソースを一元的に管理する AWS Systems Manager の主要機能を理解します。

### Systems Manager とは

AWS Systems Manager は、EC2インスタンスをはじめとするAWSリソースの運用管理を効率化するためのサービスです。多くの運用タスクを一つの画面から実行・管理できます。

大きなオフィスビルの管理会社をイメージしてください。電球の交換、エアコンのフィルター掃除、セキュリティシステムの点検――ビル全体のメンテナンスをまとめて管理してくれる存在です。Systems Manager は、複数のEC2インスタンスのメンテナンスをまとめて管理してくれるサービスです。

Systems Manager には多くの機能がありますが、試験対策として特に重要な3つを紹介します。

### パラメータストア

パラメータストア は、アプリケーションで使う設定値やパスワードなどを安全に一元管理できる機能です。

アプリケーションを作るとき、データベースの接続先やAPIキーなどの設定情報が必要です。これらをアプリケーションのコードの中に直接書き込んでしまうと、セキュリティ上の問題があります。コードを見た人全員にパスワードが見えてしまうからです。

パラメータストアに設定値を保管しておけば、アプリケーションは「パラメータストアから値を取得する」という形で安全にアクセスできます。値を変更したいときも、パラメータストアの値を書き換えるだけで済みます。

なお、類似サービスとして AWS Secrets Manager があります。Secrets Manager はデータベースの認証情報などを管理し、パスワードの自動ローテーション（定期的な自動変更）機能を備えている点が特徴です。設定値全般の管理にはパラメータストア、認証情報の自動ローテーションが必要な場合は Secrets Manager、と使い分けます。

### パッチマネージャー

パッチマネージャー は、EC2インスタンスのOSやソフトウェアに対するセキュリティパッチ（修正プログラム）の適用を自動化する機能です。

WindowsやLinuxなどのOSには、定期的にセキュリティの修正プログラム（パッチ）がリリースされます。これを適用せずに放置すると、既知の脆弱性を突かれて攻撃を受けるリスクがあります。パッチマネージャーを使えば、「毎週日曜の深夜2時にパッチを適用する」といったスケジュールを設定し、複数のインスタンスへのパッチ適用を自動化できます。

### セッションマネージャー

セッションマネージャー は、EC2インスタンスにSSHキーなしでアクセスできる機能です。

従来、EC2インスタンスに接続するにはSSH（サーバーに遠隔でログインするための仕組み）を使い、専用の鍵ファイル（SSHキー）を管理する必要がありました。この鍵ファイルの紛失や漏洩はセキュリティリスクになります。また、SSHのためにネットワークの接続口（22番ポート）を開ける必要があり、攻撃の入口になりかねません。

セッションマネージャーを使えば、AWSマネジメントコンソール（ブラウザ）やCLIからEC2インスタンスに直接アクセスできます。SSHキーの管理もポートの開放も不要です。さらに、セッションの操作内容がログとして記録されるため、監査にも対応できます。

---

> **試験のポイント**
> - Systems Manager = EC2インスタンスの運用を一元管理するサービス
> - パラメータストア = 設定値全般を安全に管理 / Secrets Manager = 認証情報の自動ローテーションに強み
> - パッチマネージャー = OSパッチの適用を自動化
> - セッションマネージャー = SSHキー不要・ポート開放不要でEC2にアクセス
> - 「SSHキーを使わずにEC2に安全にアクセスする方法は？」→ セッションマネージャー

---

## 10-6. マルチアカウント管理

### このセクションで学ぶこと

複数のAWSアカウントを組織的に管理するための AWS Organizations と AWS Control Tower を理解します。

### なぜマルチアカウントが必要なのか

企業がAWSを本格的に使い始めると、1つのAWSアカウントですべてを管理するのには限界が出てきます。開発環境と本番環境を同じアカウントに置くと、開発者が誤って本番のリソースを削除してしまうリスクがあります。部署ごとにコストを分けて管理したい場合も、アカウントを分けたほうが明確です。

そこで、目的や部署ごとにAWSアカウントを分けて管理する「マルチアカウント戦略」が一般的になっています。しかし、アカウントが10個、20個と増えてくると、今度はアカウントの管理自体が大変になります。

### AWS Organizations

AWS Organizations は、複数のAWSアカウントを一つの組織としてまとめて管理するサービスです。

会社の組織図を思い浮かべてください。社長の下に事業部があり、事業部の下にチームがある、という階層構造です。Organizations も同じように、アカウントを階層的に整理できます。

![Organizations の階層構造](../images/ch10-organizations-hierarchy.drawio.png)

Organizations の階層構造は、以下のように整理されます。まず、一番上にルート（組織全体のトップ）があります。その下にOU（Organizational Unit＝組織単位）を作って、用途や部署ごとにアカウントをグループ化します。そして各OUの中にメンバーアカウント（個々のAWSアカウント）が配置されます。

Organizations を使うと、主に2つの大きなメリットがあります。

一つ目は一括請求（Consolidated Billing）です。すべてのメンバーアカウントの利用料金を、管理アカウント（親アカウント）にまとめて請求できます。経理の手間が減るだけでなく、利用量がまとまることでボリュームディスカウントが適用される場合もあります。

二つ目が SCP（Service Control Policies＝サービスコントロールポリシー）です。SCPは、組織全体やOU単位で「このサービスは使ってはいけない」「このリージョンにはリソースを作ってはいけない」といったアクセス制限をかけるルールです。

SCPとIAMポリシーの関係は重要です。IAMポリシーは「個人やグループに対する許可」を設定するものですが、SCPは「組織全体の上限（ガードレール）」を設定するものです。SCPで禁止されている操作は、たとえIAMポリシーで許可されていても実行できません。SCPが「天井」、IAMポリシーが「その範囲内での許可」と覚えるとわかりやすいでしょう。

たとえば、SCPで「東京リージョン以外でのリソース作成を禁止」と設定すれば、たとえ管理者権限を持つIAMユーザーであっても、他のリージョンにリソースを作ることはできなくなります。

### AWS Control Tower

AWS Control Tower は、マルチアカウント環境のベストプラクティスに沿った初期設定を自動化するサービスです。

Organizations を使えばマルチアカウント管理はできますが、「どんなOUを作ればよいか」「どんなSCPを設定すべきか」「ログ管理用のアカウントはどう作るか」など、ベストプラクティスに沿った設計には多くのノウハウが必要です。Control Tower は、これらのベストプラクティスを「ランディングゾーン」というテンプレートとして提供し、マルチアカウント環境の構築を自動化してくれます。

Control Tower が設定する「ガードレール」は、必須ガードレール（すべての環境で有効にすべきルール）と推奨ガードレール（有効にすることが推奨されるルール）に分かれており、組織のセキュリティとコンプライアンスを自動的に維持してくれます。

---

> **試験のポイント**
> - Organizations = 複数のAWSアカウントを一元管理（一括請求 + SCP）
> - SCP（サービスコントロールポリシー）= 組織全体のアクセス制限（ガードレール）
> - SCPで禁止された操作は、IAMポリシーで許可されていても実行できない
> - Control Tower = マルチアカウント環境のベストプラクティス設定を自動化
> - 「全アカウントで特定のリージョン以外の利用を禁止したい」→ SCP

---

## 10-7. アドバイザリーと最適化ツール

### このセクションで学ぶこと

AWS環境を改善するためのアドバイザリーサービスと最適化ツールを理解します。

### AWS Trusted Advisor

AWS Trusted Advisor は、あなたのAWS環境をベストプラクティスに照らし合わせて自動チェックし、改善点を教えてくれるサービスです。「AWSの専属コンサルタント」のような存在です。

![Trusted Advisor の5カテゴリ](../images/ch10-trusted-advisor.drawio.png)

Trusted Advisor は、以下の5つのカテゴリでチェックを行います。

- コスト最適化: 使っていないリソースや無駄な支出がないかを検出します。「起動しているのに使われていないEC2インスタンスがあります」といった指摘です。
- パフォーマンス: サービスの設定がパフォーマンスの観点で最適かをチェックします。
- セキュリティ: セキュリティ上のリスクがないかを検出します。「S3バケットがパブリック公開されています」「MFAが有効になっていないルートアカウントがあります」といった指摘です。
- 耐障害性: 可用性を高めるための改善提案を行います。
- サービス制限（Service Limits）: AWSの各サービスに設定されている上限値に近づいていないかをチェックします。

ここで重要なのが、サポートプランとの関係です。Basic・Developer プランでは、セキュリティとサービス制限に関する基本的なチェック（コアチェック、約7項目）のみが利用できます。5カテゴリすべてのフルチェック（200項目以上）を利用するには、Business 以上のサポートプランが必要です。詳しくは付録B（サポートプラン）をあわせて確認してください。

### AWS Health Dashboard

AWS Health Dashboard は、AWSのサービスやアカウントに影響する問題を通知してくれるサービスです。2つのビューがあります。

サービスヘルス は、AWSのサービス全体の稼働状況を表示します。「東京リージョンのEC2に障害が発生しています」といった情報を確認できます。これは誰でも閲覧可能です。

アカウントヘルス は、あなたのアカウントに影響する問題を個別に通知してくれます。「あなたが使っているEC2インスタンスのホストサーバーでメンテナンスが予定されています」といった、自分に直接関係する情報が表示されます。

### AWS Compute Optimizer

AWS Compute Optimizer は、EC2インスタンスやEBSボリューム、Lambda関数などのコンピューティングリソースの使用状況を分析し、最適なサイズ（インスタンスタイプなど）を推奨してくれるサービスです。

「今使っているインスタンスは性能が高すぎて無駄が多い」「もう1段階小さいインスタンスに変えればコストを30%削減できる」といった具体的な提案をしてくれます。このサービスは第15章（コスト最適化）でもあらためて取り上げます。

### AWS Well-Architected Tool

AWS Well-Architected Tool は、AWSが推奨する設計のベストプラクティス（Well-Architected Framework）に基づいて、自分のワークロード（システム）を評価できるツールです。

質問に回答する形式でシステムの設計を評価し、改善すべきポイントをレポートとして出力してくれます。Well-Architected Framework は、以下の6つの柱で構成されています。

- 運用上の優秀性（Operational Excellence）
- セキュリティ（Security）
- 信頼性（Reliability）
- パフォーマンス効率（Performance Efficiency）
- コスト最適化（Cost Optimization）
- 持続可能性（Sustainability）

---

> **試験のポイント**
> - Trusted Advisor = 5カテゴリ（コスト最適化・パフォーマンス・セキュリティ・耐障害性・サービス制限）でチェック
> - Trusted Advisor のフルチェックは **Business 以上のサポートプラン** が必要
> - Basic / Developer はコアチェック（約7項目）のみ
> - Health Dashboard = AWSサービスの障害情報やメンテナンス通知
> - Compute Optimizer = リソースの最適なサイズを推奨
> - Well-Architected Tool = 6つの柱に基づいた設計レビュー

---

## 10-8. その他の管理サービス

### このセクションで学ぶこと

試験で概要レベルで問われる可能性のある管理系サービスを押さえます。

### AWS Service Catalog

AWS Service Catalog は、組織内で承認済みのAWSリソースのカタログを作成・管理するサービスです。

社内の備品購入カタログを想像してください。社員が自由に好きなものを買えると管理が大変ですが、承認済みの備品リスト（カタログ）から選んで注文する仕組みにすれば、統制が取れます。Service Catalog も同じで、IT管理者が「使ってよいAWSリソースの組み合わせ」をテンプレート化して提供し、利用者はそのカタログから選んで利用します。これにより、ガバナンス（統制）を保ちつつ、利用者のセルフサービスを可能にします。

### AWS License Manager

AWS License Manager は、ソフトウェアライセンスの使用状況を追跡・管理するサービスです。

企業では、WindowsやOracleなど商用ソフトウェアのライセンスを購入して使用しています。ライセンスの数を超えて使ってしまうとコンプライアンス違反になります。License Manager を使えば、ライセンスの使用状況をAWS上で一元的に追跡し、上限を超えないようにルールを設定できます。

### Service Quotas

Service Quotas は、AWSの各サービスに設定されている上限値（クォータ）を一つの画面から確認・管理するサービスです。

AWSの各サービスには、デフォルトで利用できる上限が設定されています。たとえば「1リージョンで起動できるEC2インスタンスの数」や「1アカウントで作成できるVPCの数」などです。Service Quotas を使えば、現在の使用量と上限値を確認し、必要に応じて上限の引き上げをリクエストできます。

---

> **試験のポイント**
> - Service Catalog = 承認済みリソースのカタログを提供（ガバナンスとセルフサービスの両立）
> - License Manager = ソフトウェアライセンスの使用状況を追跡
> - Service Quotas = AWSサービスの上限値を確認・引き上げリクエスト

## 章末確認問題

### 問題1

Amazon CloudWatch と AWS CloudTrail の違いについて、正しい説明はどれですか？

- **A.** CloudWatch は API 呼び出しを記録し、CloudTrail はリソースのパフォーマンスを監視する
- **B.** CloudWatch はリソースのパフォーマンスを監視し、CloudTrail は API 呼び出し（操作）を記録する
- **C.** CloudWatch と CloudTrail はどちらもリソースの構成変更を追跡するサービスである
- **D.** CloudWatch はセキュリティグループの設定を監視し、CloudTrail はログの保管のみを行う

<details>
<summary>解答・解説</summary>

**解答: B**

解説: CloudWatch は CPU 使用率やエラー数などリソースのパフォーマンスを監視するサービスです。CloudTrail は「誰が・いつ・何の操作をしたか」という API 呼び出しを記録する監査サービスです。A は両者の説明が逆になっています。リソースの構成変更の追跡は AWS Config の役割です。

</details>

---

### 問題2

AWS CloudTrail について正しい説明はどれですか？

- **A.** 管理イベントとデータイベントの両方がデフォルトで記録される
- **B.** 管理イベントはデフォルトで記録されるが、データイベントはデフォルトでは記録されない
- **C.** データイベントのみがデフォルトで記録され、管理イベントは手動で有効化する必要がある
- **D.** CloudTrail はログを無期限に保持するため、追加の設定は不要である

<details>
<summary>解答・解説</summary>

**解答: B**

解説: CloudTrail の管理イベント（リソースの作成・削除・設定変更など）はデフォルトで記録されます。データイベント（S3へのファイルアップロードや Lambda の実行など）は大量に発生するため、デフォルトでは記録されず、明示的に有効化する必要があります。また、通常の CloudTrail ログの保持期間は90日間で、それ以上の保持には CloudTrail Lake や S3 への配信が必要です。

</details>

### 問題3

ある企業のセキュリティ担当者が、「S3バケットがパブリックアクセスを許可していないか」を継続的にチェックしたいと考えています。最も適切なサービスはどれですか？

- **A.** Amazon CloudWatch
- **B.** AWS CloudTrail
- **C.** AWS Config
- **D.** AWS Trusted Advisor

<details>
<summary>解答・解説</summary>

**解答: C**

解説: AWS Config は、リソースの構成（設定）が「あるべき姿」に合っているかを継続的にチェックするサービスです。Config Rules を使って「S3バケットがパブリックアクセスを許可していないこと」というルールを定義すれば、違反しているバケットを自動的に検出できます。CloudWatch はパフォーマンス監視、CloudTrail は操作の記録が主な役割です。Trusted Advisor もセキュリティチェックを行いますが、Config のほうがより細かいカスタムルールの定義に適しています。

</details>

### 問題4

AWS Organizations の SCP（サービスコントロールポリシー）について、正しい説明はどれですか？

- **A.** SCP は IAM ポリシーの代わりに使うもので、IAM ポリシーを設定する必要がなくなる
- **B.** SCP で許可されている操作は、IAM ポリシーで拒否されていても実行できる
- **C.** SCP で禁止されている操作は、IAM ポリシーで許可されていても実行できない
- **D.** SCP は個々の IAM ユーザーに対して適用するポリシーである

<details>
<summary>解答・解説</summary>

**解答: C**

解説: SCP は組織全体やOU単位で適用される「ガードレール（上限）」です。SCP で禁止された操作は、たとえ IAM ポリシーで許可されていても実行できません。SCP は IAM の代わりではなく、IAM の上位にある制限です。SCP が許可していても、IAM ポリシーで許可されていなければ操作はできません。両方の許可が揃って初めて操作が可能になります。

</details>

### 問題5

AWS Systems Manager のセッションマネージャーの特徴として正しいものはどれですか？（2つ選択）

- **A.** SSHキーの管理が不要でEC2インスタンスにアクセスできる
- **B.** セキュリティグループで22番ポートを開ける必要がない
- **C.** EC2インスタンスのCPU使用率を監視できる
- **D.** EC2インスタンスのOSパッチを自動適用できる

<details>

</details>

## 12-1. データ分析の全体像

### このセクションで学ぶこと

データ分析に必要な工程の全体像を理解し、「ETL」「バッチ処理とストリーミング処理」「データレイクとデータウェアハウス」という基本的な概念を押さえます。

### なぜデータ分析が必要なのか

いまや、あらゆるビジネスで大量のデータが日々生まれています。ECサイトのアクセスログ、IoTセンサーの計測値、SNSへの投稿、アプリの利用履歴……。しかし、データはそのままでは「ただの数字や文字の羅列」にすぎません。データを集めて、整えて、分析してはじめて「売れ筋商品はどれか」「どの時間帯にアクセスが集中するか」といったビジネスの意思決定に役立つ情報になります。

データ分析は、大きく4つのステップに分かれます。「収集する」「加工する」「分析する」「可視化する」です。この流れをパイプライン（水道管のように、データがひとつの工程から次の工程へ順番に流れていく仕組み）と呼びます。

![データ分析パイプラインの全体図](../images/ch12-analytics-pipeline.drawio.png)

### ETL とは

データ分析の世界でよく出てくる言葉に「ETL（イーティーエル）」があります。3つの英語の頭文字を取ったものです。

Extract（抽出）は、さまざまなデータソース（データの発生元）からデータを取り出すことです。たとえば、データベースやログファイル、外部のAPIからデータを集めてきます。

Transform（変換）は、集めたデータを使いやすい形に整えることです。日付の形式をそろえたり、不要な列を削除したり、複数のデータを結合したりします。

Load（ロード）は、整えたデータを分析用の保存先に格納することです。

料理にたとえると分かりやすいです。Extractは「冷蔵庫から食材を取り出す」こと。Transformは「皮をむいたり、切ったり、下ごしらえをする」こと。Loadは「下ごしらえした食材をお皿に盛り付けて食卓に並べる」ことです。食材をそのまま食卓に出しても食べられないように、生のデータもそのままでは分析に使えません。ETLという「下ごしらえ」の工程が必要なのです。

### バッチ処理とストリーミング処理

データの処理方法には、大きく2つのアプローチがあります。

バッチ処理は、一定の期間分のデータをまとめて一括で処理する方法です。たとえば「毎晩深夜にその日1日分の売上データを集計する」というのがバッチ処理です。洗濯物をためておいて、まとめて洗濯機を回すようなものです。

ストリーミング処理は、データが発生したそばからリアルタイムに処理する方法です。たとえば「ECサイトのアクセスをリアルタイムに監視して、不正アクセスを即座に検知する」というのがストリーミング処理です。こちらは、蛇口から出る水をそのまま飲む（流れてきたデータをすぐ処理する）ようなイメージです。

どちらが優れているということではなく、用途によって使い分けます。日次のレポート作成ならバッチ処理で十分ですし、不正検知やリアルタイムダッシュボードにはストリーミング処理が必要です。

### データレイクとデータウェアハウス

データ分析の保存先には、大きく2つの考え方があります。この違いは試験でもよく問われるポイントです。

データレイクは、さまざまな形式のデータを「生のまま」大量に保存する場所です。構造化データ（テーブル形式のデータ）も、非構造化データ（画像、動画、ログファイルなど）も、形式を問わずそのまま放り込めます。たとえるなら「天然の湖」です。川から流れ込む水（データ）をそのまま貯めておき、必要になったら汲み出して使います。AWSでは、第3章で学んだ Amazon S3 がデータレイクとしてよく使われます。

データウェアハウス（DWH）は、分析しやすいように整理・構造化して保存する場所です。テーブル形式にきちんと整えたデータだけを格納します。たとえるなら「浄水場の貯水タンク」です。水をろ過して飲めるようにきれいにしてから貯めます。AWSでは Amazon Redshift がデータウェアハウスに相当します。

![データレイクとデータウェアハウスの比較](../images/ch12-datalake-vs-dw.drawio.png)

両者の使い分けをまとめると、こうなります。データレイク（S3）は「まずはあらゆるデータをとにかく貯めておきたい」ときに使います。データの形式が決まっていなくても大丈夫です。コストも安く、容量も事実上無制限です。一方、データウェアハウス（Redshift）は「整理されたデータに対して、高速にSQLで分析したい」ときに使います。大量のデータに対する複雑な集計クエリを高速に実行できます。

実際の分析基盤では、まずデータレイク（S3）にすべてのデータを集め、ETLで整えてからデータウェアハウス（Redshift）に格納する、という流れが一般的です。

なお、AWS Lake Formation というサービスを使うと、データレイクの構築・管理をより簡単に行えます。Lake Formation は、S3と Glue Data Catalog と連携して、データレイクのアクセス権限の管理やセキュリティ設定を一元化してくれるサービスです。「データレイクを安全にまとめて管理したい」というときに活躍します。

---

> **試験のポイント**
> - ETL = Extract（抽出）→ Transform（変換）→ Load（格納）のデータ処理の流れ
> - バッチ処理 = 「まとめて処理」、ストリーミング処理 = 「リアルタイムに処理」
> - データレイク = 生データをそのまま保存（S3）、データウェアハウス = 構造化して保存（Redshift）
> - AWS Lake Formation = データレイクの構築・管理・アクセス権限を一元化するサービス
> - 「データレイクとデータウェアハウスの違い」は試験頻出テーマ

## 12-2. データ収集 ― Amazon Kinesis

### このセクションで学ぶこと

リアルタイムにデータを収集・処理するためのサービスである Amazon Kinesis の役割を理解します。Kinesis Data Streams と Kinesis Data Firehose の違いを押さえます。

### リアルタイムデータ収集の課題

IoTセンサー、Webサイトのクリックストリーム、アプリのログなど、現代のシステムでは毎秒膨大な量のデータが生まれ続けます。こうした「絶え間なく流れてくるデータ」をストリーミングデータと呼びます。

ストリーミングデータを扱うには、いくつかの課題があります。データが大量に、かつ途切れなく流れてくるので、それを取りこぼさずに受け止める仕組みが必要です。さらに、受け止めたデータをリアルタイムに処理したり、分析用の保存先に確実に届けたりする仕組みも求められます。

Amazon Kinesis は、こうしたストリーミングデータの収集と処理を担うサービス群です。水道のたとえで言えば、大量の水を受け止めて、必要な場所に配管で届ける「給水システム」のようなものです。

### Kinesis Data Streams

Kinesis Data Streams は、ストリーミングデータをリアルタイムに収集・処理するためのサービスです。

仕組みはこうです。データの送り手（プロデューサー）がデータをKinesis Data Streamsに送ります。データは「シャード」と呼ばれる処理単位に分けて一時的に保持されます。そして、データの受け手（コンシューマー）がデータを取り出して処理します。

回転寿司にたとえると分かりやすいです。厨房（プロデューサー）が握った寿司（データ）をレーン（ストリーム）に流します。レーンは複数（シャード）あり、お客さん（コンシューマー）は好きなタイミングで皿を取ります。レーンの数を増やせば、より多くの寿司を同時に流せます。

Data Streams はデータをデフォルトで24時間保持します（最大365日まで延長可能）。この保持期間内であれば、コンシューマーはデータを繰り返し読み取ることができます。

Kinesis Data Streams の特徴は、受け取ったデータの処理ロジックを自分で自由に構築できる点です。たとえば、第2章で学んだ Lambda と連携して、流れてきたデータに対してリアルタイムに計算処理を行うことができます。その分、処理プログラムを自分で書く必要があり、柔軟だけど設定が多いサービスです。

### Kinesis Data Firehose（Amazon Data Firehose）

Kinesis Data Firehose（現在の正式名称は Amazon Data Firehose）は、ストリーミングデータをS3やRedshiftなどの保存先に自動的に配信するためのサービスです。

Data Streams との大きな違いは、「処理ロジックを自分で作る必要がない」ことです。Firehoseにデータを送れば、あとは指定した保存先に自動で届けてくれます。配送中に簡単なデータ変換（フォーマットの変更など）を行うこともできます。

たとえるなら、Data Streams が「自分で料理する回転寿司レーン」なら、Firehose は「宅配ピザのデリバリー」です。注文（データ）を受けたら、指定の住所（S3やRedshift）に届けるところまで全部やってくれます。途中でトッピングの追加（簡単な変換）もできますが、本格的な調理（複雑な処理）には向きません。

Kinesis Data Streams と Kinesis Data Firehose の使い分けをまとめると、リアルタイムにデータを処理したい（カスタムロジックを適用したい）場合は Data Streams、データを保存先に自動配信したい（シンプルに届けたい）場合は Firehose です。

---

> **試験のポイント**
> - Kinesis = ストリーミングデータ（リアルタイムに発生するデータ）の収集・処理サービス
> - Kinesis Data Streams = リアルタイム処理（カスタムロジックが必要な場合）
> - Kinesis Data Firehose（現: Amazon Data Firehose）= S3やRedshiftへの自動配信（シンプルに届けたい場合）
> - 「リアルタイムデータの収集」と聞いたら Kinesis を思い出す

---

## 12-3. データ加工（ETL） ― AWS Glue

### このセクションで学ぶこと

データの変換・整理を自動化するサーバーレスETLサービスである AWS Glue の役割を理解します。Glue Data Catalog によるメタデータ管理の仕組みも押さえます。

### ETL の課題

12-1で学んだETL（抽出・変換・格納）を実際にやろうとすると、けっこう大変です。データソースはデータベース、ログファイル、CSVファイルなど多岐にわたります。それぞれフォーマットが違うので、変換処理を一つ一つ作らなければなりません。さらに、ETL処理を動かすためのサーバーを用意して管理する手間もかかります。

こうした面倒を引き受けてくれるのが AWS Glue です。

### AWS Glue とは

AWS Glue は、サーバーレスのETLサービスです。「サーバーレス」とは、第2章で学んだように、サーバーの準備や管理をAWSに任せられる仕組みのことです。ユーザーはサーバーの心配をせずに、ETL処理の中身（どんなデータをどう変換するか）に集中できます。

たとえるなら、Glue は「食品加工工場」です。農家（データソース）から届いた野菜や果物（生データ）を、洗って、切って、パッケージ詰めして（変換して）、スーパーの棚（S3やRedshift）に並べてくれます。工場の建物やラインの管理（サーバー管理）はAWSが面倒を見てくれるので、利用者は「どの野菜をどう加工するか」のレシピを指定するだけです。

Glue の主な機能は2つあります。

1つ目は ETLジョブ です。データの抽出・変換・格納を行う処理そのものです。PythonやScalaでETLスクリプトを書くこともできますし、ビジュアルなGUI（画面操作）でジョブを作ることもできます。

2つ目は Glue Data Catalog です。これがGlueのもう一つの重要な機能で、次に詳しく説明します。

なお、Glueには他にも、ビジュアルにETLジョブを構築できる Glue Studio や、ノーコードでデータのクレンジング・正規化ができる Glue DataBrew といった関連ツールがあります。

### Glue Data Catalog ― データの「目次」

大量のデータを扱っていると、「このデータはどこに保存されていて、どんな形式で、どんな列（カラム）があるのか」が分からなくなることがあります。Glue Data Catalog は、こうしたデータの「メタデータ」（データについての情報）を一元管理する仕組みです。

図書館にたとえると、Data Catalog は「蔵書目録」です。図書館には何千、何万冊もの本がありますが、蔵書目録があれば「この本は3階のB棚の5番目にある」とすぐに分かります。本の中身（データ本体）はS3などに保存されていますが、Data Catalog を見れば「どこに何があるか」「どんな構造か」が一目で分かるのです。

Glue のクローラーという機能を使うと、S3やデータベースにあるデータを自動的にスキャン（調査）して、Data Catalog にメタデータを登録してくれます。手作業でデータの情報を整理する必要がありません。

そして、この Data Catalog は Athena や Redshift Spectrum など他の分析サービスからも参照できます。つまり、Glue Data Catalog は分析基盤全体の「共通の目次」として機能するのです。

---

> **試験のポイント**
> - Glue = サーバーレスのETLサービス（データの抽出・変換・格納を自動化）
> - Glue Data Catalog = データのメタデータ（場所・形式・構造の情報）を一元管理
> - Glue クローラー = データソースを自動スキャンしてData Catalogに登録
> - 「ETL」「データの変換・整理」と聞いたら Glue を思い出す

---

## 12-4. データ分析・クエリ

### このセクションで学ぶこと

AWSが提供する主要なデータ分析サービス（Athena、Redshift、EMR、OpenSearch Service）の特徴と使い分けを理解します。特に、Athena と Redshift の違いは試験でよく問われるポイントです。

### Amazon Athena ― S3のデータにSQLで問い合わせ

Amazon Athena は、S3に保存されたデータに対して、直接SQLで問い合わせ（クエリ）ができるサーバーレスのサービスです。

通常、S3に置かれたCSVファイルやログファイルの内容を分析したいときは、データをどこか別のデータベースに取り込んでから分析する、という手順を踏みます。Athena を使えば、その取り込みの手間を省いて、S3に保存されたままのデータに直接SQLを投げることができます。

たとえるなら、Athena は「出張図書館員」です。わざわざ図書館（データベース）に本（データ）を運ばなくても、倉庫（S3）に直接出向いて本を読んでくれます。しかも、図書館員を常駐させる費用（サーバーの維持費）は不要で、本を読んだ分（スキャンしたデータ量）だけ料金がかかります。

Athena の主な特徴を整理します。サーバーレスなので、インフラの管理が不要です。S3にあるデータをそのまま分析できます。課金はクエリでスキャンしたデータ量に基づくので、使わないときはコストがかかりません。Glue Data Catalog と連携して、S3上のデータの構造を把握します。内部的にはPrestoベースの分散クエリエンジン（現在はTrino）が動いており、大量のデータに対しても高速にクエリを実行できます。

Athena は、特にアドホック（その場の必要に応じた）なクエリに向いています。スキャン量課金のため、頻度が低い分析ではコスト効率に優れています。たとえば「先月のアクセスログから、特定のエラーが何回発生したか調べたい」といった一時的な分析に最適です。

### Amazon Redshift ― 大規模データウェアハウス

Amazon Redshift は、AWSが提供するフルマネージドのデータウェアハウスサービスです。12-1で学んだように、データウェアハウスは「分析用に構造化して整理されたデータを保存する場所」です。

Redshift は、テラバイト（TB。映画約200本分に相当する約1兆バイト）からペタバイト（PB。その1,000倍）規模の大量データに対する複雑な分析クエリを高速に実行するために最適化されています。Redshift は「列指向」と呼ばれる方式でデータを格納しています。通常のデータベースは行単位でデータを格納しますが、列指向では列（カラム）単位でデータを格納します。たとえば「売上金額の合計」を計算するとき、行指向だと全行を読み込む必要がありますが、列指向なら売上金額の列だけを読めばよいので、分析が高速になります。

ここで重要なのが、Athena と Redshift の使い分けです。この違いは試験でよく問われます。

Athena は、S3に保存されたデータに対してサーバーレスでSQLを実行できるサービスです。インフラの管理が不要で、たまに分析したいとき（アドホッククエリ）に向いています。クエリの頻度が低い場合はコスト効率が良いです。

Redshift は、専用のクラスター（サーバー群）を立てて運用するデータウェアハウスです。大量のデータに対して頻繁に複雑なクエリを実行する場合に向いています。常時稼働するので、クエリを実行しなくても基本料金がかかりますが、頻繁に分析するなら Athena より効率的です。

なお、Redshift には従来のクラスターを自分で構成する方式に加えて、Redshift Serverless というサーバーレス版もあります。Serverless版ではクラスターの管理が不要になりますが、大規模で継続的な分析にはプロビジョニング型（自分でクラスターを構成する方式）が適しています。

まとめると、「たまに分析する、手軽に始めたい」なら Athena、「毎日のように大規模な分析を繰り返す」なら Redshift、と覚えておくと良いでしょう。

### Amazon EMR ― ビッグデータの分散処理

Amazon EMR（Elastic MapReduce）は、Apache Hadoop や Apache Spark といったオープンソースのビッグデータ処理フレームワークをマネージドで実行できるサービスです。

たとえるなら、EMR は大量のデータを処理するための「巨大な計算工場」です。一つの機械（サーバー）では処理しきれないほどのデータを、何十台もの機械で手分けして処理します。Hadoop や Spark は、こうした分散処理を実現するためのソフトウェアです。自前でこれらを構築・運用するのは非常に手間がかかりますが、EMRを使えばクラスターの構築・管理をAWSに任せられます。

EMR は、Athena や Redshift では対応しにくい、非常に大規模なデータの変換処理や、機械学習の前処理など、高度なデータ処理に使われます。クラウドプラクティショナー試験では深い知識は求められませんが、「Hadoop/Sparkのマネージドサービス = EMR」という対応を覚えておけば十分です。

### Amazon OpenSearch Service ― 検索と分析

Amazon OpenSearch Service は、大量のデータに対する全文検索やログ分析を行うためのマネージドサービスです。もともと「Amazon Elasticsearch Service」という名前でしたが、現在はOpenSearch Serviceに改名されています。

主な用途は、アプリケーションのログ分析、Webサイトの検索機能の構築、セキュリティイベントの監視などです。ダッシュボード機能も備えており、データをリアルタイムに可視化することもできます。

試験では「ログ分析」「全文検索」というキーワードと OpenSearch Service を結びつけて覚えておきましょう。

---

> **試験のポイント**
> - Athena = S3上のデータにサーバーレスでSQLクエリ。アドホック分析向き、スキャン量課金
> - Redshift = 大規模データウェアハウス。頻繁な分析クエリ向き、常時稼働
> - Athena vs Redshift: 「たまに分析」→ Athena、「毎日大規模分析」→ Redshift
> - EMR = Hadoop/Sparkのマネージドサービス（大規模分散処理）
> - OpenSearch Service = ログ分析・全文検索のマネージドサービス

---

## 12-5. データ可視化 ― Amazon QuickSight

### このセクションで学ぶこと

データ分析の結果を「見える化」するためのBIツールである Amazon QuickSight の役割を理解します。

### なぜ「可視化」が必要なのか

ここまでで、データを「集めて」「整えて」「分析する」方法を学んできました。しかし、分析結果が数字の羅列やテーブルの形のままでは、ビジネス上の意思決定に活かしにくいです。経営者や現場の担当者が「ひと目で状況を把握できる」ようにするためには、グラフやチャートで視覚的に表現する必要があります。

こうした「データの見える化」を行うツールを、BI（Business Intelligence）ツールと呼びます。

### Amazon QuickSight とは

Amazon QuickSight は、AWSが提供するクラウドネイティブのBIサービスです。データからグラフやチャートを作成し、ダッシュボード（さまざまなグラフをまとめた画面）として共有できます。

QuickSight の大きな特徴は、AWSの他のサービスと簡単に連携できることです。Athena、Redshift、S3、RDSなど、さまざまなデータソースに接続して、そのデータをグラフやチャートにできます。つまり、この章で学んできたデータ分析パイプラインの「最終ゴール」として、QuickSight でダッシュボードを作る、という流れが自然にできるのです。

QuickSight のもう一つの特徴は、サーバーレスであることです。ダッシュボードを閲覧するユーザー数に応じた従量課金なので、BIツール用のサーバーを用意する必要がありません。

さらに、QuickSight には ML Insights という機能があり、データの異常検知や将来予測を自動で行えます。また、Amazon Q in QuickSight を使えば、自然言語（日本語や英語）で質問するだけでグラフを自動生成できます。たとえば「先月の売上トップ5の商品は？」と入力するだけで、対応するグラフが表示されるイメージです。

たとえるなら、QuickSight は「会議室のホワイトボード」です。分析チームが集めた生データ（メモや数字）を、分かりやすいグラフ（図）としてホワイトボードに描き出し、会議の参加者全員が一目で理解できるようにします。しかも、このホワイトボードはクラウド上にあるので、どこからでも見ることができます。

### データ分析パイプラインのまとめ

この章で学んだAWSサービスを、データ分析パイプラインの流れに沿ってまとめます。

データソース（Webアプリ、IoTセンサー、ログなど）からデータが発生します。それを Kinesis Data Streams や Kinesis Data Firehose でリアルタイムに収集します。収集したデータは、まず S3（データレイク）に保存します。AWS Glue がデータの変換・整理（ETL）を行い、Glue Data Catalog にメタデータを登録します。整理されたデータに対して、Athena でアドホック分析を行ったり、Redshift に格納して定常的な分析を行ったりします。そして最終的に、QuickSight でダッシュボードを作成し、ビジネスの意思決定に活用します。

このパイプライン全体を理解しておくと、試験で「このサービスはどの工程で使われるか」という問題に対応しやすくなります。

---

> **試験のポイント**
> - QuickSight = AWSネイティブのBIサービス（データの可視化・ダッシュボード作成）
> - ML Insights で異常検知・予測が可能。Amazon Q in QuickSight で自然言語からグラフを自動生成
> - Athena、Redshift、S3、RDSなどさまざまなデータソースと連携可能
> - サーバーレスで従量課金
> - データ分析パイプライン全体の流れ: 収集（Kinesis）→ 保存（S3）→ 加工（Glue）→ 分析（Athena/Redshift）→ 可視化（QuickSight）

### 問題1

データレイクとデータウェアハウスの違いについて、正しい説明はどれですか？

- **A.** データレイクは構造化されたデータのみを保存し、データウェアハウスはあらゆる形式のデータを保存する
- **B.** データレイクはあらゆる形式のデータを生のまま保存し、データウェアハウスは分析用に構造化されたデータを保存する
- **C.** データレイクとデータウェアハウスは、どちらも同じ方法でデータを保存する
- **D.** データウェアハウスはデータレイクの一部であり、独立した概念ではない

<details>
<summary>解答・解説</summary>

**解答: B**

解説: **データレイク**は、構造化データも非構造化データも「生のまま」保存する場所です（AWSではS3がよく使われます）。**データウェアハウス**は、分析しやすいように構造化・整理されたデータを保存する場所です（AWSではRedshiftが該当）。この違いは試験で頻出です。

</details>

---

### 問題2

リアルタイムに発生するストリーミングデータを収集・処理するために最も適したAWSサービスはどれですか？

- **A.** Amazon S3
- **B.** Amazon Kinesis
- **C.** Amazon Athena
- **D.** AWS Glue

<details>
<summary>解答・解説</summary>

**解答: B**

解説: **Amazon Kinesis**は、ストリーミングデータ（リアルタイムに発生し続けるデータ）を収集・処理するためのサービスです。S3はデータの保存、AthenaはS3上のデータへのSQLクエリ、GlueはETL処理に使われます。

</details>

---

### 問題3

Kinesis Data Streams と Kinesis Data Firehose の違いについて、正しい説明はどれですか？

- **A.** Data Streams はデータをS3に自動配信し、Firehose はカスタム処理ロジックを適用する
- **B.** Data Streams はカスタム処理ロジックを適用でき、Firehose はS3やRedshiftへの自動配信に特化している
- **C.** Data Streams と Firehose は同じ機能を持ち、名前が異なるだけである
- **D.** Firehose はバッチ処理専用のサービスである

<details>
<summary>解答・解説</summary>

**解答: B**

解説: **Kinesis Data Streams**は、ストリーミングデータにカスタムの処理ロジック（Lambdaなど）を適用できます。**Kinesis Data Firehose**は、データをS3やRedshiftなどの保存先に自動的に配信するサービスで、シンプルにデータを届けたい場合に向いています。

</details>

---

### 問題4

AWS Glueの説明として最も適切なものはどれですか？

- **A.** S3に保存されたデータにSQLで問い合わせを行うサーバーレスサービス
- **B.** データの抽出・変換・格納（ETL）を行うサーバーレスサービス
- **C.** リアルタイムデータを収集するストリーミングサービス
- **D.** データを可視化するBIダッシュボードサービス

<details>
<summary>解答・解説</summary>

**解答: B**

解説: **AWS Glue**は、サーバーレスのETLサービスです。データの抽出（Extract）・変換（Transform）・格納（Load）を自動化します。AはAmazon Athena、CはAmazon Kinesis、DはAmazon QuickSightの説明です。

</details>

---

### 問題5

S3に保存されているCSVファイルに対して、サーバーを用意することなくSQLで分析を行いたい場合、最も適したサービスはどれですか？

- **A.** Amazon Redshift
- **B.** Amazon RDS
- **C.** Amazon Athena
- **D.** Amazon EMR

<details>
<summary>解答・解説</summary>

**解答: C**

解説: **Amazon Athena**は、S3に保存されたデータに対してサーバーレスでSQLクエリを実行できるサービスです。サーバーの管理が不要で、スキャンしたデータ量に応じた従量課金です。Redshiftは専用クラスターが必要で、RDSはトランザクション処理向けのデータベースです。

</details>

### 問題6

大量の構造化データに対して、毎日のように複雑な集計・分析クエリを実行する場合に最も適したサービスはどれですか？

- **A.** Amazon Athena
- **B.** Amazon Redshift
- **C.** Amazon DynamoDB
- **D.** Amazon S3

<details>
<summary>解答・解説</summary>

**解答: B**

解説: **Amazon Redshift**は、大規模データウェアハウスとして、頻繁に複雑な分析クエリを実行するワークロードに最適化されています。Athenaもクエリは可能ですが、アドホック（たまに行う）分析向きです。DynamoDBはNoSQLデータベース、S3はオブジェクトストレージであり、複雑なSQLクエリの高速実行には向いていません。

</details>

### 問題7

Glue Data Catalogの役割として最も適切なものはどれですか？

- **A.** データの暗号化と復号を管理する
- **B.** データのメタデータ（場所・形式・構造の情報）を一元管理する
- **C.** データのバックアップを自動的に作成する
- **D.** データをリアルタイムにストリーミング配信する

<details>
<summary>解答・解説</summary>

**解答: B**

解説: **Glue Data Catalog**は、データのメタデータ（どこに保存されていて、どんな形式で、どんな列があるかなどの情報）を一元管理する仕組みです。Athenaなどの分析サービスがData Catalogを参照してデータの構造を把握します。

</details>

### 問題8

データ分析の結果をグラフやチャートで可視化し、ダッシュボードとして社内に共有したい場合、最も適したAWSサービスはどれですか？

- **A.** Amazon Athena
- **B.** Amazon CloudWatch
- **C.** Amazon QuickSight
- **D.** AWS Glue

<details>
<summary>解答・解説</summary>

**解答: C**

解説: **Amazon QuickSight**は、AWSネイティブのBIサービスで、データをグラフやチャートで可視化し、ダッシュボードとして共有できます。Athena、Redshift、S3などさまざまなデータソースと連携可能です。AthenaはSQLクエリ実行、CloudWatchはAWSリソースの監視、GlueはETL処理のサービスです。

</details>

## 11-1. CI/CD とは何か

### このセクションで学ぶこと

CI（継続的インテグレーション）とCD（継続的デリバリー/デプロイメント）の基本概念を理解し、なぜ自動化が重要なのかを把握します。

### 手動デプロイの課題

アプリケーションを開発したら、それをサーバーに配置して動かす必要があります。この「サーバーに配置して動かす」作業のことを「デプロイ」と呼びます。

昔ながらのやり方では、開発者がコードを書き終えたら、手作業でサーバーにファイルをアップロードし、設定を変更し、動作確認をしていました。この手動デプロイには、いくつかの深刻な問題があります。

まず、ヒューマンエラーのリスクです。手作業には必ずミスがつきまといます。ファイルを1つ上げ忘れた、設定ファイルを書き間違えた、手順を1つ飛ばした——こうしたミスが、サービスの停止やデータの消失につながることがあります。

次に、リリースの遅延です。手動作業は時間がかかるうえ、作業できる人が限られます。「デプロイ職人」と呼ばれるベテランしかリリースできない、という状況になりがちです。結果として、新機能やバグ修正をお客様に届けるスピードが遅くなります。

最後に、品質のばらつきです。手動でテストを行うと、テストの範囲や深さが人によって異なります。「今回は大丈夫だろう」と思ってテストを省略した結果、本番環境で障害が発生するケースも珍しくありません。

### CI（継続的インテグレーション）とは

こうした課題を解決するために生まれたのが、CI/CDという考え方です。

CI は「Continuous Integration（継続的インテグレーション）」の略です。日本語にすると少し難しく聞こえますが、やっていることはシンプルです。「開発者がコードを変更するたびに、自動でビルドとテストを実行する」という仕組みです。

たとえるなら、工場の品質検査ラインのようなものです。製品（コード）が作られるたびに、自動で検査装置（ビルド・テスト）を通します。不良品（バグ）があれば、その場ですぐに検出できます。人の目で一つずつチェックするよりも速く、正確で、漏れがありません。

CIを導入すると、開発者がコードを変更してリポジトリ（コードの保管場所）にプッシュ（自分のPCで書いたコードをチーム共有の保管場所にアップロードすること）した瞬間に、自動でビルド（ソースコードを実行可能な形に変換すること）が行われ、続けてテストが走ります。テストに失敗すれば、すぐに開発者に通知が届きます。問題を早期に発見できるので、修正も簡単です。

### CD（継続的デリバリー / デプロイメント）とは

CD には、厳密には2つの意味があります。

1つ目は「Continuous Delivery（継続的デリバリー）」です。CIでテストに合格したコードを、いつでもデプロイできる状態に保つことを指します。ただし、実際のデプロイ（本番環境への反映）は、人間が最終承認をしてから行います。

2つ目は「Continuous Deployment（継続的デプロイメント）」です。こちらはさらに一歩進んで、テストに合格したコードを自動的に本番環境にデプロイするところまでを含みます。人間の承認なしに、コード変更が自動で本番に反映されます。

試験では、この2つを厳密に区別する問題は出にくいので、「CD = テスト済みのコードを自動（または半自動）でデプロイする仕組み」と理解しておけば十分です。

CI/CDを組み合わせると、開発者がコードを書いてプッシュするだけで、ビルド → テスト → デプロイが自動的に流れるようになります。これを「パイプライン」と呼びます。工場のベルトコンベアをイメージしてください。原材料（ソースコード）を入れると、組み立て（ビルド）→ 検査（テスト）→ 出荷（デプロイ）が自動で進んでいく仕組みです。

---

> **試験のポイント**
> - CI = 「コード変更のたびに自動でビルド・テストを実行する仕組み」
> - CD = 「テスト済みのコードを自動（または半自動）でデプロイする仕組み」
> - CI/CD の目的: ヒューマンエラーの防止、リリース頻度の向上、品質の安定化
> - パイプライン = ソースコードからデプロイまでの自動化された一連の流れ

---

## 11-2. AWS CodeBuild / CodeDeploy / CodePipeline

### このセクションで学ぶこと

AWSが提供するCI/CDサービスの役割と、それぞれの関係を理解します。

### パイプラインを構成する3つのサービス

AWSでは、CI/CDパイプラインを構築するために、主に3つのサービスを組み合わせて使います。それぞれの役割を、工場のベルトコンベアのたとえに合わせて説明します。

**AWS CodePipeline** は、パイプライン全体を管理するサービスです。ベルトコンベアの制御盤に相当します。「ソースコードを取得 → ビルド・テスト → デプロイ」という一連の流れ（ステージ）を定義し、各ステージを順番に実行します。あるステージが失敗すれば、次のステージには進みません。

**AWS CodeBuild** は、ビルドとテストを実行するサービスです。工場の組み立て・検査ラインに相当します。ソースコードを受け取り、コンパイル（ビルド）やテストの実行を行います。CodeBuild はフルマネージドサービスなので、ビルド用のサーバーを自分で用意・管理する必要はありません。ビルドが実行されている時間だけ課金される、従量課金モデルです。オンプレミス環境でJenkinsのビルドジョブを使ったことがある方は、「ビルドサーバーの管理が不要になったJenkins」のようなイメージで捉えると分かりやすいでしょう。

**AWS CodeDeploy** は、デプロイを自動化するサービスです。工場の出荷ラインに相当します。ビルド・テスト済みのアプリケーションを、EC2インスタンス、Lambda関数、ECSコンテナなどのデプロイ先に自動で配置します。デプロイ中に問題が発生した場合は、自動でロールバック（前のバージョンに戻すこと）する機能も備えています。

### パイプラインの流れ

3つのサービスがどう連携するか、全体の流れを見てみましょう。

![CI/CDパイプラインのフロー](../images/ch11-cicd-pipeline.drawio.png)

開発者がコードを変更してGitHubなどの外部リポジトリサービスにプッシュすると、CodePipelineがそれを検知してパイプラインを開始します。なお、以前はAWS CodeCommitというソースコード管理サービスも選択肢でしたが、2024年7月以降は新規利用が停止されています。

まず、ソースステージでコードを取得します。次に、ビルドステージでCodeBuildがビルドとテストを実行します。テストに合格すれば、デプロイステージでCodeDeployがアプリケーションをデプロイ先に配置します。

この一連の流れが自動で進むため、開発者はコードを書いてプッシュするだけで、あとはパイプラインが面倒を見てくれます。

### 試験で問われやすい3つの役割の違い

試験では、CodePipeline・CodeBuild・CodeDeployの役割を混同させる選択肢がよく出ます。それぞれの担当範囲を明確に区別しておきましょう。

CodePipelineは「指揮者」です。パイプライン全体のステージ管理と実行制御を担当します。自分自身がビルドやデプロイを行うわけではなく、各ステージを適切なサービスに委託して、全体の流れを管理します。オンプレミスのJenkinsパイプラインやGitHub Actionsのワークフローに相当するAWSネイティブサービスです。

CodeBuildは「ビルド・テスト担当」です。ソースコードをコンパイルし、テストを実行し、デプロイ可能な成果物（アーティファクト）を生成します。

CodeDeployは「デプロイ担当」です。成果物をEC2、Lambda、ECSなどの実行環境に配置します。主なデプロイ戦略として、ローリングデプロイとBlue/Greenデプロイがあります。ローリングデプロイは、稼働中のインスタンスを少しずつ更新する方式で、デプロイ中は新旧バージョンが混在します。一方、Blue/Greenデプロイは、新環境（Green）を丸ごと用意してからトラフィックを切り替える方式で、切り替えが瞬時に行え、問題があれば旧環境（Blue）にすぐ戻せるのが利点です。

---

> **試験のポイント**
> - CodePipeline = 「CI/CDの全体フローを管理するパイプラインサービス」
> - CodeBuild = 「ビルドとテストを実行するフルマネージドサービス」
> - CodeDeploy = 「EC2 / Lambda / ECS へのデプロイを自動化するサービス」
> - パイプラインの流れ: ソース → CodeBuild（ビルド・テスト）→ CodeDeploy（デプロイ）
> - CodeBuild はフルマネージド（サーバーの管理が不要）で従量課金

---

## 11-3. Infrastructure as Code（IaC）

### このセクションで学ぶこと

インフラをコードで管理する「Infrastructure as Code」の概念を理解し、AWS CloudFormation と AWS CDK の役割を把握します。

### なぜインフラをコードで管理するのか

前のセクションでは、アプリケーションのデプロイを自動化する話をしました。では、そのアプリケーションが動く「インフラ」（サーバー、ネットワーク、データベースなど）はどうでしょうか。

従来、インフラの構築はマネジメントコンソール（Webブラウザの管理画面）で手作業で行うのが一般的でした。EC2インスタンスを立てて、セキュリティグループを設定して、RDSデータベースを作って、S3バケットを用意して……。この手作業にも、アプリケーションのデプロイと同じ課題があります。手順を間違えるリスク、人によって設定が微妙に異なるリスク、そして「この環境、誰がいつどう設定したのか分からない」という管理の問題です。

Infrastructure as Code（IaC）は、こうした課題を解決する考え方です。日本語にすると「コードとしてのインフラ」。つまり、インフラの構成をテキストファイル（コード）で定義し、そのファイルを使ってインフラを自動的に構築・管理する手法です。

たとえるなら、IaCは「家の設計図」です。設計図（コード）があれば、同じ家を何棟でも正確に建てられます。設計図がなければ、大工さんの記憶と勘に頼ることになり、毎回微妙に違う家ができてしまいます。

IaCの主な利点は3つあります。

1つ目は**再現性**です。同じテンプレートから、まったく同じ環境を何度でも構築できます。開発環境、テスト環境、本番環境を完全に同じ構成で作れるため、「テスト環境では動いたのに本番で動かない」というトラブルを防げます。

2つ目は**バージョン管理**です。インフラの定義がテキストファイルなので、Gitなどのバージョン管理システムで変更履歴を追跡できます。「いつ、誰が、何を変更したか」が一目で分かります。

3つ目は**自動化**です。コードを実行するだけでインフラが構築されるため、手作業による設定ミスがなくなり、構築にかかる時間も大幅に短縮されます。

### AWS CloudFormation

AWS CloudFormation は、AWSにおけるIaCの中核サービスです。第0章で簡単に紹介しましたが、ここではもう少し詳しく見ていきましょう。

CloudFormationでは、「テンプレート」と呼ばれるテキストファイルにAWSリソースの構成を記述します。テンプレートはJSON形式やYAML形式（いずれも人間が読みやすいテキストベースの設定ファイルの書き方）で書きます。たとえば「EC2インスタンスを1台、t3.microのサイズで起動し、セキュリティグループでポート80を開放する」といった内容を記述します。

このテンプレートをCloudFormationに渡すと、CloudFormationがテンプレートの内容を読み取り、記述されたAWSリソースを自動的に構築します。構築されたリソースの集合を「スタック」と呼びます。

![CloudFormationの動作概念図](../images/ch11-cloudformation-flow.drawio.png)

CloudFormationの便利な点は、スタック単位でリソースを管理できることです。不要になったらスタックを削除するだけで、関連するリソースがすべてまとめて削除されます。「あのリソース消し忘れた！」という課金トラブルを防げるわけです。

また、既存のスタックに変更を加えたい場合は「変更セット」という仕組みが使えます。変更セットは、テンプレートの変更内容を実際に適用する前にプレビューできる機能です。「この変更を適用すると、どのリソースが追加・変更・削除されるか」を事前に確認してから、安心して変更を実行できます。

さらに、CloudFormationには「ドリフト検出」という機能もあります。テンプレートで定義した状態と、実際のリソースの状態が異なっていないか（誰かが手動で変更していないか）を確認する機能です。IaCで管理しているつもりでも、緊急対応などで手動変更が入ることがあります。ドリフト検出を使えば、そうした「テンプレートと実態のズレ」を発見できます。

### AWS CDK（Cloud Development Kit）

CloudFormationは非常に便利ですが、JSONやYAMLでインフラを記述するのは、プログラマーにとっては少し面倒に感じることがあります。特に大規模なインフラになると、テンプレートが数千行になることもあり、管理が大変です。

AWS CDK（Cloud Development Kit）は、この課題を解決するツールです。Python、TypeScript、Javaなど、馴染みのあるプログラミング言語を使ってインフラを定義できます。

CDKで書いたコードは、内部的にCloudFormationテンプレートに変換されます。つまり、CDKは「プログラミング言語で書ける CloudFormation」と考えることができます。プログラミング言語の力（変数、ループ、条件分岐など）を使えるので、同じようなリソースを繰り返し定義する場合でも、簡潔に書くことができます。

試験で押さえておくべきポイントは、CDKとCloudFormationの関係です。CDKはCloudFormationの「代替」ではなく、CloudFormationの「上位レイヤー」です。CDKで書いたコードは最終的にCloudFormationテンプレートとして処理されます。

なお、AWS以外のクラウドも含めてIaCを実践する場合は、HashiCorpのTerraformが広く使われています。TerraformはAWS・Azure・Google Cloudなどマルチクラウドに対応していますが、CloudFormation/CDKはAWSに特化しておりAWSサービスとの統合が深いという特徴があります。CLF-C02試験ではCloudFormation/CDKが出題対象です。

---

> **試験のポイント**
> - IaC = 「インフラの構成をコードで定義し、自動で構築・管理する手法」
> - IaCの利点: 再現性、バージョン管理、自動化
> - CloudFormation = 「JSON/YAMLのテンプレートからAWSリソースを自動構築するサービス」
> - テンプレート → CloudFormation → スタック（リソースの集合）の流れを理解する
> - 変更セット = 「変更を適用する前にプレビューできる仕組み」
> - ドリフト検出 = 「テンプレートの定義と実際のリソース状態のズレを検出する機能」
> - CDK = 「プログラミング言語でインフラを定義 → CloudFormationテンプレートに変換」
> - CDKはCloudFormationの上位レイヤー（内部的にCloudFormationを使用）

---

## 11-4. 分散トレーシングとデバッグ

### このセクションで学ぶこと

マイクロサービスアーキテクチャにおけるデバッグの難しさと、AWS X-Rayによる解決方法を理解します。

### マイクロサービスのデバッグが難しい理由

第6章や第8章で学んだように、現代のアプリケーションは複数の小さなサービスが連携して動く「マイクロサービス」の構成をとることが多くなっています。たとえば、ECサイトの注文処理では、注文サービス → 在庫サービス → 決済サービス → 通知サービスのように、複数のサービスがリレーのようにリクエストを受け渡します。

このような構成では、「なんだか処理が遅い」「たまにエラーが出る」といった問題が発生したとき、原因の特定が非常に難しくなります。1つのリクエストが5つも10つものサービスを経由するため、「どのサービスで遅延やエラーが起きているのか」を追跡するのは、糸の絡まった毛糸玉をほどくような作業になります。

### AWS X-Ray とは

AWS X-Ray は、この問題を解決するための「分散トレーシング」サービスです。

分散トレーシングとは、複数のサービスをまたがるリクエストの流れを追跡・可視化する技術です。X-Rayを使うと、1つのリクエストがどのサービスを、どの順番で、どれくらいの時間をかけて通過したかが一目で分かるようになります。

たとえるなら、X-Rayは「荷物の追跡サービス」のようなものです。宅配便で荷物を送ると、「集荷 → 配送センターA → 配送センターB → お届け先」のように、荷物が今どこにあるかをリアルタイムで追跡できます。X-Rayも同じように、リクエストという「荷物」がサービス間をどう流れているかを追跡してくれます。

X-Rayが特に役立つのは、パフォーマンスのボトルネック（処理の遅い箇所）を特定する場面です。「注文処理に5秒かかっているが、そのうち4秒は決済サービスで費やされている」といった具合に、問題箇所をピンポイントで見つけることができます。

X-Rayは、EC2上のアプリケーション、Lambda関数、ECSコンテナ、API Gatewayなど、さまざまなAWSサービスと連携して使えます。

---

> **試験のポイント**
> - X-Ray = 「分散システム（マイクロサービス）のリクエストフローを可視化し、問題箇所を特定するサービス」
> - ユースケース: パフォーマンスのボトルネック特定、エラー原因の追跡
> - 「マイクロサービス間のリクエストを追跡・可視化」というキーワードが出たら X-Ray を選ぶ

---

## 11-5. フロントエンド・モバイル開発

### このセクションで学ぶこと

フロントエンドやモバイルアプリの開発を支援するAWSサービスの概要を把握します。CLF-C02試験では深い理解は求められませんが、サービス名と基本的な役割を知っておきましょう。

### AWS Amplify

AWS Amplify は、フロントエンド開発者やモバイルアプリ開発者がフルスタック（画面からバックエンドまで）のWebアプリケーションを素早く構築・デプロイするためのサービスです。

通常、Webアプリケーションを作るには、フロントエンド（画面側）だけでなく、認証（ログイン機能）、API、データベース、ストレージ（ファイル保管）など、バックエンド側の仕組みもたくさん必要です。これらを一つずつ自分で設定するのは大変ですが、Amplifyを使うと、こうしたバックエンド機能を簡単なコマンドや設定で追加できます。

また、AmplifyにはGitリポジトリと連携した自動デプロイ機能もあります。コードをプッシュするだけで、ビルドからデプロイまでが自動で行われます。フロントエンド開発に特化したCI/CDの仕組みが組み込まれている、と考えるとイメージしやすいでしょう。

### AWS AppSync

AWS AppSync は、GraphQL（グラフキューエル）というAPI技術を使ったマネージドサービスです。

GraphQLは、従来のREST APIに比べて、「必要なデータだけを効率的に取得できる」という特徴を持つAPI仕様です。たとえば、ユーザー情報の中から名前とメールアドレスだけが欲しい場合、REST APIでは全てのユーザー情報が返ってきてしまうことがありますが、GraphQLでは欲しいデータだけを指定して取得できます。

AppSyncは、このGraphQL APIを簡単に構築・運用できるサービスです。リアルタイムのデータ同期やオフラインアクセスなどの機能も備えており、モバイルアプリやリアルタイム通信が必要なアプリケーションに適しています。

試験では、Amplifyは「フロントエンド/モバイルアプリの統合開発プラットフォーム」、AppSyncは「マネージドGraphQLサービス」という基本的な位置づけだけ押さえておけば十分です。

## 章末確認問題

### 問題1

CI（継続的インテグレーション）の説明として最も適切なものはどれですか？

- **A.** テスト済みのアプリケーションを自動的に本番環境にデプロイする仕組み
- **B.** コード変更のたびに自動でビルドとテストを実行する仕組み
- **C.** インフラの構成をコードで定義して自動構築する手法
- **D.** 複数のサービス間のリクエストを追跡・可視化する技術

<details>
<summary>解答・解説</summary>

**解答: B**

解説: CI（継続的インテグレーション）は、**コード変更のたびに自動でビルドとテストを実行する仕組み**です。AはCD（継続的デプロイメント）、CはIaC（Infrastructure as Code）、DはX-Rayなどの分散トレーシングの説明です。

</details>

### 問題2

AWS CloudFormationの説明として正しいものはどれですか？

- **A.** プログラミング言語を使ってインフラを定義するツール
- **B.** ソースコードのビルドとテストを実行するサービス
- **C.** JSON/YAMLのテンプレートからAWSリソースを自動構築するサービス
- **D.** アプリケーションのデプロイを自動化するサービス

<details>
<summary>解答・解説</summary>

**解答: C**

解説: AWS CloudFormationは、**JSON/YAMLのテンプレートからAWSリソースを自動構築するサービス**です。AはAWS CDKの説明、BはAWS CodeBuild、DはAWS CodeDeployの説明です。CloudFormationのテンプレートで定義されたリソースの集合は「スタック」と呼ばれます。

</details>

### 問題3

AWS CodePipelineの役割として最も適切なものはどれですか？

- **A.** ソースコードのビルドとテストを実行する
- **B.** アプリケーションをEC2やLambdaにデプロイする
- **C.** ソースからビルド、デプロイまでのCI/CDパイプライン全体を管理する
- **D.** インフラの構成をテンプレートで定義する

<details>
<summary>解答・解説</summary>

**解答: C**

解説: AWS CodePipelineは、**ソースからビルド、デプロイまでのCI/CDパイプライン全体を管理するサービス**です。パイプラインの各ステージ（ソース取得 → ビルド・テスト → デプロイ）を定義し、順番に実行します。AはCodeBuild、BはCodeDeploy、DはCloudFormationの役割です。

</details>

### 問題4

AWS X-Rayのユースケースとして最も適切なものはどれですか？

- **A.** EC2インスタンスのCPU使用率やメモリ使用量を監視する
- **B.** 複数のマイクロサービスをまたがるリクエストを追跡し、ボトルネックを特定する
- **C.** AWSアカウントで行われたAPI操作の監査ログを記録する
- **D.** アプリケーションのログをリアルタイムで収集・分析する

<details>
<summary>解答・解説</summary>

**解答: B**

解説: AWS X-Rayは、**分散システム（マイクロサービス）のリクエストフローを可視化し、ボトルネックやエラーの原因を特定する**ためのサービスです。AはCloudWatch メトリクス、CはCloudTrail、DはCloudWatch Logsの説明です。

</details>

### 問題5

AWS CDK（Cloud Development Kit）の説明として正しいものはどれですか？

- **A.** JSON/YAMLでインフラを定義するサービスで、CloudFormationの代替となる
- **B.** PythonやTypeScriptなどのプログラミング言語でインフラを定義し、内部的にCloudFormationテンプレートを生成する
- **C.** フロントエンドアプリケーションの開発・デプロイを自動化するプラットフォーム
- **D.** コンテナイメージをビルドしてデプロイするためのCI/CDツール

<details>
<summary>解答・解説</summary>

**解答: B**

解説: AWS CDKは、**PythonやTypeScriptなどのプログラミング言語でインフラを定義**できるツールです。CDKで書いたコードは**内部的にCloudFormationテンプレートに変換**されて処理されます。CDKはCloudFormationの「代替」ではなく「上位レイヤー」であるため、Aは誤りです。CはAmplifyの説明です。

</details>

### 問題6

ある企業のアプリケーションは、API Gateway → Lambda → DynamoDBという構成で動いています。最近「レスポンスが遅い」という報告が増えました。どのサービスで処理時間がかかっているかを特定するために、最も適したAWSサービスはどれですか？

- **A.** Amazon CloudWatch
- **B.** AWS CloudTrail
- **C.** AWS X-Ray
- **D.** AWS Config

<details>
<summary>解答・解説</summary>

**解答: C**

解説: **AWS X-Ray**は、複数のサービスをまたがるリクエストの流れを追跡し、各サービスでの処理時間を可視化します。「API Gateway → Lambda → DynamoDB」のようなサービスチェーンのどこでボトルネックが発生しているかを特定するのに最適です。CloudWatchは個々のサービスのメトリクス監視、CloudTrailはAPI操作の監査ログ記録、AWS Configはリソースの構成変更の追跡に使います。

</details>

# 問題作成スタイルガイド

このファイルは、`docs/questions/` 配下の問題バンクに問題を追加する際に守るべき基準・条件を定義します。
問題作成エージェント・レビューエージェントはいずれも、このガイドを参照して作業を行ってください。

---

## 1. 対象試験の基本情報

| 項目 | 内容 |
|------|------|
| 試験名 | AWS Certified Cloud Practitioner（CLF-C02） |
| 問題形式 | 択一選択（4択から1つ）/ 複数選択（5択から2つ） |
| 合格ライン | 1000点満点中700点（目安：正答率70%） |
| 対象読者 | IT初心者・クラウド未経験者 |

### 試験ドメインと配点

| ドメインID | ドメイン名 | 配点 |
|-----------|-----------|------|
| `domain1` | クラウドのコンセプト | 24% |
| `domain2` | セキュリティとコンプライアンス | 30% |
| `domain3` | クラウドテクノロジーとサービス | 34% |
| `domain4` | 請求、料金、サポート | 12% |

### 章とドメインの対応

| ドメインID | 対応する章 |
|-----------|-----------|
| `domain1` | ch00, ch07, ch14 |
| `domain2` | ch09 |
| `domain3` | ch01, ch02, ch03, ch04, ch05, ch06, ch08, ch10, ch11, ch12, ch13, ch16, appA |
| `domain4` | ch15, appB |

---

## 2. JSONフィールド仕様

問題は各章の JSON ファイル（`docs/questions/<chapter>-*.json`）の `questions` 配列に追記します。

### フィールド定義

| フィールド | 必須 | 型 | 説明 |
|-----------|------|----|------|
| `id` | ✅ | string | 問題の一意なID（命名規則は後述） |
| `type` | ✅ | string | `"single"`（択一）または `"multi"`（複数選択） |
| `domain` | ✅ | string | 試験ドメイン名（下記の表記に統一） |
| `text` | ✅ | string | 問題文 |
| `choices` | ✅ | object | 選択肢（キー: A〜D or A〜E） |
| `answer` | ✅ | string \| array | 正解。`single` は文字列、`multi` は配列 |
| `explanation` | ✅ | string | 解説文 |

### `domain` フィールドの表記（この表記に統一すること）

```
第1分野: クラウドのコンセプト
第2分野: セキュリティとコンプライアンス
第3分野: クラウドテクノロジーとサービス
第4分野: 請求、料金、サポート
```

---

## 3. 問題IDの命名規則

```
{chapter}-{連番3桁}
```

**例:**
- `ch00-001`, `ch00-002`, ...
- `ch09-001`, `ch09-002`, ...
- `appA-001`, `appB-001`, ...

**ルール:**
- 連番は章の中でユニークであること
- 既存の最大番号の次の番号を採番する（ギャップを作らない）
- 章をまたいだ番号の重複は許容される（`ch01-001` と `ch02-001` は共存可）

---

## 4. 問題タイプの使い分け

### `"single"` — 択一選択問題（4択）

- 選択肢は **A〜D の4つ**
- `answer` は文字列（例: `"B"`）
- 全問題のうち **約80%** を占めるようにする

```json
{
  "id": "ch09-001",
  "type": "single",
  "domain": "第2分野: セキュリティとコンプライアンス",
  "text": "問題文",
  "choices": { "A": "...", "B": "...", "C": "...", "D": "..." },
  "answer": "B",
  "explanation": "解説"
}
```

### `"multi"` — 複数選択問題（5択から2つ）

- 選択肢は **A〜E の5つ**
- `answer` は **必ず2要素の配列**（例: `["A", "C"]`）
- 問題文に「**2つ選んでください**」と明示する
- 全問題のうち **約20%** を目安に含める

```json
{
  "id": "ch09-010",
  "type": "multi",
  "domain": "第2分野: セキュリティとコンプライアンス",
  "text": "〜に該当するサービスを2つ選んでください。",
  "choices": { "A": "...", "B": "...", "C": "...", "D": "...", "E": "..." },
  "answer": ["A", "C"],
  "explanation": "解説"
}
```

## 5. 問題文の書き方ルール

### 推奨する問題スタイル

CLF-C02 はシナリオベースの問題が多いため、以下のスタイルを積極的に使う。

**サービス選択型**
> 「〜を実現するために最も適切なAWSサービスはどれですか？」

**シナリオ型**
> 「ある企業は〜という課題を抱えています。この課題を解決するために最も適切なサービスはどれですか？」

**比較型**
> 「〜と〜の違いとして正しいものはどれですか？」

**責任共有モデル型**
> 「責任共有モデルにおいて、AWSが責任を持つのはどれですか？」

### 問題文の必須要件

- [ ] 1問で1つの概念だけを問う（複合問題にしない）
- [ ] 主語・述語が明確で、文意が1通りに解釈できる
- [ ] 「最も適切な」「最もコスト効率が高い」など、絞り込みの基準を明示する
- [ ] 問題文の末尾は「〜はどれですか？」「〜を選んでください。」で統一する
- [ ] 固有名詞（AWSサービス名）は正式名称を使う（例: 「ラムダ」ではなく「AWS Lambda」）

### 否定型問題について

否定型（「〜**でない**ものはどれですか？」）は使用を**最小限**にとどめる。
使用する場合は `**ではない**` のように太字で強調する。

```
AWSの責任共有モデルにおいて、AWSが責任を持つ範囲として**正しくない**ものはどれですか？
```

---

## 6. 選択肢の書き方ルール

- [ ] 全選択肢の長さ・語尾のスタイルを統一する（体言止め or 文末統一）
- [ ] 正解は A〜D（またはA〜E）のいずれかにランダムに配置する（正解が常にBにならないよう注意）
- [ ] 不正解の選択肢（ダミー）は**もっともらしいが明確に誤っている**内容にする
- [ ] 「上記のすべて」「該当なし」は使用しない
- [ ] 選択肢の並び順は、文字列の場合は五十音順・英数字の場合はアルファベット順を基本とする
- [ ] 似たサービスを並べる場合（例: CloudWatch / CloudTrail / Config）は、混同を誘う設計にしてよい

---

## 7. 解説（`explanation`）の書き方ルール

解説は以下の構成で書く。

```
① 正解の理由（なぜその選択肢が正しいのかを端的に）
② 不正解の選択肢の説明（なぜ誤りなのかを補足）
③ 試験でよく問われる観点・ひっかけパターンの注意喚起（任意）
```

**必須要件:**
- [ ] 300文字以上を目安に、初心者が理解できる言葉で書く
- [ ] 専門用語を使う場合は、初出時に括弧で補足する（例: 「DDoS攻撃（大量アクセスでサービスを麻痺させる攻撃）」）
- [ ] 「なぜそのサービスが使われるのか」まで踏み込んで説明する
- [ ] 全選択肢（不正解含む）に言及することが望ましい

## 8. 品質基準

### 難易度バランス

章ごとの問題は以下の難易度比率を目安とする。

| 難易度 | 目安 | 基準 |
|--------|------|------|
| 易（基本） | 40% | サービス名と用途の直接対応。暗記で正解できる。 |
| 中（応用） | 45% | 複数サービスの比較・使い分け。理解が必要。 |
| 難（シナリオ） | 15% | 複合的な要件を読み解き、最適解を選ぶ。 |

### 問題数の目標設計

**基本思想**: 受験者が `--weighted` オプションで50問の模擬試験を **5回繰り返しても重複が少ない**状態を目指す。

| 段階 | バンク全体の問題数 | 状態 |
|------|-----------------|------|
| 最低ライン | 150問以上 | 模擬試験を3回繰り返せる多様性 |
| **目標ライン** | **250問以上** | 模擬試験を5回繰り返せる多様性 |

目標ラインは以下の計算から導出している。
`各ドメインの配点(%) × 50問 × 5回 ÷ そのドメインを担当する章数`

#### 章ごとの問題数目標

| 分類 | 対象章 | 目標（5回分） | 最低ライン（3回分） |
|------|--------|-------------|-----------------|
| domain2 単独担当 | `ch09` | **75問以上** | 45問 |
| domain1・3・4 の主要章 | `ch00` `ch01` `ch03` `ch04` `ch05` `ch07` `ch10` `ch14` `ch15` | **20〜30問** | 12問 |
| domain1・3・4 の補助章 | `ch02` `ch06` `ch08` `ch11` `ch12` `ch13` `ch16` `appA` `appB` | **10〜15問** | 6問 |

#### パターン枯渇時の例外規定

章が扱うサービス・概念が少なく、**最低ラインの達成が困難な場合**は以下を適用する。

- 作成可能なすべてのパターンを網羅した上で、最低ラインを下回っても許容する
- 問題作成エージェントはレビュー時に枯渇を申告すること
  例: 「`appB` はサポートプランのみを扱う章であり、最大8問が上限と判断しました」
- レビューエージェントは申告内容を確認し、妥当であればマスターファイルへの追加を承認する

## 9. 避けるべきパターン（禁止事項）

- ❌ 試験範囲外のサービス・機能を問う問題（CLF-C02の出題範囲に含まれないもの）
- ❌ 試験メタ情報（問題数・試験時間・合格スコアなど）を問う問題（`appC` 章は試験対策のまとめであり、本試験では問われないため問題バンクの対象外）
- ❌ 正解が2つ以上存在する択一選択問題
- ❌ 文言が異なるだけで内容が同じ重複問題
- ❌ AWSの公式ドキュメントや学習ドキュメント（`docs/chapters/`）に記載のない情報を含む問題
- ❌ 特定のリージョン・アカウントに依存した問題（一般的な話として出題すること）
- ❌ 価格・スペックの具体的な数値を問う問題（変更される可能性があるため）
  - 例外: 以下は試験頻出として定着している数値のため許容する
    - Aurora のパフォーマンス倍率（MySQL互換: 5倍、PostgreSQL互換: 3倍）
    - AWS Lambda の実行時間上限（15分）

## 10. レビューチェックリスト

問題をマスターファイルに追加する前に、レビューエージェントは以下を確認する。

### フォーマットチェック

- [ ] `id` が命名規則に従っており、同章内で重複していない
- [ ] `type` が `"single"` または `"multi"` のいずれかである
- [ ] `domain` の表記が定義された4パターンのいずれかと完全一致している
- [ ] `single` の選択肢は A〜D の4つ、`multi` の選択肢は A〜E の5つである
- [ ] `single` の `answer` は文字列、`multi` の `answer` は2要素の配列である
- [ ] `answer` に含まれるキーが `choices` に存在する
- [ ] 全フィールドが空文字でない

### 内容チェック

- [ ] 問題文が1つの概念のみを問うている
- [ ] 問題文の末尾が規定のスタイルで終わっている
- [ ] 正解が `choices` の中に1つ（multiは2つ）あり、誤りのない内容である
- [ ] 不正解の選択肢がもっともらしい内容になっている
- [ ] `domain` の値が、問題の章（`chapter`）の対応ドメインと一致している
- [ ] 解説に正解の理由が記載されている
- [ ] `docs/chapters/` の学習ドキュメントに記載された内容の範囲内である
- [ ] 禁止パターン（セクション9）に該当しない

### 全体バランスチェック（章単位）

- [ ] `single` と `multi` の比率がおおむね 80:20 になっている
- [ ] 難易度が易・中・難に分散している（偏っていない）
- [ ] 正解の選択肢（A〜D or A〜E）が特定のキーに偏っていない

## 11. 記述例（完成形）

### 択一選択問題の例

```json
{
  "id": "ch09-001",
  "type": "single",
  "domain": "第2分野: セキュリティとコンプライアンス",
  "text": "ある企業のセキュリティ担当者が、AWS環境内で不審なAPIコールが行われていないか監査ログを確認したいと考えています。最も適切なサービスはどれですか？",
  "choices": {
    "A": "Amazon CloudWatch",
    "B": "AWS CloudTrail",
    "C": "AWS Config",
    "D": "Amazon GuardDuty"
  },
  "answer": "B",
  "explanation": "AWS CloudTrail は「誰が・いつ・何の操作をしたか」というAPIコールの記録（監査ログ）を保存するサービスです。不審な操作の調査や、コンプライアンス要件への対応に使用します。CloudWatch はCPUやメモリなどのパフォーマンス監視に使うサービスであり、操作ログの記録は行いません。AWS Config はリソースの設定変更を追跡するサービスです。GuardDuty はAIによる脅威検知サービスですが、監査ログの確認という目的には CloudTrail が適切です。"
}
```

### 複数選択問題の例

```json
{
  "id": "ch09-010",
  "type": "multi",
  "domain": "第2分野: セキュリティとコンプライアンス",
  "text": "DDoS攻撃やWebアプリケーションへの不正アクセスからAWSリソースを保護するサービスを2つ選んでください。",
  "choices": {
    "A": "AWS WAF",
    "B": "AWS Shield",
    "C": "Amazon GuardDuty",
    "D": "AWS KMS",
    "E": "Amazon Macie"
  },
  "answer": ["A", "B"],
  "explanation": "AWS Shield はDDoS攻撃（大量のアクセスでサービスを麻痺させる攻撃）からの保護を提供します。Standardは全アカウントに無料で自動適用されます。AWS WAF（Web Application Firewall）はSQLインジェクションやXSSといったWebアプリケーションの脆弱性を突く攻撃を防ぐファイアウォールです。GuardDuty はAIによる脅威検知サービスですが、攻撃を「防御」するものではなく「検知」するサービスです。KMS は暗号化キーの管理サービス、Macie は機密データの検出サービスであり、いずれも攻撃防御とは目的が異なります。"
}
```
